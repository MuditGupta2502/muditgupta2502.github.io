Multi-Tenant Database Design Patterns

Architecture | 18 min read

SaaS systems often support thousands of tenants with different usage patterns. The main design choice is where to place tenant boundaries.

Tenant Isolation Models

ModelIsolationCostOps Complexity
Shared DB, shared schemaLowLowestLow
Shared DB, separate schemaMediumMediumMedium
Separate DB per tenantHighHighestHigh

Shared Schema Pattern

CREATE TABLE invoices ( id BIGSERIAL PRIMARY KEY, tenant_id BIGINT NOT NULL, amount NUMERIC(12,2) NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE INDEX idx_invoices_tenant_created ON invoices (tenant_id, created_at DESC); -- Enforce isolation with RLS ALTER TABLE invoices ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation ON invoices USING (tenant_id = current_setting('app.tenant_id')::BIGINT);

Migration Path

-- Common journey: -- Stage 1: shared schema for MVP -- Stage 2: move noisy tenants to dedicated shards -- Stage 3: enterprise tenants get dedicated databases -- Keep an internal tenant-routing table to map tenant -> cluster/database

Summary

  • Start with shared schema for speed, but design with tenant_id everywhere from day one.
  • Use RLS + composite indexes by tenant for performance and safety.
  • Support progressive isolation for large enterprise tenants later.
