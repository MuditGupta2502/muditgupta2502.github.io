CAP Theorem Deep Dive: Consistency, Availability and PACELC

Distributed Systems | 26 min read

Eric Brewer's CAP theorem (2000) states that a distributed data store can guarantee at most two of three properties during a network partition: Consistency, Availability, and Partition Tolerance. Understanding CAP — and its limitations — is fundamental to choosing and designing distributed databases.

The Three Properties

PropertyDefinitionPractical meaning
ConsistencyEvery read receives the most recent write or an errorAll nodes return the same (latest) data at any moment
AvailabilityEvery request receives a (non-error) response, without guarantee of latest dataThe system always responds; response may be stale
Partition ToleranceThe system continues operating despite network message loss or delay between nodesThe system works even when nodes can't talk to each other
CAP is about network partitions: Partition Tolerance is not optional in any distributed system — networks DO fail. The real choice is: when a partition occurs, do you prioritize consistency or availability? The theorem is better stated as CP vs AP during a partition, not "pick 2 of 3".

CP Systems: Prioritize Consistency

When a partition occurs, CP systems stop responding (become unavailable) rather than return potentially stale data.

-- CP Systems: HBase, MongoDB (majority write concern), ZooKeeper, etcd -- Example: ZooKeeper (coordination service) -- 5-node ZooKeeper cluster, quorum = 3 -- Network partition splits cluster: 3 nodes on one side, 2 on the other -- Side with 3 nodes: forms quorum → continues serving reads AND writes -- Side with 2 nodes: cannot form quorum → REFUSES all operations (returns error) -- Consistency is maintained: no split-brain, no stale reads -- 2 nodes become unavailable (sacrifice availability for consistency) -- Real impact: -- Any service connecting ONLY to the 2-node partition gets "no leader available" error -- Services on the 3-node partition work normally -- Correct choice for: distributed locks, leader election, configuration service -- (better to fail than to allow two services to think they're both the leader)

AP Systems: Prioritize Availability

When a partition occurs, AP systems continue responding with potentially stale data rather than return errors.

-- AP Systems: Cassandra (ONE consistency), DynamoDB, CouchDB, Riak, DNS -- Example: Cassandra with 3 nodes, replication factor N=3, read/write at ONE -- Network partition splits: Nodes A+B on one side, Node C on the other -- Node C: isolated, but still accepts reads and writes → returns stale data! -- Nodes A+B: accept reads and writes → fully up-to-date -- Writes that hit Node C while isolated: go into its local storage -- When partition heals: anti-entropy background repair syncs the differences -- Final state: eventual consistency, potential conflict resolution needed -- Real impact for users: -- No errors, no downtime — but potentially stale reads -- E.g., a user who adds to cart on Node C during partition might not see their -- updated cart on Node A immediately after partition heals -- Acceptable for: shopping carts, user sessions, social feeds, telemetry

The PACELC Model: Beyond CAP

PACELC (Daniel Abadi, 2012) extends CAP to also describe the latency-consistency trade-off that applies even when no partition exists.

-- PACELC: If Partition (P), choose Availability (A) or Consistency (C); -- Else (E — normal operation), choose Latency (L) or Consistency (C) -- Most real-world decisions aren't about partitions (rare) but about latency vs consistency -- when the system is healthy (the "else" branch). -- Example: synchronous vs asynchronous replication -- Synchronous (ELC — else latency or consistency: chooses C): -- Every write waits for replica to confirm -- Higher latency, strong consistency -- PostgreSQL synchronous_standby_names = 'replica1' -- Asynchronous (ELL — chooses L): -- Write acks immediately, replica catches up in background -- Lower latency, replica may lag seconds/minutes behind -- PACELC classification of common databases: -- PostgreSQL (sync): PA/EC (partition: available; else: consistent) -- Cassandra (LWW): PA/EL (partition: available; else: low latency) -- MongoDB (strong): PC/EC (partition: consistent; else: consistent) -- DynamoDB: PA/EL (partition: available; else: low latency by default)

Network Partitions in Practice

-- Network partitions are more common than disk failures: -- Types: -- 1. Complete partition: nodes cannot communicate at all (datacenter link failure) -- 2. Partial partition: some nodes can reach some others (asymmetric routing issue) -- 3. Slow network: messages arrive but very delayed (appear as timeouts) -- 4. Process pauses: long GC pauses, VM migration (node appears unresponsive) -- Partial partitions are the hardest: -- Leader believes it's connected to quorum -- But some messages between specific pairs are lost -- Results in inconsistent state that is hard to detect -- Detection: heartbeats + timeouts -- Every node sends heartbeats every 1-5 seconds -- If no heartbeat in N seconds, assume partition -- Timeout tuning trade-offs: -- Too short (1s): false positives on slow network → unnecessary failovers -- Too long (30s): 30s of downtime before detection -- Production sweet spot: 5-10s timeout with 3 missed heartbeats

Google Spanner: Practically CA+P

-- Spanner achieves "practically CA" by minimizing partition probability: -- -- Redundant fiber within datacenters -- -- Cross-datacenter connections with SLA -- -- TrueTime: GPS + atomic clocks give bounded time uncertainty (< 7ms) -- -- External consistency (essentially linearizability) via TrueTime -- How TrueTime works: -- Every read/write tagged with TrueTime interval [earliest, latest] -- Commit waits until earliest > latest of conflicting transaction -- This "commit wait" (typically < 10ms) guarantees causal ordering globally -- Spanner is CP in CAP terms, but partitions are so rare (hardware redundancy) -- and partition tolerance is so brief that it's practically CA in practice -- The cost: higher write latency due to commit wait
CAP theorem limitations to know:
  • CAP uses a very specific definition of "consistency" (linearizability), not the ACID "C"
  • CAP is binary — real systems have tunable consistency (quorum)
  • CAP says nothing about performance, durability, or operational complexity
  • PACELC is more practically useful because it covers the normal (no-partition) case too

Summary

  • CAP: during a network partition, distributed systems must choose CP (consistency) or AP (availability)
  • CP systems refuse requests when they can't confirm consistency (ZooKeeper, etcd, MongoDB)
  • AP systems return possibly stale data rather than error (Cassandra, DynamoDB, Riak)
  • PACELC extends CAP: even without partitions, systems trade latency (faster) vs consistency (accurate)
  • Most decisions are latency vs consistency (PACELC's E branch), not partition-handling (rare)
  • Google Spanner uses TrueTime to achieve practical global consistency with bounded partition risk
