In-Memory Databases: Redis Data Structures, Eviction and Persistence

NoSQL & Caching | 27 min read

Redis is the world's most widely used in-memory data structure server. Unlike a simple key-value cache, Redis offers rich data types, atomic operations, Lua scripting, and configurable persistence — making it suitable for caching, session storage, rate limiting, leaderboards, message queues, and real-time analytics simultaneously.

Core Data Structures

String

-- Most versatile type: stores text, integers, serialized JSON, binary blobs -- Max size: 512 MB SET user:1001:name "Priya Sharma" GET user:1001:name -- "Priya Sharma" -- Integer string: atomic increment/decrement SET page_views 0 INCR page_views -- 1 (atomic, no race condition) INCRBY page_views 10 -- 11 DECR page_views -- 10 -- TTL: expire after 3600 seconds SET session:abc123 "user_data_json" EX 3600 TTL session:abc123 -- 3599 (seconds remaining) PERSIST session:abc123 -- remove the TTL (make permanent) -- Set-if-not-exists (atomic, good for distributed locks): SET lock:resource_123 "holder_id" NX EX 30 -- NX = only set if key does NOT exist -- EX 30 = expire in 30 seconds (auto-release on crash)

Hash

-- Hash: field-value map. Like a JSON object but with atomic field operations -- Perfect for storing object/entity data HSET customer:1001 name "Priya" city "Mumbai" tier "gold" total_orders 42 HGET customer:1001 tier -- "gold" HMGET customer:1001 name city -- ["Priya", "Mumbai"] HGETALL customer:1001 -- {name: "Priya", city: "Mumbai", tier: "gold", ...} -- Increment a numeric field atomically: HINCRBY customer:1001 total_orders 1 -- 43 -- Check/delete individual fields: HEXISTS customer:1001 email -- 0 (false) HDEL customer:1001 old_field -- Memory: small hashes (< 128 fields, values < 64 bytes) use ziplist encoding -- → very compact in memory; good for caching many small objects

List

-- Doubly-linked list. Fast head/tail operations; slow random access. -- Use cases: message queues, activity feeds, recent items LPUSH recent_searches:user1001 "wireless keyboard" -- push to left (head) LPUSH recent_searches:user1001 "mechanical mouse" LRANGE recent_searches:user1001 0 9 -- get first 10 items LTRIM recent_searches:user1001 0 19 -- keep only last 20 -- Queue pattern (FIFO — producer pushes left, consumer pops right): LPUSH queue:email_send "job_payload_json" -- enqueue BRPOP queue:email_send 30 -- blocking pop (waits up to 30s for item) -- BRPOP is used by worker processes that wait for new jobs -- Stack pattern (LIFO): LPUSH undo_stack:session123 "action_json" -- push LPOP undo_stack:session123 -- pop from same end

Set

-- Unordered collection of unique strings -- Use cases: tags, unique visitors, permissions, social connections SADD product:101:tags "wireless" "keyboard" "electronics" SMEMBERS product:101:tags -- {"wireless", "keyboard", "electronics"} SISMEMBER product:101:tags "wireless" -- 1 (true) -- Set operations (very fast, O(N)): SINTER user:1001:friends user:1002:friends -- mutual friends SUNION user:1001:interests user:1002:interests -- combined interests SDIFF user:1001:friends user:1002:friends -- friends only user1001 has -- HyperLogLog: approximate unique count (very memory-efficient, ~12KB per key) PFADD unique_visitors:2024-01-15 "ip_1" "ip_2" "ip_3" PFCOUNT unique_visitors:2024-01-15 -- ~3 (< 1% error rate) -- Use for "DAU" style metrics without storing millions of user IDs

Sorted Set (ZSet)

-- Set where each member has a floating-point score. Members are unique; -- scores can repeat. Members stored sorted by score. -- Use cases: leaderboards, priority queues, time-series indexes -- Game leaderboard: ZADD leaderboard:game_001 9450.0 "player_priya" ZADD leaderboard:game_001 8320.0 "player_raj" ZADD leaderboard:game_001 9820.0 "player_sam" -- Top 10 players (highest scores first): ZREVRANGE leaderboard:game_001 0 9 WITHSCORES -- Returns: [player_sam 9820, player_priya 9450, player_raj 8320, ...] -- Rank of a specific player: ZREVRANK leaderboard:game_001 "player_priya" -- 1 (0-based, so rank #2) -- Players with score in range: ZRANGEBYSCORE leaderboard:game_001 8000 9000 WITHSCORES -- Rate limiting with sorted set (sliding window): -- Key: ratelimit:user:1001 -- Members: request timestamps, Score: timestamp ZADD ratelimit:user:1001 1704067200.123 "req_uuid_1" ZREMRANGEBYSCORE ratelimit:user:1001 0 (NOW - 60) -- remove older than 60s ZCARD ratelimit:user:1001 -- count requests in last 60s; reject if > limit

Stream

-- Redis 5.0+: append-only log with consumer groups (like Kafka) -- Use case: event sourcing, audit logs, message queues XADD events:orders * customer_id 1001 action "purchase" total 299.00 -- * = auto-generate ID (timestamp-based) -- Consumer group: for distributing work across workers XGROUP CREATE events:orders workers $ MKSTREAM XREADGROUP GROUP workers worker_1 COUNT 10 STREAMS events:orders > -- > = read messages not yet delivered to this group -- Acknowledge processed messages: XACK events:orders workers message_id_1 message_id_2 -- Unacknowledged messages are "pending" — can re-deliver to another worker on failure

Eviction Policies

When Redis reaches its maxmemory limit, it must either reject new writes or evict existing keys. The policy you choose dramatically impacts which data gets evicted.

PolicyEvictsUse Case
noeviction (default)Nothing — writes return errorsDangerous; only for persistence-only use
allkeys-lruLeast Recently Used across all keysGeneral-purpose cache
allkeys-lfuLeast Frequently Used across all keysSkewed access (hot/cold data)
volatile-lruLRU among keys with TTL setMix of volatile cache + persistent data
volatile-lfuLFU among keys with TTL setSame, frequency-weighted
allkeys-randomRandom keyUniform access pattern (rare)
volatile-ttlKey with earliest expiryCache where TTL reflects importance
-- Set memory limit and eviction policy in redis.conf: maxmemory 2gb maxmemory-policy allkeys-lfu -- LFU vs LRU: -- LRU: evicts items not accessed recently — good for temporal locality -- LFU: evicts items accessed infrequently — better when hot keys are stable -- Example: if a viral post is accessed 1M times/day and then goes quiet, -- LFU will keep it longer (high frequency) even when recently less accessed

Persistence Modes

-- RDB (Redis Database): point-in-time snapshot -- redis.conf: save 900 1 (save every 900s if at least 1 key changed) -- save 300 10 -- save 60 10000 -- Compact binary format; fast to load on restart -- Trade-off: if Redis crashes between snapshots, data since last snapshot is lost CONFIG SET save "900 1 300 10 60 10000" -- AOF (Append-Only File): log every write command appendonly yes appendfsync everysec -- fsync to disk every second -- always -- fsync every write (safest, but slow) -- no -- OS decides when to flush (fastest, less safe) -- Trade-off: AOF file is larger; startup is slower (replay all commands) -- Recovery: at most 1 second of data lost (with everysec) -- RDB + AOF hybrid (recommended for production): aof-use-rdb-preamble yes -- On restart: load from RDB snapshot first, then replay AOF from that point -- Faster than pure AOF startup, with better durability than pure RDB -- No persistence (pure cache): appendonly no save "" -- disable all RDB saves -- All data lost on restart; appropriate when Redis is a pure cache layer

Atomic Operations and Lua Scripting

-- Redis is single-threaded for commands: no concurrent command conflicts -- But multi-step operations need atomic execution → use Lua scripts or MULTI/EXEC -- MULTI/EXEC (transactions): queue commands, execute atomically MULTI HSET cart:user1001 item:101 2 -- set quantity 2 for product 101 HINCRBY cart:user1001 total 499 -- increment total price EXPIRE cart:user1001 3600 -- reset TTL EXEC -- All three execute atomically — no interleaving -- WATCH/MULTI/EXEC (optimistic concurrency): WATCH cart:user1001 -- watch for changes val = GET cart:user1001 -- read current state MULTI SET cart:user1001 newval -- attempt update result = EXEC -- If cart:user1001 changed between WATCH and EXEC, EXEC returns nil → retry -- Lua script (fully atomic, reusable): EVAL " local current = redis.call('GET', KEYS[1]) if tonumber(current) >= tonumber(ARGV[1]) then redis.call('DECRBY', KEYS[1], ARGV[1]) return 1 -- success end return 0 -- insufficient funds " 1 wallet:user1001 100 -- Atomically: check balance >= 100, then deduct 100 -- No race condition possible

Pub/Sub

-- Publish-subscribe: publishers send messages to channels; subscribers receive them -- Messages are NOT stored — fire-and-forget (use Streams for durability) -- Subscriber (connection 1): SUBSCRIBE notifications:user:1001 notifications:system -- Publisher (connection 2): PUBLISH notifications:user:1001 "{\"type\":\"order_shipped\",\"order_id\":456}" -- Pattern subscribe (wildcard): PSUBSCRIBE notifications:user:* -- subscribes to all user notifications -- Subscriber receives: {channel, pattern, message} tuples -- Use cases: real-time chat, live notifications, invalidation events for cache
Redis Pub/Sub limitation: Messages are not persisted. If a subscriber goes offline, it misses all messages published while disconnected. For reliable message delivery, use Redis Streams (XADD/XREADGROUP) or a dedicated message broker.

Redis Cluster for Horizontal Scaling

-- Redis Cluster: automatic sharding across 16384 hash slots -- Key → CRC16(key) mod 16384 → hash slot → assigned to a node -- 3-master + 3-replica cluster: each master handles ~5461 hash slots -- Automatic failover: if a master dies, its replica is promoted -- Hash tags force multi-key operations to the same slot: SET {user:1001}:session "data" SET {user:1001}:cart "data" -- Both keys hash on {user:1001} → guaranteed same slot → MGET works cross-key -- Limitation: multi-key commands (MGET, MSET, SUNION) only work when all keys -- are in the same slot. Use hash tags when you need transactional across keys.

Summary

  • Redis offers 8 data types: String, Hash, List, Set, Sorted Set, Stream, Bitmap, HyperLogLog
  • Each type has O(1) or O(log N) operations for its primary use case — velocity is a feature
  • Choose eviction policy based on access pattern: allkeys-lfu for skewed hot/cold data
  • Persistence: RDB for fast snapshots, AOF for durability, hybrid for both; disable for pure cache
  • Lua scripts and MULTI/EXEC provide atomicity for multi-step operations
  • Redis Cluster shards across 16384 hash slots; use hash tags to co-locate related keys
