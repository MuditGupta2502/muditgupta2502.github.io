Backup Strategies: Full, Incremental, Differential and PITR

Operations | 20 min read

Backups are your last line of defense against data loss from hardware failure, accidental deletion, ransomware, or corrupted application code. A backup strategy that hasn't been tested is not a backup strategy — it's wishful thinking.

RTO vs RPO

Before choosing a backup strategy, define your requirements:

MetricDefinitionExample
RPO (Recovery Point Objective)Maximum acceptable data loss (time span)"We can lose at most 1 hour of data"
RTO (Recovery Time Objective)Maximum acceptable downtime"System must be back in 4 hours"

A 24-hour backup without WAL archiving gives RPO=24h. PITR with continuous WAL archiving can give RPO of seconds. RTO depends on backup size and replay time.

Logical vs Physical Backups

AttributeLogical (pg_dump / mysqldump)Physical (pg_basebackup / filesystem snapshot)
Output formatSQL statements or custom dump formatBinary copy of data files
PortabilityCross-version, cross-platformSame major version and architecture
SpeedSlow for large databasesFast (close to raw I/O speed)
GranularityDatabase, schema, or table levelEntire cluster
PITR supportNoYes (with WAL archiving)
ConsistencySnapshot consistent (transaction-consistent)Requires careful handling during backup

Backup Types: Full, Incremental, Differential

-- Full backup: complete copy of all data -- Pro: simplest restore (single backup file) -- Con: expensive storage and time for large databases -- Typical schedule: weekly -- Incremental backup: only changes since the LAST backup (full or incremental) -- Pro: minimal storage per backup -- Con: restore requires: full + all incrementals in sequence → complex -- Typical schedule: daily (after weekly full) -- Differential backup: changes since the LAST FULL backup -- Pro: restore requires only: full + latest differential -- Con: grows over time (Saturday's diff includes Mon-Fri changes) -- Typical schedule: daily -- Example weekly schedule: -- Sunday: Full backup (10GB) -- Monday: Incremental (500MB changed Mon) -- Tuesday: Incremental (300MB changed Tue) -- Wednesday: Differential (800MB changed Mon+Tue+Wed) -- -- To restore to Tuesday state: -- Incremental strategy: Full + Mon incremental + Tue incremental -- Differential strategy: Full + Tuesday differential (simpler but larger) -- WAL-based continuous backup (PostgreSQL PITR): -- Combines a base backup (physical) + streaming WAL -- Can restore to ANY point in time, not just backup boundaries -- This is the gold standard for most production PostgreSQL deployments

Logical Backup with pg_dump

-- Single database backup (SQL format): pg_dump -U postgres -d mydb -F p -f /backup/mydb_$(date +%Y%m%d).sql -- -F p = plain SQL (default); -F c = custom (compressed, parallel restore) -- Custom format (faster restore with parallel workers): pg_dump -U postgres -d mydb -F c -f /backup/mydb.dump -j 4 -- Backup specific tables: pg_dump -U postgres -d mydb -t orders -t order_items -F c -f /backup/orders.dump -- Restoring a SQL dump: psql -U postgres -d mydb_restore < /backup/mydb.sql -- Restoring a custom format dump: pg_restore -U postgres -d mydb_restore -F c -j 4 /backup/mydb.dump -- -j 4 = parallel restore with 4 workers (much faster for large databases) -- Full cluster backup (includes roles, tablespaces): pg_dumpall -U postgres -f /backup/cluster_full.sql -- Continuous consistency tip: -- pg_dump uses a single transaction (REPEATABLE READ) for consistency -- No table locks; safe for production use -- But very large tables may cause slow backups and bloat vacuum

Physical Backup with pg_basebackup

-- Creates a copy of the data directory; suitable as base for PITR pg_basebackup \ -h localhost \ -U replication_user \ -- must have REPLICATION privilege -D /backup/base/ \ -- destination directory -Ft \ -- tar format -z \ -- gzip compression -P \ -- progress reporting --wal-method=stream -- stream WAL during backup (ensures consistency) -- Options explained: -- --wal-method=fetch: copy WAL files after backup completes (may miss some) -- --wal-method=stream: runs a second replication slot during backup (safer) -- --checkpoint=fast: force immediate checkpoint (faster start, more I/O spike) -- --checkpoint=spread: spread checkpoint I/O (gentler on production, slower) -- Output: -- base.tar.gz — data directory -- pg_wal.tar.gz — WAL files needed to make backup consistent -- To verify backup: -- Start a PostgreSQL instance against it in recovery mode; confirm it can start

WAL Archiving for PITR

-- Step 1: Enable WAL archiving (postgresql.conf): wal_level = replica -- or logical archive_mode = on archive_command = 'test ! -f /wal/archive/%f && cp %p /wal/archive/%f' -- The test prevents overwriting (WAL filenames can be reused after timeline change) -- For cloud storage: use wal-g, pgBackRest, or Barman -- Step 2: First base backup: pg_basebackup -D /backup/base -Ft -z -P --wal-method=stream -- Step 3: WAL files are archived continuously (every ~16MB segment or 5 minutes) -- Step 4: Recovery after disaster -- a) Restore base backup: tar -xzf /backup/base/base.tar.gz -C /var/lib/postgresql/data/ -- b) Create recovery settings (postgresql.conf or recovery.conf): restore_command = 'cp /wal/archive/%f %p' recovery_target_time = '2024-11-04 09:00:00' -- restore to just before disaster recovery_target_action = 'promote' -- c) Start PostgreSQL — it replays WAL until 09:00:00, then promotes -- d) Verify data, then reconnect applications -- wal-g for cloud storage backup (popular open-source tool): -- WALG_S3_PREFIX=s3://my-bucket/postgres-backups wal-g backup-push $PGDATA -- WALG_S3_PREFIX=s3://my-bucket/postgres-backups wal-g backup-fetch $PGDATA LATEST -- Handles compression, encryption, and incremental file-level backups

Snapshot-Based Backups

-- Cloud provider storage snapshots: -- AWS EBS snapshot, Azure Managed Disk snapshot, GCP Persistent Disk snapshot -- These are block-level incremental snapshots by default (only changed blocks copied) -- Snapshot consistency: EBS snapshots are "crash-consistent" but NOT application-consistent -- To get application consistency: -- 1. CHECKPOINT in PostgreSQL (flush dirty pages to disk) -- 2. Immediately take the EBS snapshot -- 3. Resume normal operation -- PostgreSQL-level snapshot consistency for EBS: -- a) Run: SELECT pg_backup_start('EBS snapshot'); -- b) Take EBS snapshot -- c) Run: SELECT pg_backup_stop(); -- d) The WAL files listed in pg_backup_stop output must be archived -- AWS RDS automated backups: -- Daily snapshot during backup window -- Transaction logs retained for 0-35 days -- Enables PITR to any second within that retention period -- Managed fully by AWS — no manual WAL archiving needed -- ZFS/LVM snapshots for bare metal: -- lvcreate --snapshot --name db_snap -L 10G /dev/vg/postgres_data -- Mount snapshot read-only for backup, then remove -- Near-instant snapshot creation; incremental storage cost

Backup Testing Matrix

TestFrequencyWhat to Verify
Restore drillMonthlyFull restore completes; data is readable and consistent
PITR drillQuarterlyCan restore to specific time; row-level verification
Backup integrityWeeklypg_restore --list on dump; checksum verify on physical
Off-site restoreAnnuallyRestore in different region/cloud from archived backups
Critical mistakes to avoid:
  • Backing up to the same disk as the database — one failure destroys both
  • Never testing restores — discovered at disaster time that backup is corrupt
  • WAL archiving silently failing — check pg_stat_archiver.last_failed_wal
  • Forgetting archive_cleanup_command — WAL directory fills up, archiving stops
  • Not backing up pg_hba.conf and postgresql.conf — recovery without config is painful

Summary

  • Define RPO and RTO first — they drive every decision in your backup strategy
  • Logical backups (pg_dump): portable, granular, but slow and no PITR support
  • Physical backups (pg_basebackup): fast, support PITR, but tied to same major version
  • PITR = physical base backup + continuous WAL archiving → RPO down to seconds
  • Incremental strategy needs full + all incrementals for restore; differential needs full + latest diff
  • Always test restores — an untested backup is not a backup
