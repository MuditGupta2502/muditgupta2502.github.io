Query Processing Pipeline: From SQL Text to Execution

Query Internals | 26 min read

When you run a SQL query, a remarkable pipeline of transformations happens inside the database engine before a single row is read. Understanding this pipeline is the foundation of query optimization — if you know how the database interprets and plans your query, you can write queries that guide the optimizer toward efficient execution plans.

The Six Stages of Query Processing

SQL Text ↓ 1. Parsing — Syntax check, produce parse tree (AST) ↓ 2. Binding — Semantic validation, resolve names, type check ↓ 3. Logical Plan — Abstract relational algebra tree ↓ 4. Query Rewriting — Algebraic equivalences, predicate pushdown, subquery unnesting ↓ 5. Physical Plan — Choose algorithms: index scan vs seq scan, hash join vs merge join ↓ 6. Execution — Run the plan, stream results to client

Stage 1: Parsing

The parser checks SQL syntax and converts the text into a parse tree (Abstract Syntax Tree). It knows nothing about tables or columns yet — only grammar.

-- Input SQL: SELECT c.name, SUM(o.total) AS total_spent FROM Customers c JOIN Orders o ON c.customer_id = o.customer_id WHERE c.city = 'Mumbai' GROUP BY c.name HAVING SUM(o.total) > 1000; -- Simplified parse tree output: SELECT_STATEMENT SELECT_LIST: [c.name, SUM(o.total) AS total_spent] FROM_CLAUSE: [Customers AS c, Orders AS o] JOIN: ON c.customer_id = o.customer_id WHERE: c.city = 'Mumbai' GROUP_BY: [c.name] HAVING: SUM(o.total) > 1000 -- Syntax errors caught here: SELECT * FORM Orders; -- syntax error: unknown keyword "FORM"

Stage 2: Binding (Semantic Analysis)

The binder validates the parse tree against the system catalog (schema). It verifies that tables and columns exist, resolves aliases, checks type compatibility, and validates aggregation rules.

-- Semantic errors caught at binding: SELECT * FROM MissingTable; -- ERROR: relation does not exist SELECT fake_col FROM Orders; -- ERROR: column "fake_col" does not exist SELECT name, SUM(total) FROM Orders GROUP BY customer_id; -- ERROR: "name" must appear in GROUP BY or used in an aggregate -- The binder also: -- Expands SELECT * to actual column list -- Replaces table/column names with internal OIDs -- Validates function signatures and argument types -- Resolves ambiguous column references

Stage 3: Logical Plan

The bound parse tree is translated into a logical plan — a tree of relational algebra operations representing what to compute (not how). This is still abstract.

-- Naïve logical plan (before optimization): HAVING σ(SUM(total) > 1000) ↑ AGGREGATE γ(c.name, SUM(o.total)) ↑ FILTER σ(city = 'Mumbai') ← WHERE applied AFTER join in naïve plan ↑ JOIN ⊳⊲ (c.customer_id = o.customer_id) ↑ ↑ Customers Orders

Stage 4: Query Rewriting (Logical Optimization)

The rewriter applies equivalence-preserving transformations to reduce cost without changing the result. This is rule-based and always applied before cost-based optimization.

Predicate Pushdown (Most Important)

-- Before pushdown: filter after join → all rows joined, then filtered JOIN(c.customer_id = o.customer_id) ├── Customers (all 10M rows) ← wasteful └── Orders (all 50M rows) ↑ σ(city = 'Mumbai') applied here -- After pushdown: filter before join → reduce rows entering join JOIN(c.customer_id = o.customer_id) ├── σ(city='Mumbai') Customers → maybe 500K rows (5% of 10M) └── Orders → join now processes far fewer combinations

Projection Pushdown

-- Carry only needed columns through the plan -- If we need only c.name and SUM(o.total): π(customer_id, name) Customers ← drop email, phone, address early π(customer_id, total) Orders ← drop order_date, status, etc. early

Subquery Unnesting

-- Correlated EXISTS → semi-join SELECT c.* FROM Customers c WHERE EXISTS (SELECT 1 FROM Orders o WHERE o.customer_id = c.customer_id); -- Rewritten internally as: SELECT DISTINCT c.* FROM Customers c JOIN Orders o ON c.customer_id = o.customer_id; -- Semi-join: returns each customer at most once, short-circuits on first match

Stage 5: Physical Plan (Cost-Based Optimization)

The physical planner maps each logical operation to real algorithms and chooses the combination with the lowest estimated cost.

Table Access Methods

MethodWhen ChosenCost Characteristic
Sequential ScanLarge fraction of table needed; no selective indexO(n) predictable sequential I/O
Index ScanHighly selective WHERE predicate with matching B-tree indexO(log n) index + random I/O per row
Index-Only ScanAll needed columns are in the index (covering index)O(log n) — no heap reads at all
Bitmap Index ScanModerate selectivity; OR of multiple indexesBuilds bitmap in memory; batched heap I/O

Join Algorithms

AlgorithmComplexityWhen Best
Nested Loop JoinO(n × m) or O(n log m) with indexSmall outer, indexed inner (OLTP lookups)
Hash JoinO(n + m) averageLarge unsorted tables, no useful index
Merge JoinO(n + m)Both inputs already sorted on join key
-- How the optimizer estimates costs: -- Cost = (pages_to_read × page_cost) + (rows_to_process × cpu_cost) -- PostgreSQL cost parameters (in arbitrary units): -- seq_page_cost = 1.0 (one sequential I/O page read) -- random_page_cost = 4.0 (random I/O is 4x more expensive by default) -- cpu_tuple_cost = 0.01 (processing one tuple) -- cpu_index_tuple_cost = 0.005 -- Cardinality estimates use table statistics: SELECT attname, n_distinct, correlation, most_common_vals, histogram_bounds FROM pg_stats WHERE tablename = 'orders'; -- n_distinct: -0.05 means ~5% of rows are distinct -- correlation: 1.0 = perfectly correlated with physical order (fast index scan) -- 0.0 = random (index scan = many random I/Os) -- Update statistics when distribution changes significantly ANALYZE orders;

Stage 6: Execution

The executor runs the physical plan using a volcano/iterator model: each operator is a node in a tree that pulls rows from its children on demand. Results stream to the client as they're produced (important for LIMIT optimizations).

-- Volcano model: each operator has a "get_next()" method -- Parent calls get_next() on child; child calls get_next() on ITS child -- Results flow bottom-up through the tree on demand -- This means LIMIT can be very efficient: SELECT * FROM Orders ORDER BY order_date DESC LIMIT 10; -- If order_date is indexed, only 10 rows are read via index scan -- Without the index, a full sort is needed before LIMIT can be applied -- Parallel execution (PostgreSQL parallel query): -- Nodes like Parallel Seq Scan, Parallel Hash Join split work across workers -- Gather node at top collects results from all workers EXPLAIN SELECT COUNT(*) FROM Orders WHERE status = 'delivered'; -- May show: Finalize Aggregate → Gather → Partial Aggregate → Parallel Seq Scan

Reading EXPLAIN Output

EXPLAIN (ANALYZE, BUFFERS) SELECT c.name, SUM(o.total) FROM Customers c JOIN Orders o ON c.customer_id = o.customer_id WHERE c.city = 'Mumbai' GROUP BY c.name; -- Sample output: HashAggregate (cost=15420.23..15521.43 rows=10120 width=42) (actual time=234.123..236.891 rows=8543 loops=1) Group Key: c.name Buffers: shared hit=4200 read=1034 -> Hash Join (cost=1234.10..14234.00 rows=47449 width=42) (actual time=12.340..198.234 rows=43218 loops=1) Hash Cond: (o.customer_id = c.customer_id) Buffers: shared hit=3100 read=934 -> Seq Scan on orders o (cost=0.00..10234.00 rows=500000 width=16) (actual time=0.023..102.341 rows=500000 loops=1) -> Hash (cost=1098.43..1098.43 rows=10854 width=34) (actual time=11.234..11.234 rows=10854 loops=1) -> Index Scan using idx_city on customers c (cost=0.56..1098.43 rows=10854) Index Cond: (city = 'Mumbai') -- Key fields to check: -- cost=X..Y: X=startup cost, Y=total cost (in cost units, not milliseconds) -- rows= ESTIMATED vs actual rows in ANALYZE output -- loops=: how many times this node ran -- Buffers: shared hit=pages from cache, read=pages from disk -- Large gap between estimated rows and actual rows = stale statistics; run ANALYZE
Most important EXPLAIN signal: A large discrepancy between estimated and actual rows is the #1 cause of bad query plans. If estimated=100 but actual=500000, the optimizer grossly underestimated and may have chosen a nested loop instead of a hash join. Fix by running ANALYZE table_name to update statistics.

Query Plan Hints

-- PostgreSQL: disable specific plan methods (for debugging/forcing) SET enable_nestloop = OFF; -- force hash or merge join SET enable_hashjoin = OFF; -- force merge or nested loop SET enable_seqscan = OFF; -- force index usage (if index exists and selectivity is high) -- MySQL: explicit hints SELECT /*+ INDEX(orders idx_order_date) */ * FROM orders WHERE order_date > '2024-01-01'; SELECT /*+ NO_INDEX(orders) */ * FROM orders WHERE status = 'pending'; -- PostgreSQL pg_hint_plan extension SELECT /*+ HashJoin(c o) SeqScan(o) */ c.name, SUM(o.total) FROM Customers c JOIN Orders o ON c.customer_id = o.customer_id GROUP BY c.name;
Optimization hierarchy (most to least impactful):
  1. Write the query correctly — avoid functions on indexed columns in WHERE, use SARGable predicates
  2. Update statistics (ANALYZE)
  3. Add missing indexes on join columns, filter columns
  4. Rewrite the query (CTE vs subquery, EXISTS vs IN)
  5. Adjust cost parameters only if hardware characteristics differ significantly
  6. Use plan hints as last resort — they become maintenance debt

Summary

  • SQL queries go through: Parse → Bind → Logical Plan → Rewrite → Physical Plan → Execute
  • Parsing catches syntax errors; Binding catches semantic errors (bad column/table names)
  • Logical optimization: predicate pushdown, projection pushdown, subquery unnesting are rule-based
  • Physical optimization: cost-based — uses table statistics to choose access paths and join algorithms
  • EXPLAIN ANALYZE shows estimated vs actual row counts — mismatches indicate stale statistics
  • Run ANALYZE when data distribution changes significantly to keep the optimizer informed
