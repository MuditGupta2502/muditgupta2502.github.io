Database Security: Protecting Data at Every Layer

Security & Compliance | 11 min read

Why Database Security Matters

Databases often contain sensitive data — personal information, financial records, trade secrets. A breach can result in financial loss, legal liability, reputational damage, and loss of customer trust.

Security Must Address:

  • Confidentiality: Data visible only to authorized users
  • Integrity: Data remains accurate and unchanged by unauthorized parties
  • Availability: Authorized users can access data when needed

Authentication

Authentication verifies the identity of users before granting access.

Methods:

  • Username/Password: Most common, should be hashed and salted
  • Multi-Factor Authentication (MFA): Requires additional verification (OTP, biometrics)
  • OAuth/LDAP: Centralized authentication through external providers
  • Certificate-Based: X.509 certificates for secure connections

Strong Password Policies

-- Requirements for strong passwords
- Minimum 12 characters
- Mix of uppercase, lowercase, numbers, symbols
- No dictionary words or personal information
- Regular password changes (e.g., 90 days)
- Password history (prevent reuse)

Authorization & Access Control

Authorization determines what authenticated users can do. The principle of least privilege (PoLP) ensures users get only the minimum necessary access.

Role-Based Access Control (RBAC)

-- Define roles with specific permissions
CREATE ROLE admin;
CREATE ROLE analyst;
CREATE ROLE viewer;

-- Grant permissions to roles
GRANT SELECT, INSERT, UPDATE, DELETE ON database.* TO admin;
GRANT SELECT ON database.* TO analyst;
GRANT SELECT ON reports.* TO viewer;

-- Assign roles to users
GRANT admin TO user1;
GRANT analyst TO user2;
GRANT viewer TO user3;

Attribute-Based Access Control (ABAC)

Grants access based on user attributes, resource attributes, and environmental conditions. 

-- Grant select on employees table only to HR department users during business hours
IF (user.department = 'HR' AND time BETWEEN '9:00' AND '17:00')
  ALLOW SELECT ON Employees

Column-Level Security

Hide sensitive columns from unauthorized users. A customer service representative sees names but not credit card numbers. 

-- Create a view that hides credit card numbers
CREATE VIEW SafeCustomers AS
SELECT customer_id, name, email, phone
FROM Customers;

GRANT SELECT ON SafeCustomers TO customer_service_role;

Row-Level Security (RLS)

Control row access based on user attributes. A sales manager sees only their team's data. 

-- SQL Server example
CREATE SECURITY POLICY SalesRLS
ADD FILTER PREDICATE dbo.fn_securitypredicate(user_id)
ON dbo.Sales
WITH (STATE = ON);

Encryption

Data at Rest

Encrypt data stored on disk. Even if someone gains physical access to storage devices, data remains unreadable. 

-- Transparent Data Encryption (TDE) in SQL Server
ALTER DATABASE MyDatabase SET ENCRYPTION ON;

Data in Transit

Encrypt data traveling between client and database server using TLS/SSL. 

-- Connect to MySQL with SSL
mysql -h server.com -u user -p --ssl-mode=REQUIRED

Data in Use

Encrypt sensitive data (PII, credit cards) at the application level before storing. 

-- Encrypt before storing
CREATE TABLE Customers (
  customer_id INT,
  name VARCHAR(100),
  ssn VARBINARY(255), -- Encrypted SSN
  encrypted_at TIMESTAMP
);

-- Application encrypts SSN before INSERT/UPDATE

SQL Injection Prevention

SQL injection is a common attack where malicious SQL code is executed through user input.

Vulnerable Code: 

-- DO NOT DO THIS
$user_id = $_GET['id'];
$query = "SELECT * FROM users WHERE id = " . $user_id;
// Attacker can pass: 1' OR '1'='1

Secure Approach: Parameterized Queries 

-- Use prepared statements
$stmt = $conn->prepare("SELECT * FROM users WHERE id = ?");
$stmt->bind_param("i", $user_id);
$stmt->execute();

Audit Logging

Log all database activities to detect and investigate unauthorized access. 

-- Enable audit logging (SQL Server)
CREATE SERVER AUDIT AuditLog
TO FILE (FILEPATH = 'C:\Audit\')
WITH (QUEUE_DELAY = 1000, ON_FAILURE = CONTINUE);

-- Audit successful logins and data access
CREATE DATABASE AUDIT SPECIFICATION AuditDataAccess
FOR SERVER AUDIT AuditLog
ADD (SELECT, DELETE, UPDATE ON DATABASE::MyDatabase BY PUBLIC);

Backup & Disaster Recovery

Regular encrypted backups ensure data can be restored after incidents like ransomware attacks or hardware failures. 

-- Regular backup schedule
BACKUP DATABASE MyDatabase
TO DISK = 'C:\Backups\MyDatabase_full.bak'
WITH ENCRYPTION (ALGORITHM = AES_256,
  SERVER CERTIFICATE = MasterKey);

Network Security

  • Firewall: Restrict database ports (default 3306 for MySQL) to trusted IP ranges
  • VPN: Connect to databases through secure tunnels
  • Disable External Access: Don't expose databases directly to internet
  • Network Segmentation: Separate database servers in isolated networks

Compliance Standards

GDPR (General Data Protection Regulation): EU regulation protecting personal data. Requires consent, encryption, right to deletion (right to be forgotten).

HIPAA (Health Insurance Portability and Accountability Act): Protects healthcare data in the US.

PCI DSS (Payment Card Industry Data Security Standard): Protects credit card data.

SOC 2 (Service Organization Control): Audit standard for service providers.

Security Best Practices Summary

  • Change default credentials immediately
  • Keep DBMS updated with security patches
  • Use strong authentication and authorization
  • Encrypt sensitive data (at rest, in transit, in use)
  • Use parameterized queries to prevent SQL injection
  • Enable audit logging for compliance
  • Implement principle of least privilege
  • Regular security testing and penetration testing
  • Regular encrypted backups with off-site copies
  • Monitor database for suspicious activity

Conclusion

Database security is not a one-time implementation but an ongoing process. By implementing layered security controls, enforcing compliance standards, and maintaining vigilant monitoring, you can significantly reduce the risk of data breaches and protect sensitive information.
