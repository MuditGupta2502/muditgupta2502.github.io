Database Security, Transactions & Concurrency Control

Security & Reliability | 25 min read

Part 1: Transactions and ACID

What Is a Transaction?

A transaction is a sequence of one or more database operations that must be executed as a single logical unit of work. Either all operations succeed (COMMIT), or none of them take effect (ROLLBACK). Transactions are the fundamental mechanism for maintaining data integrity in the presence of errors and concurrent access.

-- Classic bank transfer: must be atomic BEGIN; UPDATE Accounts SET balance = balance - 500 WHERE account_id = 1; -- debit UPDATE Accounts SET balance = balance + 500 WHERE account_id = 2; -- credit COMMIT; -- both succeed together -- If anything fails between the two statements: ROLLBACK; -- balance returns to pre-transaction state; money neither lost nor created

ACID Properties

Atomicity

A transaction is atomic: either all of its operations are applied, or none are. If the server crashes after the debit but before the credit, the DBMS recovery system uses the transaction log to roll back the partial transaction on restart. The debit either never happened, or both operations complete.

Consistency

A transaction takes the database from one valid state to another valid state. All constraints (keys, foreign keys, checks, triggers) must hold true before and after the transaction. If a transaction would violate a constraint, it is aborted.

Isolation

Concurrent transactions behave as if they execute serially — one at a time. Uncommitted changes from one transaction should not be visible to another. In practice, isolation is tunable: stronger isolation is safer but slower; weaker isolation is faster but can produce anomalies.

Durability

Once a transaction is committed, it persists even if the system crashes immediately after. The DBMS achieves durability by writing to a Write-Ahead Log (WAL) before modifying actual data pages. On recovery, the WAL is replayed to restore the committed state.

Transaction Isolation Levels and Read Anomalies

SQL defines four isolation levels. Higher isolation levels prevent more anomalies but require more locking or versioning overhead.

Anomalies Explained

  • Dirty Read: Transaction T2 reads a value written by T1 before T1 commits. If T1 rolls back, T2 has read data that never existed.
  • Non-Repeatable Read: T2 reads a row, T1 updates and commits that row, T2 reads again and sees a different value.
  • Phantom Read: T2 executes a range query, T1 inserts a row matching the range and commits, T2 re-executes the same query and sees an additional row.
  • Lost Update: T1 and T2 both read a value, compute new values, and write back. T1's write is overwritten by T2 without T1's update being applied.

Concurrency Control: Locking

Two-Phase Locking (2PL) is the classic concurrency control protocol. Each transaction has two phases:

  1. Growing phase: The transaction acquires locks but does not release any.
  2. Shrinking phase: The transaction releases locks but acquires no new ones.

2PL guarantees serializability but can lead to deadlocks.

Lock Types

  • Shared lock (S-lock / read lock): Multiple transactions can hold a shared lock on the same row simultaneously. Prevents others from writing, not reading.
  • Exclusive lock (X-lock / write lock): Only one transaction can hold this. Prevents any other reads or writes on the row.

Deadlock

A deadlock occurs when two or more transactions are each waiting for a lock held by the other. The DBMS detects deadlocks using a wait-for graph and resolves them by aborting the cheapest transaction (the victim).

-- Deadlock scenario T1: Lock row A → try to lock row B (waits for T2) T2: Lock row B → try to lock row A (waits for T1) -- Neither can proceed. DBMS detects the cycle and aborts one transaction.

MVCC (Multi-Version Concurrency Control)

PostgreSQL, MySQL InnoDB, and Oracle use MVCC as an alternative to pure locking. When a row is updated, the DBMS does not immediately overwrite it. Instead, it creates a new version of the row and marks the old version as superseded. Readers can access the old version; writers work on the new version. This allows reads and writes to proceed concurrently without blocking each other.

Part 2: Database Security

The CIA Triad in Database Context

  • Confidentiality: Only authorized users can read sensitive data.
  • Integrity: Data can only be modified according to defined rules by authorized parties.
  • Availability: Authorized users can access data when needed. Backups and replication support this.

SQL Injection

SQL injection is the most dangerous database vulnerability. It occurs when user-supplied input is directly interpolated into a SQL string, allowing attackers to modify the query structure.

Never concatenate user input directly into SQL strings. This applies to all languages and all databases.
-- Vulnerable code (Python example) query = "SELECT * FROM Users WHERE username='" + username + "' AND password='" + password + "'" -- If username = "admin'--" -- The query becomes: SELECT * FROM Users WHERE username='admin'--' AND password='...' -- The -- comments out the password check. Admin access granted with any password. -- Destructive injection: username = "'; DROP TABLE Users;--" SELECT * FROM Users WHERE username=''; DROP TABLE Users;--' -- Drops the entire Users table.

Prevention: Prepared Statements / Parameterized Queries

With parameterized queries, the SQL structure is sent to the DBMS first (compiled once), and user values are sent separately as parameters. The query structure cannot be modified by any parameter value.

-- Python with psycopg2 (PostgreSQL) cursor.execute( "SELECT * FROM Users WHERE username = %s AND password_hash = %s", (username, hashed_password) -- parameters are separate from the query string ) -- Java with JDBC PreparedStatement stmt = conn.prepareStatement( "SELECT * FROM Users WHERE username = ? AND password_hash = ?" ); stmt.setString(1, username); stmt.setString(2, hashedPassword); -- PHP with PDO $stmt = $pdo->prepare("SELECT * FROM Users WHERE username = :user AND password_hash = :pass"); $stmt->execute([':user' => $username, ':pass' => $hashedPassword]);

Authentication and Authorization

Authentication — Verifying Identity

Before granting any access, the DBMS must verify who the connecting user is.

  • Password-based auth: Create dedicated database users with strong passwords. Never use root for application connections.
  • Certificate-based auth (SSL/TLS): Client presents a certificate that the DBMS verifies.
  • IAM/SSO integration: Cloud databases (RDS, Cloud SQL) can use IAM roles instead of database passwords.
  • Peer authentication (PostgreSQL): Uses OS user identity on local connections.
-- Create a restricted application user (PostgreSQL) CREATE USER app_user WITH PASSWORD 'SecureRandomPassword123!'; -- Never connect applications with superuser credentials -- Bad practice: -- Connect as: postgres (superuser) -- Good practice: -- Connect as: app_user (minimal privileges)

Authorization — What Users Can Do

Authorization is controlled by the GRANT and REVOKE system. The principle of least privilege: grant only the permissions actually needed, nothing more.

-- Grant DML on specific tables only GRANT SELECT, INSERT, UPDATE ON Students TO app_user; GRANT SELECT ON Departments TO app_user; -- Grant execute on a stored procedure (not direct table access) GRANT EXECUTE ON PROCEDURE enroll_student TO app_user; -- Grant SELECT on a view (not the underlying table) GRANT SELECT ON StudentEnrollmentSummary TO reporting_user; -- Revoke a privilege REVOKE DELETE ON Students FROM app_user; -- Check grants in PostgreSQL SELECT grantee, table_name, privilege_type FROM information_schema.role_table_grants WHERE grantee = 'app_user';

Role-Based Access Control (RBAC)

Managing privileges per-user does not scale. RBAC assigns privileges to roles, then assigns roles to users. When a user's job changes, you change role membership — not individual grants.

-- Create roles CREATE ROLE read_only; CREATE ROLE data_entry; CREATE ROLE db_admin; -- Assign privileges to roles GRANT SELECT ON ALL TABLES IN SCHEMA public TO read_only; GRANT SELECT, INSERT, UPDATE ON Students, Enrollments TO data_entry; GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO db_admin; -- Assign roles to users GRANT read_only TO analyst_user; GRANT data_entry TO enrollment_staff; GRANT db_admin TO dba_user; -- Revoke role from user when role changes REVOKE data_entry FROM enrollment_staff; GRANT read_only TO enrollment_staff;

Row-Level Security (RLS)

Row-level security restricts which rows a user can see or modify, even within a table they have SELECT permissions on. This is useful for multi-tenant databases where each tenant should only see their own data.

-- PostgreSQL row-level security ALTER TABLE Orders ENABLE ROW LEVEL SECURITY; -- Policy: users only see orders belonging to their organization CREATE POLICY orders_tenant_isolation ON Orders USING (org_id = current_setting('app.current_org_id')::INT); -- When querying as a tenant user, only their rows are visible SET app.current_org_id = 42; SELECT * FROM Orders; -- Returns only orders where org_id = 42

Views as a Security Layer

Views provide column-level and row-level filtering without requiring row-level security. The pattern: grant users access to views only; never grant direct table access.

-- Sensitive table CREATE TABLE Employees ( emp_id INT, name TEXT, email TEXT, salary DECIMAL, ssn CHAR(11) ); -- View that hides salary and SSN CREATE VIEW PublicEmployeeInfo AS SELECT emp_id, name, email FROM Employees; -- HR view that shows salary but not SSN CREATE VIEW HREmployeeView AS SELECT emp_id, name, email, salary FROM Employees; -- Grant access to views only GRANT SELECT ON PublicEmployeeInfo TO general_staff; GRANT SELECT ON HREmployeeView TO hr_role; -- No one except superusers can SELECT from Employees directly

Encryption

Encryption in Transit (TLS)

All connections between the application and the database should use TLS (SSL). Without TLS, credentials and query results travel in plaintext over the network.

-- PostgreSQL: require SSL in pg_hba.conf hostssl all all 0.0.0.0/0 scram-sha-256 -- Connection string with SSL required postgresql://app_user:password@db.example.com:5432/university?sslmode=require

Encryption at Rest

Encryption at rest protects data if storage media is stolen or improperly decommissioned. Options:

  • Transparent Data Encryption (TDE): The DBMS engine encrypts all data files automatically. SQL Server, Oracle, MySQL enterprise edition support this.
  • Table/column-level encryption: Specific sensitive columns (SSN, credit card) are encrypted with application keys. The DBMS stores ciphertext.
  • Filesystem encryption: The OS-level (dm-crypt, BitLocker) encrypts the disk the database lives on.
-- PostgreSQL pgcrypto: column-level encryption CREATE EXTENSION IF NOT EXISTS pgcrypto; -- Store encrypted SSN INSERT INTO Employees (emp_id, name, ssn_encrypted) VALUES (1, 'Mudit', pgp_sym_encrypt('123-45-6789', 'secret-key')); -- Decrypt for authorized users SELECT emp_id, name, pgp_sym_decrypt(ssn_encrypted::bytea, 'secret-key') AS ssn FROM Employees WHERE emp_id = 1;

Audit Logging

Audit logs record who did what to which data and when. They are essential for compliance (GDPR, HIPAA, SOX) and for forensic investigation after a security incident.

-- PostgreSQL pgaudit extension (logs specific objects/actions) -- In postgresql.conf: -- shared_preload_libraries = 'pgaudit' -- pgaudit.log = 'read, write, ddl' -- Or implement via triggers CREATE TABLE AuditLog ( log_id SERIAL PRIMARY KEY, table_name TEXT, operation CHAR(1), -- I = insert, U = update, D = delete changed_by TEXT DEFAULT current_user, changed_at TIMESTAMP DEFAULT NOW(), old_data JSONB, new_data JSONB ); CREATE OR REPLACE FUNCTION log_student_changes() RETURNS TRIGGER AS $$ BEGIN IF TG_OP = 'DELETE' THEN INSERT INTO AuditLog(table_name, operation, old_data) VALUES ('Students', 'D', row_to_json(OLD)); ELSIF TG_OP = 'UPDATE' THEN INSERT INTO AuditLog(table_name, operation, old_data, new_data) VALUES ('Students', 'U', row_to_json(OLD), row_to_json(NEW)); END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER students_audit AFTER UPDATE OR DELETE ON Students FOR EACH ROW EXECUTE FUNCTION log_student_changes();

Backup and Recovery

Security includes availability. The ability to recover from data loss or corruption is critical.

  • Full backup: Complete dump of all data. Slowest to create; simplest to restore.
  • Incremental backup: Only changes since the last backup. Faster to create; slower to restore.
  • Point-in-Time Recovery (PITR): Combines a base backup with WAL (transaction log) replay to restore to any moment in time.
  • Replication: A standby server continuously applies changes from the primary. If the primary fails, the standby can take over (failover).
-- PostgreSQL: create a base backup pg_basebackup -h localhost -U replication_user -D /backups/pg_base -Fp -Xs -P -- Restore to a specific point in time -- In postgresql.conf during recovery: -- recovery_target_time = '2026-03-15 14:30:00'
Test your backups regularly. An untested backup is not a backup — it is a hope. The target is a verified RTO (Recovery Time Objective) and RPO (Recovery Point Objective).

Security Checklist

  1. Use parameterized queries everywhere — never concatenate user input into SQL.
  2. Apply least-privilege: application users should have only the permissions they need.
  3. Use roles to manage permissions at scale.
  4. Enforce SSL/TLS for all database connections.
  5. Enable encryption at rest for sensitive data.
  6. Enable audit logging for sensitive tables and privileged operations.
  7. Regularly rotate database passwords and revoke unused accounts.
  8. Keep the DBMS engine patched and updated.
  9. Test backup and recovery procedures on a schedule.
  10. Run applications as non-superuser database accounts.

What You Should Remember

  • ACID ensures correctness of transactions: Atomicity (all or nothing), Consistency (valid state), Isolation (concurrent transactions appear serial), Durability (committed data survives crashes).
  • Isolation levels trade safety for performance. READ COMMITTED is the common default; SERIALIZABLE is safest; READ UNCOMMITTED is rarely used except in specialized read workloads.
  • 2PL guarantees serializability; MVCC allows reads and writes to proceed concurrently without blocking.
  • SQL injection is prevented by parameterized queries, not input sanitization alone.
  • GRANT gives permissions; REVOKE removes them. Use roles (RBAC) to manage access at scale.
  • Views act as a security layer — grant access to views, not base tables, wherever possible.
  • Encrypt data in transit (TLS) and at rest (TDE or column encryption) for sensitive data.
  • Audit logs are mandatory for compliance and forensic investigation.
  • Backups are meaningless unless regularly tested and verified.
