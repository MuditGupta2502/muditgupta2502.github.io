Replication, Fault Tolerance and Distributed Database Architecture

Distributed Systems | 28 min read

Replication maintains multiple copies of data on separate servers. It serves two purposes: fault tolerance (a copy survives if a server crashes) and read scaling (replicas handle reads so the primary handles only writes). The trade-off you accept depends on whether replication is synchronous or asynchronous.

Primary-Replica (Single Leader) Replication

One node (primary/leader) accepts all writes. Other nodes (replicas/standbys) apply those writes asynchronously or synchronously and serve reads.

-- PostgreSQL streaming replication setup: -- Primary: postgresql.conf wal_level = replica -- log enough for replicas max_wal_senders = 5 -- how many replicas can connect wal_keep_size = 256MB -- amount of WAL kept on primary for slow replicas -- Replica: recovery.conf (PG11-) / postgresql.conf (PG12+) primary_conninfo = 'host=db-primary port=5432 user=replication_user' hot_standby = on -- replica can serve read queries while replicating -- Replica lag monitoring: SELECT client_addr, state, sent_lsn, write_lsn, flush_lsn, replay_lsn, (sent_lsn - replay_lsn) AS replication_lag_bytes FROM pg_stat_replication; -- replica: how far behind is my replica? SELECT now() - pg_last_xact_replay_timestamp() AS replication_delay;

Synchronous vs Asynchronous Replication

AsynchronousSynchronous
DurabilityPossible data loss on primary failure (replica may be seconds behind)Zero data loss — replica confirms before primary acknowledges
Write latencyPrimary-only write speed (fast)Network RTT added to every commit (slower)
Replica availabilityWrites proceed even if replica is downWrites BLOCK if the synchronous replica is down
Use caseMost web apps, analytics replicas, read scalingFinancial systems, any data you cannot afford to lose
-- PostgreSQL: configure specific replicas as synchronous synchronous_standby_names = 'FIRST 1 (replica1, replica2)' -- "At least ONE of replica1/replica2 must confirm before commit" -- Or require ALL listed replicas to confirm: synchronous_standby_names = 'ALL (replica1, replica2)' -- Every write waits for both replicas — maximum durability, maximum latency -- Semi-synchronous (MySQL): -- Primary waits for at least one replica to RECEIVE the binlog event (not apply it) -- Faster than full sync; provides durability guarantee against primary loss -- MySQL: plugin: rpl_semi_sync_master_enabled = ON

Failover and Promotion

-- Manual failover (PostgreSQL): -- 1. Stop the primary (or primary crashes) -- 2. Promote the most up-to-date replica: pg_ctl promote -D /var/lib/postgresql/data -- OR create a trigger file if standby_mode is configured touch /tmp/failover.trigger -- 3. Update connection string in application (or use HAProxy/Patroni) -- Automated failover tools: -- Patroni (etcd/Consul/ZooKeeper based): automatic leader election -- Repmgr: lighter-weight, event-driven -- AWS RDS Multi-AZ: fully managed automatic failover (60s typically) -- Patroni cluster setup concept: -- 3 nodes + etcd (distributed consensus) -- Only node elected as leader accepts writes -- Any standby can become leader; configuration changes automatically distributed -- HAProxy routes connections: writes → leader, reads → any replica

Multi-Primary (Multi-Leader) Replication

Multiple nodes accept writes simultaneously. Write conflicts can occur when two primaries receive conflicting updates to the same row.

-- Multi-primary use cases: -- 1. Multi-datacenter writes: each datacenter has a local primary for low-latency writes -- Writes replicate asynchronously across datacenters -- Conflict resolution handles rare cross-DC conflicts -- 2. Offline-first applications (mobile apps): -- Phone app writes locally while offline -- Syncs to server when connectivity restored -- Server must resolve conflicts with server data -- Conflict types: -- Write-write conflict: two primaries update same row simultaneously -- Insert-insert: two primaries insert row with same PK -- Reorder conflict: transactions A then B on primary1, but B then A on primary2 -- Conflict resolution strategies: -- Last-write-wins (LWW): highest timestamp wins (can silently lose writes) -- Custom application logic: registered conflict resolver function -- CRDT types: automatically merge without conflicts (see BASE article)

Leaderless Replication

Cassandra and DynamoDB-style: any node can accept writes. Quorum writes/reads provide tunable consistency.

-- N=3 replicas. Write goes to all 3 directly (no leader routing). -- If W=2: write succeeds when 2 of 3 confirm -- If R=2: read queries 2 of 3 nodes; coordinator returns latest (by version or timestamp) -- Read repair: if 2 replicas return version 5 and 1 returns version 3, -- coordinator writes version 5 to the stale replica in the background -- Hinted handoff: -- If destination node is down, another node stores the data "on its behalf" -- When the down node recovers, hinted handoff delivers the buffered writes -- Limited hint buffer size → stale replicas possible if node is down for long -- Anti-entropy / Merkle trees: -- Background process compares Merkle trees of data ranges between replicas -- Where trees differ, repair by syncing the different segments -- Ensures eventual convergence even without active reads triggering read repair

Split-Brain: The Most Dangerous Failure Mode

-- Split-brain: network partition causes two nodes to BOTH believe they are the leader -- Both accept writes → data divergence that is very hard to reconcile -- Example: -- Primary in DC-East loses network to DC-West -- DC-West's failover system promotes a replica to primary -- Now BOTH DC-East and DC-West believe they are primary -- Both accept writes for the same table -- When network recovers: conflicting writes on both sides -- Prevention (split-brain fencing / STONITH — "Shoot The Other Node In The Head"): -- 1. Quorum: node only acts as leader if it can see majority of nodes -- (so if 3-node cluster partitions to 2+1, only the 2-node side leads) -- 2. Fence the old leader: forcibly revoke its access to storage (SCSI reservation) -- 3. Epoch/term numbering (Raft / Paxos): old leader's commands from "term N" are -- rejected when the new leader is in "term N+1" -- Raft consensus algorithm (used by etcd, Consul, TiKV): -- Nodes elect a leader via majority vote (quorum) -- Leader sends heartbeats to prevent new elections -- If majority don't hear heartbeat → new election -- New leader has term number higher than old; old leader's writes rejected

Replication Topology Patterns

TopologyDescriptionBest For
Single leader + async replicasOne write node; multiple read replicasMost web apps, read-heavy workloads
Single leader + sync standbyOne sync replica for HA; others async for readsHA + read scaling (financial apps)
Multi-leader (active-active)Multiple write nodes in different DCsMulti-region low-latency writes
Leaderless (Dynamo-style)Any node accepts writes; quorum consistencyCassandra, DynamoDB — high availability priority
Chain replicationWrites go head→middle→tail; tail serves readsObject stores (HDFS design), strong consistency
Production recommendation for most applications:
  1. 1 synchronous replica for zero-data-loss HA (auto-fails over in <30s)
  2. 1-2 async read replicas for read traffic
  3. Monitoring of replica lag — alert at >5 seconds
  4. Patroni or managed RDS Multi-AZ for automated failover

Summary

  • Primary-replica replication: single write node, multiple read nodes; async is fast but may lose data
  • Synchronous replication: zero data loss but writes block until replica confirms; adds latency
  • Failover needs automatic promotion (Patroni, PgBouncer, RDS Multi-AZ) for sub-minute recovery
  • Multi-primary enables multi-region writes but requires conflict resolution strategy
  • Leaderless (quorum-based) replication: any node takes writes; W+R>N gives strong consistency
  • Split-brain prevention requires quorum + fencing; Raft/Paxos algorithms handle this with epoch numbers
