DBaaS: AWS RDS, Aurora, Cloud SQL, Azure SQL and CosmosDB

Cloud | 22 min read

Database-as-a-Service (DBaaS) means a cloud provider manages the infrastructure, patching, backups, and high availability, leaving you to focus on schema design and queries. This is the right choice for most teams — but it comes with real trade-offs in control, cost, and portability.

What DBaaS Handles vs What You Still Own

OS patchingYouProvider

Engine upgrades (major)YouYou (must initiate; brief downtime)

AWS RDS

-- RDS: managed relational databases on AWS -- Supported engines: PostgreSQL, MySQL, MariaDB, Oracle, SQL Server -- Key features: -- Multi-AZ deployment: synchronous standby in a different Availability Zone -- Primary fails → DNS failover to standby (typically 60-120 seconds) -- Standby is NOT readable (only for HA, not for reads) -- Read replicas: up to 5 replicas for read scaling -- Asynchronous replication; replica lag usually < 1 second for low-write workloads -- Can promote to standalone instance if primary fails (manual process) -- Cross-region replicas supported (for disaster recovery) -- RDS PITR: automated backups retained 0-35 days -- Restore to any second within retention window -- Restoration creates a NEW DB instance (cannot restore in-place) -- Storage options: -- gp2 (general purpose SSD): balanced performance, burstable -- gp3: baseline 3000 IOPS; independently scale IOPS and throughput -- io1/io2: provisioned IOPS for highest performance (expensive) -- Storage auto scaling: enable to grow automatically when 90% full -- Parameter groups: configure individual engine parameters -- Most static parameters require reboot; dynamic parameters apply immediately -- Custom parameter group for production: -- shared_buffers = {DBInstanceClassMemory/4} -- 25% of instance RAM -- max_connections = LEAST({DBInstanceClassMemory/9531392}, 5000) -- log_min_duration_statement = 1000 -- RDS limitations: -- No superuser (cannot run pg_ctl, access filesystem, install arbitrary extensions) -- Extensions available: pg_stat_statements, postgis, uuid-ossp, pgcrypto, pg_trgm, etc. -- Cannot install custom extensions not in RDS approved list

Amazon Aurora

-- Aurora: AWS-built cloud-native relational database -- Compatible with MySQL 5.7/8.0 and PostgreSQL 14/15 -- Same API and wire protocol as open-source; most apps can connect without changes -- Aurora architecture innovation: -- Traditional RDS: each replica stores a complete copy of data -- Aurora: all instances (writer + readers) share the SAME distributed storage volume -- Aurora storage: -- Storage: 6 copies of data spread across 3 AZs (2 copies per AZ) -- Can withstand: 2 full AZ failures without losing write availability -- Can withstand: 3 full AZ failures without losing read availability -- Storage grows automatically in 10GB increments up to 128TB -- I/O is distributed across 100+ storage nodes → very high throughput -- WAL is the revolution: -- Traditional: engine generates WAL, writes data pages, ships full data pages to replica -- Aurora: writer sends ONLY WAL (redo log) to shared storage -- Replicas apply redo log in place — no data page shipping -- This makes replication lag < 20ms even for write-heavy workloads -- Aurora Global Database: -- Primary region: one writer + up to 15 read replicas -- Secondary regions: up to 5 secondary regions, each with up to 16 read replicas -- Replication lag to secondary regions: typically < 1 second (via storage-level replication) -- Disaster recovery: promote secondary region to primary in ~60 seconds -- Use case: serve reads from closest region (read from EU for EU users, US for US users) -- Aurora Serverless v2: -- Capacity measured in Aurora Capacity Units (ACUs): 1 ACU = 2GB RAM + proportional CPU -- Scales from min_capacity to max_capacity in seconds -- Billing: per ACU-second consumed (not per hour like provisioned) -- Good for: variable/unpredictable workloads, dev/test, intermittent use -- Aurora vs RDS comparison: -- Aurora: 5x faster than MySQL, 3x faster than PostgreSQL (AWS claim for storage-intensive workloads) -- Aurora: higher baseline cost (~20% more than equivalent RDS) -- Aurora: much faster replica failover (< 30 seconds vs 60-120s for RDS Multi-AZ) -- Aurora: can have 15 read replicas (RDS: 5)

Google Cloud SQL and AlloyDB

-- Cloud SQL: managed MySQL, PostgreSQL, and SQL Server on GCP -- Similar feature set to AWS RDS -- Notable: Cloud SQL Proxy for secure connectivity without public IPs -- High availability: regional persistent disk, automatic failover to standby -- Cross-region replicas supported for disaster recovery -- AlloyDB (2022): GCP's Aurora-equivalent for PostgreSQL -- PostgreSQL-compatible; significantly higher performance than Cloud SQL -- Shared log storage architecture (similar to Aurora) -- Built-in ML integration: BigQuery ML integration, real-time inference -- HTAP: handles OLTP and analytics on same instance (columnar in-memory engine) -- Use case: high-performance PostgreSQL with Google Cloud-native features -- Cloud SQL for PostgreSQL notable limits: -- Max 96 vCPUs, 624GB RAM -- Storage: up to 64TB -- Extensions supported: same set as community PostgreSQL; fewer than RDS Aurora

Azure SQL and Azure Cosmos DB

-- Azure SQL Database: managed SQL Server -- Serverless tier: compute auto-pauses after inactivity; billed only when active -- Hyperscale tier: up to 100TB; add read replicas in minutes (named pages service) -- Elastic Pools: multiple databases share one resource pool (cost-efficient for multi-tenant) -- Azure Cosmos DB: globally distributed NoSQL database -- Multi-model: document (MongoDB API), key-value, column-family (Cassandra API), graph (Gremlin) -- Write to any region simultaneously (multi-master write) -- 5 consistency levels (from strongest to weakest): -- Strong: reads always see latest committed write (highest latency) -- Bounded Staleness: reads lag by at most N versions or T seconds -- Session: user sees their own writes, monotonic reads/writes within session -- Consistent Prefix: reads never see out-of-order writes -- Eventual: lowest latency, reads may return non-latest data -- Cosmos DB partitioning: -- Choose a partition key carefully — all queries without partition key = full scan = expensive -- Good partition key: high cardinality, uniform distribution, often in WHERE clause -- E.g., for orders: partition key = customer_id (each customer's orders in one partition) -- SLA: 99.999% availability for multi-region accounts; 10ms read/write p99 latency guaranteed

Self-Hosted vs DBaaS Decision Guide

FactorChoose Self-HostedChoose DBaaS
Team expertiseStrong DBA team availableSmall or DevOps-focused team
Cost sensitivityVery cost-sensitive at scale (DBaaS markup 2-5×)Up to ~$50K/year — DBaaS overhead worth the ops savings
ComplianceOn-premises required (air-gapped, government)Cloud compliances (SOC2, HIPAA, PCI) satisfied by major providers
CustomizationNeed custom extensions, OS access, custom kernel paramsStandard extensions sufficient
Multi-region HAComplex to set up properlyBuilt-in with Aurora Global or Cosmos DB
Vendor lock-in concernHigh portability neededAcceptable to depend on AWS/GCP/Azure
Aurora vs RDS pricing example (2024 ap-south-1):
RDS PostgreSQL db.r6g.2xlarge Multi-AZ: ~$720/month
Aurora PostgreSQL db.r6g.2xlarge (1 writer + 1 reader): ~$860/month
Cost difference: ~$140/month (~$1680/year)
Aurora benefit: faster failover, 15 replicas, global database, auto-scaling storage
Decision: for most production workloads, Aurora's HA and performance justify the cost.

Summary

  • DBaaS off-loads OS, backup, HA, and patching management — the right choice for most teams
  • RDS: standard managed relational DB; Multi-AZ for HA; read replicas for reads; PITR up to 35 days
  • Aurora: shared distributed storage; WAL-only replication; 15 read replicas; <30s failover; Global Database for multi-region
  • Aurora Serverless v2: auto-scales ACUs; ideal for variable workloads with no management overhead
  • Cosmos DB: globally distributed NoSQL; 5 configurable consistency levels; multi-master writes; sub-10ms SLA
  • Self-host when: extreme cost sensitivity at large scale, regulatory air-gap, or need for unsupported customization
