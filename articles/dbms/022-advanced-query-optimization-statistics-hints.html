Advanced Query Optimization: Statistics, Hints and Parallel Query

Query Internals | 25 min read

Modern databases use cost-based optimizers that rely on statistical models to choose plans. When these statistics are stale, skewed, or missing, the optimizer makes poor choices. Understanding how statistics work, where they fail, and how to guide the optimizer is essential for production database tuning.

RBO vs CBO: Two Eras of Optimization

Rule-Based Optimizer (RBO)Cost-Based Optimizer (CBO)
Decision basisFixed priority rulesEstimated row counts × algorithm costs
Uses statistics?NoYes — deeply dependent
Index preferenceAlways prefers index scanMay choose seq scan if selectivity is low
Modern usageLegacy (Oracle pre-8i, some systems)PostgreSQL, MySQL 8+, SQL Server, Oracle 9i+

RBO predictability was appealing (you knew it would always use the index), but it produced terrible plans for queries where a full scan was faster. CBO is smarter but depends on accurate statistics.

What Planner Statistics Actually Are

-- PostgreSQL: statistics visible in pg_stats SELECT tablename, attname, null_frac, avg_width, n_distinct, most_common_vals, most_common_freqs, histogram_bounds, correlation FROM pg_stats WHERE tablename = 'orders' AND attname = 'status'; -- null_frac : 0.02 means 2% of rows are NULL -- avg_width : average byte width of this column's value -- n_distinct : -0.05 means ~5% of rows have distinct values (negative = fraction) -- 50 means exactly 50 distinct values -- most_common_vals : ['pending','delivered','cancelled'] -- most_common_freqs : [0.12, 0.75, 0.13] — 75% are 'delivered' -- histogram_bounds : bucket boundaries for non-common values -- correlation : 1.0 = perfectly correlated with physical storage order -- -1.0 = reverse correlated; 0.0 = random

How the Optimizer Uses These

-- Selectivity estimate for: status = 'delivered' -- Found in most_common_vals → use most_common_freqs[1] = 0.75 -- Estimated rows = 0.75 × total_rows -- Selectivity for: status = 'processing' (not in MCV) -- MCV covers: 0.12 + 0.75 + 0.13 = 1.00 -- Wait, that's 100% — but actually MCVs don't always cover everything -- Real example: 0.85 coverage → remaining fraction = 0.15 -- Non-MCV values split equally over histogram buckets -- Selectivity for range: order_date BETWEEN '2024-01-01' AND '2024-03-31' -- Uses histogram_bounds to estimate fraction of rows in that range -- histogram_bounds might be: ['2023-01-01','2023-07-01','2024-01-01','2024-07-01','2025-01-01'] -- 2024-01-01 to 2024-03-31 covers ~1/4 of the 3rd bucket → selectivity ≈ 0.125

Multi-Column Statistics: The Most Underused Feature

The optimizer assumes column independence by default. If two columns are correlated (e.g., city and region), its estimates will be wildly off.

-- Problem: underestimation due to assumed independence -- Suppose 20% of orders are from Mumbai, 5% have status='premium_delivery' -- The optimizer estimates: 0.20 × 0.05 = 1% of rows match BOTH -- But if premium delivery is only available in Mumbai, the real answer is 5% -- This 5x underestimate causes the optimizer to choose a nested loop (bad) -- Solution: extended statistics on correlated columns CREATE STATISTICS orders_city_status (mcv, dependencies, ndistinct) ON city, status FROM orders; ANALYZE orders; -- populate the new statistics -- Now the optimizer knows about the correlation and estimates correctly

Statistics Staleness: When Plans Go Bad

-- Check when statistics were last updated SELECT schemaname, tablename, last_analyze, last_autoanalyze FROM pg_stat_user_tables WHERE tablename = 'orders'; -- Check if statistics are stale: -- If last_analyze was weeks ago and many rows changed, stats are stale -- Rule of thumb: analyze when >10-20% of rows changed -- Manual analyze: ANALYZE orders; -- update stats for all columns ANALYZE orders(status, customer_id); -- targeted (PostgreSQL 11+) -- Increase statistics target for high-cardinality columns: ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 500; -- Default is 100 histogram buckets; 500 gives better range estimates ANALYZE orders;

Auto-Analyze Configuration

-- PostgreSQL autovacuum / autoanalyze thresholds -- Table is analyzed when: changed_rows > autovacuum_analyze_threshold + -- autovacuum_analyze_scale_factor × n_live_tup SHOW autovacuum_analyze_threshold; -- default: 50 rows SHOW autovacuum_analyze_scale_factor; -- default: 0.2 (20% of table) -- For a 10M row table: triggers after 50 + 0.2 × 10M = 2,000,050 changes -- That's a LOT of change before auto-analyze fires! -- For large tables, set a lower scale factor: ALTER TABLE orders SET (autovacuum_analyze_scale_factor = 0.01); -- 1% = 100K rows

Join Order Optimization: The Exponential Problem

The number of possible join orderings grows exponentially: for n tables, there are n! orderings and even more tree shapes. Optimizers use heuristics.

-- 3 tables: 6 orderings (manageable) -- 5 tables: 120 orderings (fast) -- 8 tables: 40,320 orderings (still fast) -- 12 tables: 479,001,600 orderings (PostgreSQL switches to genetic algorithm!) -- 20 tables: 2.4 × 10^18 orderings (never enumerate all of these) -- PostgreSQL switches to GEQO (genetic algorithm) at join_collapse_limit: SHOW join_collapse_limit; -- default: 8 SHOW geqo_threshold; -- default: 12 -- For queries joining many tables, help the optimizer by ordering FROM clause: -- Put the most selective (smallest result) joins first -- Use explicit JOIN ... ON instead of cross join in WHERE clause

Forcing Join Order

-- PostgreSQL: disable join order reordering SET join_collapse_limit = 1; -- now optimizer uses your FROM clause order SET from_collapse_limit = 1; -- no FROM flattening either SELECT c.name, o.total, p.name FROM Customers c -- start with smallest table JOIN Orders o ON c.customer_id = o.customer_id JOIN Products p ON o.product_id = p.product_id WHERE c.tier = 'gold'; -- Reset after your session RESET join_collapse_limit; RESET from_collapse_limit;

CTEs as an Optimization Fence

-- PostgreSQL pre-12: CTEs were ALWAYS materialized (evaluated once, stored in temp table) -- This prevented predicate pushdown INTO the CTE — a "fence" in the plan -- Example where this matters: WITH active_orders AS ( SELECT * FROM Orders WHERE status = 'active' ) SELECT * FROM active_orders WHERE customer_id = 12345; -- Pre-12 behavior: ALL active orders fetched first, then filtered by customer_id -- Post-12 behavior: CTE is inlined (not materialized) → predicate pushed into CTE -- Result: only orders with status='active' AND customer_id=12345 fetched -- Force materialization in PostgreSQL 12+ (useful for performance isolation): WITH active_orders AS MATERIALIZED ( SELECT * FROM Orders WHERE status = 'active' ) SELECT * FROM active_orders WHERE customer_id = 12345; -- Force inlining (default in PG12+, but explicit if you want to override): WITH active_orders AS NOT MATERIALIZED ( SELECT * FROM Orders WHERE status = 'active' ) SELECT * FROM active_orders WHERE customer_id = 12345;

The SARGable Predicate Principle

A SARGable (Search ARGument able) predicate can use an index. Non-SARGable predicates force a sequential scan.

-- NOT SARGable (index on order_date cannot be used): WHERE YEAR(order_date) = 2024 -- function on column WHERE UPPER(email) = 'USER@EXAMPLE.COM' -- function on column WHERE order_total + 100 > 5000 -- expression on column WHERE CAST(customer_id AS TEXT) = '123' -- implicit cast -- SARGable equivalents: WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01' WHERE email = 'user@example.com' -- store emails lowercase consistently WHERE order_total > 4900 WHERE customer_id = 123 -- use correct type; no cast -- PostgreSQL functional indexes (workaround when function is unavoidable): CREATE INDEX idx_email_lower ON Customers (LOWER(email)); WHERE LOWER(email) = 'user@example.com'; -- now uses the functional index

Parallel Query Execution

-- PostgreSQL parallel query (available since PG9.6) -- Gather / Gather Merge nodes collect results from parallel workers EXPLAIN SELECT COUNT(*), SUM(total) FROM Orders WHERE order_date >= '2024-01-01'; -- With parallel enabled: Finalize Aggregate (cost=...) -> Gather (cost=...) -- collects from workers Workers Planned: 4 -> Partial Aggregate (cost=...) -- each worker computes partial sum -> Parallel Seq Scan on orders -- each worker reads a fraction -- Configuration: SHOW max_parallel_workers_per_gather; -- default: 2 SET max_parallel_workers_per_gather = 4; SHOW min_parallel_table_scan_size; -- tables smaller than this: no parallelism SHOW parallel_tuple_cost; -- cost of passing tuple to Gather node -- Parallel-safe functions (UDFs must be marked PARALLEL SAFE): CREATE FUNCTION expensive_fn(x int) RETURNS int LANGUAGE sql PARALLEL SAFE AS $$ SELECT x * 2 $$;

Partition Pruning

-- For partitioned tables, the optimizer skips irrelevant partitions entirely CREATE TABLE Orders ( order_id BIGSERIAL, order_date DATE NOT NULL, total NUMERIC ) PARTITION BY RANGE (order_date); CREATE TABLE Orders_2024_Q1 PARTITION OF Orders FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE Orders_2024_Q2 PARTITION OF Orders FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); -- ... etc -- Query that benefits from partition pruning: SELECT SUM(total) FROM Orders WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'; -- EXPLAIN shows only Orders_2024_Q1 is scanned — Q2, Q3, Q4 partitions are skipped -- This is "static pruning" (happens at plan time) -- Dynamic partition pruning (PostgreSQL 11+): also happens at runtime for parameterized queries -- Useful in prepared statements where partition key is a bind variable
Common pitfall — index on expression vs column: If you store order_date as TEXT and write WHERE order_date > '2024-01-01', the comparison uses string semantics ('2024-01-01' < '2025-01-01' works, but '9' > '20' alphabetically). Always use proper DATE/TIMESTAMP types so the optimizer can estimate cardinality correctly and range comparisons return correct results.

Summary

  • CBO chooses plans based on estimated row counts multiplied by algorithm costs
  • Statistics must be fresh — use ANALYZE regularly; tune autovacuum thresholds for large tables
  • Column independence assumption breaks for correlated columns; use CREATE STATISTICS to fix
  • Make predicates SARGable: avoid functions/expressions on the indexed column in WHERE
  • CTEs in PostgreSQL 12+ are inlined by default (no longer an optimization fence)
  • Parallel query handles analytical queries automatically once workers_per_gather > 1
  • Partitioned tables give free partition pruning when partition key is in the WHERE clause
