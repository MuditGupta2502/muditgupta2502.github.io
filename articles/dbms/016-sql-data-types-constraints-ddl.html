SQL Data Types, Constraints & DDL In Depth

SQL Fundamentals | 25 min read

Choosing the right data type and constraint for each column is one of the most impactful design decisions in database engineering. The wrong types mean wasted storage, slow queries, and subtle bugs. Missing constraints mean corrupted data. This article goes deep on every major SQL data type category and every constraint type, with real-world examples showing why each choice matters.

Numeric Types

TypeStorageRangeUse When
TINYINT1 byte-128 to 127 (or 0 to 255 UNSIGNED)Flags, statuses, small counts
SMALLINT2 bytes-32,768 to 32,767Age, small dimension counts
INT / INTEGER4 bytes-2.1B to 2.1BGeneral-purpose IDs, quantities
BIGINT8 bytes-9.2 × 10¹⁸ to 9.2 × 10¹⁸Large IDs, timestamp ms, financial totals
DECIMAL(p,s)VariableExact, up to p digits total, s after decimalMoney, prices — never use FLOAT for money!
FLOAT / REAL4 bytesApprox ±3.4 × 10³⁸Scientific data, GPS coords (acceptable imprecision)
DOUBLE PRECISION8 bytesApprox ±1.8 × 10³⁰⁸Higher-precision floating point
Critical: Never use FLOAT or DOUBLE for monetary values. Floating-point arithmetic has rounding errors: 0.1 + 0.2 = 0.30000000000000004 in IEEE 754. Use DECIMAL(10,2) for prices. E.g. price DECIMAL(10,2) NOT NULL CHECK(price >= 0).

Sequences and Auto-Increment

-- MySQL / MariaDB CREATE TABLE Orders ( order_id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY, ... ); -- PostgreSQL: SERIAL (creates an implicit sequence) or BIGSERIAL CREATE TABLE Orders ( order_id BIGSERIAL PRIMARY KEY, -- equivalent to BIGINT DEFAULT nextval(...) ... ); -- PostgreSQL IDENTITY (ANSI standard, preferred over SERIAL) CREATE TABLE Orders ( order_id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, ... ); -- When to use BIGINT not INT for PKs: -- If table can exceed 2 billion rows, or IDs are generated from sharding/distributed systems

String Types

TypeStorageUse When
CHAR(n)Exactly n bytes (padded)Fixed-length codes: country codes (CHAR(2)), status flags (CHAR(1))
VARCHAR(n)Actual length + 1-2 bytesVariable-length strings up to n: names, emails, addresses

-- Good practice email VARCHAR(254) NOT NULL, -- 254 is RFC 5321 max email length country_code CHAR(2) NOT NULL, -- ISO 3166-1 alpha-2: always exactly 2 chars product_name VARCHAR(255) NOT NULL, description TEXT, -- PostgreSQL has no VARCHAR length limit effectively -- MySQL: TEXT columns cannot have DEFAULT values or be fully indexed -- Use VARCHAR over TEXT where you need indexing or defaulting

Character Set and Collation

-- MySQL: Always use utf8mb4, not utf8 (which MySQL calls utf8 but is actually limited UTF-8) CREATE TABLE Users ( username VARCHAR(50) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci NOT NULL ); -- utf8mb4 supports full Unicode including emoji (4-byte sequences) -- utf8mb4_unicode_ci: case-insensitive, Unicode-aware collation -- PostgreSQL uses UTF-8 by default; use ICU collations for locale awareness CREATE TABLE Users ( username TEXT COLLATE "en-US-x-icu" NOT NULL );

JSON and Semi-Structured Types

-- PostgreSQL JSON types JSON -- stored as text, no indexing on contents JSONB -- stored as binary, supports GIN indexes, faster queries — prefer JSONB CREATE TABLE Products ( product_id BIGSERIAL PRIMARY KEY, name TEXT NOT NULL, attributes JSONB -- flexible attributes differ per product type ); INSERT INTO Products VALUES (1, 'Sony Headphones', '{"brand":"Sony","impedance":32,"wireless":true,"color":"black"}'); -- Query inside JSONB SELECT name, attributes->>'brand' AS brand FROM Products WHERE attributes->>'wireless' = 'true'; -- GIN index makes JSONB queries fast CREATE INDEX ON Products USING GIN(attributes); -- MySQL JSON type (introduced in 5.7) CREATE TABLE Events ( event_id INT PRIMARY KEY, payload JSON NOT NULL ); -- Extract with JSON_EXTRACT or -> SELECT JSON_EXTRACT(payload, '$.user_id') FROM Events;

UUID Types

-- PostgreSQL native UUID type CREATE EXTENSION IF NOT EXISTS "uuid-ossp"; CREATE TABLE Sessions ( session_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), user_id BIGINT NOT NULL, expires_at TIMESTAMPTZ NOT NULL ); -- Or use gen_random_uuid() (built-in since PostgreSQL 13) session_id UUID PRIMARY KEY DEFAULT gen_random_uuid() -- When to use UUID vs BIGINT: -- UUID: distributed systems (no coordination needed for ID generation), public-facing IDs -- BIGINT: simpler, smaller (8 bytes vs 16), faster B-tree indexing, internal-only IDs -- UUIDv7 (ordered): retains UUID benefits but generates time-ordered values for better index locality

Constraints In Depth

PRIMARY KEY

-- Single column PK CREATE TABLE Customers ( customer_id BIGSERIAL PRIMARY KEY, ... ); -- Composite PK (junction table) CREATE TABLE Enrollments ( student_id INT NOT NULL, course_id INT NOT NULL, enrolled_on DATE NOT NULL, PRIMARY KEY (student_id, course_id) ); -- Column-level vs Table-level constraint syntax -- Column-level: right after the column type -- Table-level: required for composite constraints; also for named constraints CONSTRAINT pk_enrollments PRIMARY KEY (student_id, course_id)

FOREIGN KEY & Referential Actions

CREATE TABLE Orders ( order_id BIGSERIAL PRIMARY KEY, customer_id BIGINT NOT NULL, CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) REFERENCES Customers(customer_id) ON DELETE RESTRICT -- prevent deleting customer who has orders ON UPDATE CASCADE -- if customer_id changes, update orders too ); -- ON DELETE / ON UPDATE options: -- RESTRICT: Raise error if referenced row would be deleted/updated (default in most DBs) -- CASCADE: Automatically delete/update child rows -- SET NULL: Set FK to NULL in child rows (FK column must be nullable) -- SET DEFAULT: Set FK to its default value -- NO ACTION: Like RESTRICT but checked at end of transaction (PostgreSQL distinction) -- Real-world examples: -- Order line items: ON DELETE CASCADE (delete order → delete its line items) -- Audit logs: ON DELETE SET NULL (user deleted → keep log, set user_id = NULL) -- Category parent: ON DELETE RESTRICT (can't delete parent if children exist)

UNIQUE Constraint

-- Single column unique ALTER TABLE Users ADD CONSTRAINT uq_users_email UNIQUE (email); -- Composite unique (unique combination, not individual columns) CREATE TABLE Bookings ( room_id INT NOT NULL, date DATE NOT NULL, user_id INT NOT NULL, UNIQUE (room_id, date) -- one booking per room per date ); -- Partial unique index (PostgreSQL) — unique only when condition is met CREATE UNIQUE INDEX uq_active_username ON Users(username) WHERE deleted_at IS NULL; -- Multiple users can have the same username only if they're deleted (soft-delete pattern)

CHECK Constraint

-- Domain checks CREATE TABLE Products ( product_id BIGSERIAL PRIMARY KEY, price DECIMAL(10,2) NOT NULL, discount DECIMAL(5,2) NOT NULL DEFAULT 0, stock INT NOT NULL DEFAULT 0, status VARCHAR(20) NOT NULL, CONSTRAINT chk_price_positive CHECK (price > 0), CONSTRAINT chk_discount_range CHECK (discount >= 0 AND discount <= price), CONSTRAINT chk_stock_nonneg CHECK (stock >= 0), CONSTRAINT chk_status_valid CHECK (status IN ('active','inactive','discontinued')) ); -- Cross-column check CREATE TABLE Events ( event_id INT PRIMARY KEY, start_time TIMESTAMPTZ NOT NULL, end_time TIMESTAMPTZ NOT NULL, CONSTRAINT chk_event_duration CHECK (end_time > start_time) );

NOT NULL and DEFAULT

CREATE TABLE Articles ( article_id BIGSERIAL PRIMARY KEY, title TEXT NOT NULL, -- always required content TEXT NOT NULL, published BOOLEAN NOT NULL DEFAULT FALSE, -- safe default view_count INT NOT NULL DEFAULT 0, -- initialized to 0 created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), deleted_at TIMESTAMPTZ NULL -- NULL = not deleted (soft delete) );

DDL Operations: ALTER TABLE

-- Add a column ALTER TABLE Users ADD COLUMN phone VARCHAR(20); -- Rename a column (PostgreSQL, MySQL 8+) ALTER TABLE Users RENAME COLUMN phone TO phone_number; -- Change data type (be careful — may require data migration) ALTER TABLE Products ALTER COLUMN price TYPE NUMERIC(12,2); -- Add a constraint after creation ALTER TABLE Orders ADD CONSTRAINT chk_total_positive CHECK (total > 0); -- Drop a constraint ALTER TABLE Orders DROP CONSTRAINT chk_total_positive; -- Add foreign key to existing table ALTER TABLE OrderItems ADD CONSTRAINT fk_orderitems_order FOREIGN KEY (order_id) REFERENCES Orders(order_id) ON DELETE CASCADE; -- Drop a column (dangerous if referenced by FK or app code) ALTER TABLE Users DROP COLUMN legacy_field; -- Production tip: in large tables, adding NOT NULL columns requires careful strategy -- In PostgreSQL: add the column with a DEFAULT first (non-blocking), then set NOT NULL (takes lock) ALTER TABLE BigTable ADD COLUMN created_at TIMESTAMPTZ DEFAULT NOW(); ALTER TABLE BigTable ALTER COLUMN created_at SET NOT NULL;

Schema Design Pattern: E-Commerce Orders

CREATE TABLE Customers ( customer_id BIGSERIAL PRIMARY KEY, email VARCHAR(254) UNIQUE NOT NULL, name VARCHAR(200) NOT NULL, phone VARCHAR(20), created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), CONSTRAINT chk_email_format CHECK (email ~* '^[^@]+@[^@]+\.[^@]+$') ); CREATE TABLE Products ( product_id BIGSERIAL PRIMARY KEY, sku VARCHAR(50) UNIQUE NOT NULL, name TEXT NOT NULL, price DECIMAL(10,2) NOT NULL CHECK (price > 0), stock INT NOT NULL DEFAULT 0 CHECK (stock >= 0), is_active BOOLEAN NOT NULL DEFAULT TRUE ); CREATE TABLE Orders ( order_id BIGSERIAL PRIMARY KEY, customer_id BIGINT NOT NULL REFERENCES Customers(customer_id) ON DELETE RESTRICT, status VARCHAR(20) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','confirmed','shipped','delivered','cancelled')), total DECIMAL(10,2) NOT NULL CHECK (total >= 0), created_at TIMESTAMPTZ NOT NULL DEFAULT NOW() ); CREATE TABLE OrderItems ( order_item_id BIGSERIAL PRIMARY KEY, order_id BIGINT NOT NULL REFERENCES Orders(order_id) ON DELETE CASCADE, product_id BIGINT NOT NULL REFERENCES Products(product_id) ON DELETE RESTRICT, quantity INT NOT NULL CHECK (quantity > 0), unit_price DECIMAL(10,2) NOT NULL CHECK (unit_price > 0), -- snapshot at order time UNIQUE (order_id, product_id) -- same product appears once per order );

Summary

  • Use DECIMAL for money, BIGINT or UUID for distributed IDs, TIMESTAMPTZ for timestamps
  • Constraints are the database's last line of defense — use CHECK, NOT NULL, and UNIQUE liberally
  • FK referential actions (CASCADE, RESTRICT, SET NULL) represent business rules — choose deliberately
  • Prefer named constraints (CONSTRAINT name ...) so you can drop/modify them in migrations
  • On large production tables, schema changes need careful query planning to avoid long locks
