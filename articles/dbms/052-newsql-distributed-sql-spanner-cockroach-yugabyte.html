NewSQL and Distributed SQL: Spanner, CockroachDB and YugabyteDB

Distributed Systems | 18 min read

NewSQL aims to keep relational SQL and ACID while scaling horizontally like NoSQL. It relies on consensus replication and distributed transaction protocols under the hood.

Core Architecture Concepts

-- Data split into ranges/tablets/shards -- Each range replicated across nodes (typically 3 replicas) -- One replica elected leader using Raft/Paxos -- Writes go to leader -> replicated quorum -> commit -- Reads can be leader or follower reads depending on consistency needs

System Highlights

-- Google Spanner: -- Globally distributed SQL with external consistency -- Uses TrueTime API for globally ordered commits -- CockroachDB: -- Postgres wire compatibility, Raft replication, SERIALIZABLE isolation by default -- YugabyteDB: -- Postgres-compatible YSQL layer over distributed DocDB storage -- Strong for geo-distributed transactional workloads

When to Use

  • Need multi-region writes with strong consistency.
  • Need high availability with automatic failover and no single primary bottleneck.
  • Can accept added operational and latency complexity compared to single-region PostgreSQL.

Summary

  • Distributed SQL solves global scale + SQL consistency but is not a free upgrade.
  • Latency of consensus and cross-region links must be considered at design time.
