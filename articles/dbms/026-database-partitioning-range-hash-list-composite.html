Database Partitioning: Range, Hash, List and Composite Strategies

Scaling & Architecture | 25 min read

Partitioning splits a large table into smaller physical pieces — each called a partition — while the database presents them as a single logical table. This improves query performance (partition pruning skips irrelevant data), maintenance operations (archiving old partitions), and in some systems, parallel execution across partitions.

Why Partition?

  • Performance: Queries with a filter on the partition key skip entire partitions — scanning 1/24 of data instead of all of it
  • Maintenance: Drop a partition (instant) instead of DELETE on 100M rows (hours, LOT of WAL)
  • Storage tiering: Move cold partitions to slower/cheaper storage (tablespace)
  • Bulk load: Load data into a new partition offline, then attach it atomically
  • Parallel query: Multiple workers can scan different partitions simultaneously

Range Partitioning

Rows are distributed based on whether the partition key value falls within a specified value range. Most natural for time-series data.

-- PostgreSQL declarative partitioning (PG10+) CREATE TABLE orders ( order_id BIGSERIAL, order_date DATE NOT NULL, customer_id BIGINT, total NUMERIC(10, 2), status TEXT ) PARTITION BY RANGE (order_date); -- Create partitions per quarter: CREATE TABLE orders_2024_q1 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2024-04-01'); CREATE TABLE orders_2024_q2 PARTITION OF orders FOR VALUES FROM ('2024-04-01') TO ('2024-07-01'); CREATE TABLE orders_2024_q3 PARTITION OF orders FOR VALUES FROM ('2024-07-01') TO ('2024-10-01'); CREATE TABLE orders_2024_q4 PARTITION OF orders FOR VALUES FROM ('2024-10-01') TO ('2025-01-01'); -- Default partition (catches rows not matching any partition): CREATE TABLE orders_other PARTITION OF orders DEFAULT; -- Queries that use partition key get automatic pruning: SELECT SUM(total) FROM orders WHERE order_date >= '2024-04-01' AND order_date < '2024-07-01'; -- Only scans orders_2024_q2 — all other partitions skipped -- Drop old partition instantly (instead of DELETE + VACUUM): DROP TABLE orders_2023_q1; -- microseconds, no WAL bloat -- Detach before drop (keeps the table, just removes it from the parent): ALTER TABLE orders DETACH PARTITION orders_2023_q1;
Range partitioning pitfall — hot partition: With time-based partitioning, all writes always go to the "current" partition (latest time range). This creates an I/O hotspot. Mitigate by using sub-partitioning (partition by date + hash on customer_id) or accepting it — recent data is the hottest anyway, so it benefits from being all in memory in the buffer cache.

Hash Partitioning

Rows are distributed based on a hash function applied to the partition key. Each partition receives a mathematically even share of rows regardless of value distribution.

-- Hash partition by customer_id into 8 buckets: CREATE TABLE customers ( customer_id BIGINT, name TEXT, email TEXT, city TEXT ) PARTITION BY HASH (customer_id); CREATE TABLE customers_p0 PARTITION OF customers FOR VALUES WITH (MODULUS 8, REMAINDER 0); CREATE TABLE customers_p1 PARTITION OF customers FOR VALUES WITH (MODULUS 8, REMAINDER 1); -- ... p2 through p7 -- All 8 partitions receive roughly equal rows automatically -- Good for: distributing workload evenly when no natural range exists -- Hash partitioning does NOT enable partition pruning for queries without the key: SELECT * FROM customers WHERE city = 'Mumbai'; -- scans ALL 8 partitions SELECT * FROM customers WHERE customer_id = 12345; -- prunes to ONE partition

List Partitioning

Rows are assigned to partitions based on discrete values of the partition key. Useful for geographic regions, status codes, or business units.

CREATE TABLE transactions ( trans_id BIGSERIAL, region TEXT NOT NULL, amount NUMERIC, trans_date DATE ) PARTITION BY LIST (region); CREATE TABLE transactions_india PARTITION OF transactions FOR VALUES IN ('north_india', 'south_india', 'east_india', 'west_india'); CREATE TABLE transactions_usa PARTITION OF transactions FOR VALUES IN ('us_east', 'us_west', 'us_central'); CREATE TABLE transactions_europe PARTITION OF transactions FOR VALUES IN ('uk', 'germany', 'france', 'netherlands'); CREATE TABLE transactions_other PARTITION OF transactions DEFAULT; -- GDPR example: isolate EU data in a specific tablespace: CREATE TABLESPACE eu_tablespace LOCATION '/mnt/eu_storage'; ALTER TABLE transactions_europe SET TABLESPACE eu_tablespace; -- Partition pruning for list: SELECT * FROM transactions WHERE region = 'uk' AND trans_date > '2024-01-01'; -- Only transactions_europe is scanned

Composite (Sub-)Partitioning

Partitions can themselves be partitioned — called sub-partitioning. The first level uses one strategy and the second uses another.

-- Partition orders by year (range), then by customer hash (hash): -- This distributes both temporally AND evenly within each time period CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01') PARTITION BY HASH (customer_id); CREATE TABLE orders_2024_p0 PARTITION OF orders_2024 FOR VALUES WITH (MODULUS 4, REMAINDER 0); CREATE TABLE orders_2024_p1 PARTITION OF orders_2024 FOR VALUES WITH (MODULUS 4, REMAINDER 1); CREATE TABLE orders_2024_p2 PARTITION OF orders_2024 FOR VALUES WITH (MODULUS 4, REMAINDER 2); CREATE TABLE orders_2024_p3 PARTITION OF orders_2024 FOR VALUES WITH (MODULUS 4, REMAINDER 3); -- Total: 4 physical partitions per year -- Query pruning cascades: year filter eliminates 2022, 2023, etc.; -- customer_id filter further eliminates ~3 of 4 sub-partitions within 2024

Indexes on Partitioned Tables

TypeDescriptionTrade-offs
Local (Default)One index per partition, only indexes that partition's rowsFast maintenance, supports DROP PARTITION; can't enforce unique across all partitions
GlobalSingle index across all partitions (not yet natively in PostgreSQL)Enables global uniqueness; expensive to maintain; must be rebuilt on partition ops
Partitioned indexPostgreSQL: CREATE INDEX on parent → automatically creates local indexes on all childrenConvenient but still local semantics; no global uniqueness enforcement
-- Create index on the parent table — PostgreSQL creates one per partition automatically: CREATE INDEX idx_orders_customer ON orders (customer_id); -- → Creates: idx_orders_2024_q1_customer, idx_orders_2024_q2_customer, etc. -- Caveat: UNIQUE constraint on a partitioned table must include the partition key: ALTER TABLE orders ADD UNIQUE (order_id, order_date); -- CANNOT enforce uniqueness across partitions without partition key in constraint -- Workaround: use a separate global lookup table or UUID-based keys

Partition Maintenance Patterns

-- Archive old partition (move to cold storage tablespace): ALTER TABLE orders_2022_q1 SET TABLESPACE cold_storage; -- Detach partition for offline bulk modification: ALTER TABLE orders DETACH PARTITION orders_2022_q1 CONCURRENTLY; -- PG14+ -- CONCURRENTLY allows reads/writes to continue during detach -- Reload massaged/updated data, re-attach: -- (modify the standalone table) ALTER TABLE orders ATTACH PARTITION orders_2022_q1 FOR VALUES FROM ('2022-01-01') TO ('2022-04-01'); -- Automate partition creation (cron-based script pattern): -- Every month: create next month's partition in advance -- Prevents insert failures when data is inserted into the future range
Real-world pattern — IoT telemetry table: A sensor data table with 1 billion rows per year partitioned by week gives 52 partitions. Time-range queries touch 1-4 partitions. Weekly partitions can be bulk-loaded via ATTACH. Old partitions (>1 year) can be detached, compressed, and stored in object storage — all with zero downtime.

Summary

  • Range partitioning: ideal for time-series data; enables drop-by-time-range maintenance
  • Hash partitioning: even distribution when no natural range exists; good for even parallel work
  • List partitioning: discrete category values; great for geographic or status-based isolation
  • Composite partitioning combines strategies at two levels — common pattern: range by time + hash by ID
  • Indexes on partitioned tables are local by default; global uniqueness requires partition key in constraints
  • DETACH PARTITION (CONCURRENTLY) enables zero-downtime archival; ATTACH brings data back atomically
