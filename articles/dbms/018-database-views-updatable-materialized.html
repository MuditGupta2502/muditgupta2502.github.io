Database Views: Simple, Updatable, Materialized & Advanced Patterns

SQL Design | 26 min read

A view is a stored query that acts like a virtual table. Views are one of the most powerful tools for schema abstraction, security, and simplification. But beyond the basics, views have subtleties: when can you update through them? How do materialized views handle staleness? This article explores all of it with real production use cases.

Regular Views (Virtual Tables)

A regular view stores a query definition, not data. Every time you query the view, the database executes the underlying query. Views are transparent — the optimizer often pushes predicates down into the view's underlying query.

-- Create a view for active customers with order history summary CREATE VIEW customer_order_summary AS SELECT c.customer_id, c.name, c.email, c.loyalty_tier, COUNT(o.order_id) AS total_orders, COALESCE(SUM(o.total), 0) AS lifetime_value, MAX(o.order_date) AS last_order_date FROM Customers c LEFT JOIN Orders o ON c.customer_id = o.customer_id AND o.status != 'cancelled' GROUP BY c.customer_id, c.name, c.email, c.loyalty_tier; -- Query the view just like a table SELECT * FROM customer_order_summary WHERE loyalty_tier = 'gold' ORDER BY lifetime_value DESC; SELECT name, total_orders FROM customer_order_summary WHERE last_order_date > NOW() - INTERVAL '30 days';

View Benefits

  • Simplification: Complex joins, aggregations, and business logic are hidden behind a simple name
  • Security & access control: Grant access to a view without exposing the underlying tables — including sensitive columns
  • Schema abstraction: The underlying table structure can change without breaking applications that query the view (as long as the view's output columns remain stable)
  • Code reuse: Centralize query logic — fix a calculation in one place instead of in every query
-- Security example: expose only non-sensitive customer info to reporting tool CREATE VIEW public_customers AS SELECT customer_id, name, city, loyalty_tier FROM Customers; -- Email, phone, address never exposed to the reporting user -- Grant the view, not the table GRANT SELECT ON public_customers TO reporting_user;

Updatable Views

A view is updatable if you can run INSERT, UPDATE, or DELETE on it and the database can cleanly propagate that change to the underlying base table. This requires strict conditions.

Conditions for Updatability (Standard SQL)

  • The view is based on a single base table (no joins)
  • No DISTINCT, GROUP BY, HAVING, or window functions in the view's query
  • No set operations (UNION, INTERSECT, EXCEPT)
  • No subqueries in the SELECT list
  • All NOT NULL base columns without defaults are included in the view
-- Simple updatable view CREATE VIEW active_products AS SELECT product_id, name, price, stock FROM Products WHERE is_active = TRUE; -- UPDATE through the view — propagates to Products table UPDATE active_products SET price = price * 1.05 WHERE category = 'Electronics'; -- This works: single base table, no aggregation -- INSERT through the view (note: is_active defaults to TRUE for new rows) INSERT INTO active_products (name, price, stock) VALUES ('New Widget', 29.99, 100); -- DELETE through the view DELETE FROM active_products WHERE stock = 0; -- Non-updatable example (JOIN view) CREATE VIEW order_details AS SELECT o.order_id, c.name, o.total FROM Orders o JOIN Customers c ON o.customer_id = c.customer_id; -- UPDATE order_details SET total = 100 WHERE order_id = 1; → ERROR in most databases

WITH CHECK OPTION

When you update rows through a view, the updated row might no longer satisfy the view's WHERE condition and would "disappear" from the view after the update. WITH CHECK OPTION prevents this by enforcing that all DML operations through the view must keep the row visible in the view.

CREATE VIEW active_products AS SELECT product_id, name, price, stock, is_active FROM Products WHERE is_active = TRUE WITH CHECK OPTION; -- This UPDATE would move the row OUT of the view: UPDATE active_products SET is_active = FALSE WHERE product_id = 42; -- ERROR: new row violates check option for view "active_products" -- The row would fail the WHERE is_active = TRUE condition after the update -- Without WITH CHECK OPTION, the update would succeed silently and the product -- would disappear from the view — surprising and error-prone -- LOCAL vs CASCADED (for nested views) -- WITH LOCAL CHECK OPTION: only the immediate view's condition is checked -- WITH CASCADED CHECK OPTION (default): all nested view conditions are checked

Materialized Views

Unlike regular views, materialized views physically store the query results. They're a form of controlled denormalization: precomputed, stored on disk, queryable with indexes — but potentially stale until refreshed.

-- PostgreSQL materialized view CREATE MATERIALIZED VIEW daily_revenue_by_category AS SELECT DATE_TRUNC('day', o.order_date) AS day, p.category, COUNT(o.order_id) AS order_count, SUM(oi.quantity * oi.unit_price) AS revenue FROM Orders o JOIN OrderItems oi ON o.order_id = oi.order_id JOIN Products p ON oi.product_id = p.product_id WHERE o.status = 'delivered' GROUP BY 1, 2; -- Create index on the materialized view CREATE INDEX ON daily_revenue_by_category(day, category); CREATE INDEX ON daily_revenue_by_category(day); -- Query it like a table — no computation at query time! SELECT * FROM daily_revenue_by_category WHERE day >= '2024-01-01' ORDER BY day, revenue DESC;

Refresh Strategies

-- Manual refresh (blocks reads in PostgreSQL unless CONCURRENTLY) REFRESH MATERIALIZED VIEW daily_revenue_by_category; -- Non-blocking refresh (requires at least one UNIQUE index on the view) REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue_by_category; -- CONCURRENTLY: builds a new version in background, swaps atomically — no read lock -- Scheduled refresh via cron job or pg_cron extension SELECT cron.schedule('refresh-revenue-mv', '0 * * * *', -- every hour 'REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue_by_category'); -- CHECK if a materialized view is out of date SELECT schemaname, matviewname, ispopulated FROM pg_matviews WHERE matviewname = 'daily_revenue_by_category';

Materialized Views in Other Databases

-- Oracle: supports fast (incremental) refresh via materialized view logs CREATE MATERIALIZED VIEW LOG ON Orders WITH ROWID, SEQUENCE (order_id, total); CREATE MATERIALIZED VIEW mv_order_summary REFRESH FAST ON COMMIT -- auto-refresh on each COMMIT AS SELECT customer_id, SUM(total) AS total FROM Orders GROUP BY customer_id; -- SQL Server: "Indexed Views" — automatically maintained as data changes CREATE VIEW v_order_totals WITH SCHEMABINDING AS SELECT customer_id, COUNT_BIG(*) AS cnt, SUM(total) AS total FROM dbo.Orders GROUP BY customer_id; CREATE UNIQUE CLUSTERED INDEX idx_v_order_totals ON v_order_totals (customer_id); -- SQL Server auto-maintains this index as rows are inserted/updated/deleted -- Snowflake / BigQuery: materialized views auto-incrementally refresh CREATE OR REPLACE MATERIALIZED VIEW mv_daily_sales AS SELECT DATE_TRUNC('day', created_at) AS day, SUM(amount) AS revenue FROM sales GROUP BY 1;

Views for Row-Level Security

Before row-level security (RLS) policies were commonplace, views were the standard way to restrict which rows different users could see.

-- View that limits data access to the user's own region (multi-tenant) CREATE VIEW my_region_customers AS SELECT * FROM Customers WHERE region = current_setting('app.user_region'); -- current_setting() reads a PostgreSQL session variable set at login -- Set the session variable when a user connects SET app.user_region = 'APAC'; SELECT * FROM my_region_customers; -- only APAC customers visible -- Modern approach: PostgreSQL Row-Level Security (RLS) policies ALTER TABLE Customers ENABLE ROW LEVEL SECURITY; CREATE POLICY region_isolation ON Customers USING (region = current_setting('app.user_region'));

View Dependency and Schema Management

-- List all views that depend on a table SELECT viewname, definition FROM pg_views WHERE definition ILIKE '%customers%'; -- Dropping a view DROP VIEW customer_order_summary; DROP VIEW IF EXISTS customer_order_summary; DROP VIEW customer_order_summary CASCADE; -- also drops dependent views -- Replacing a view (preserves grants) CREATE OR REPLACE VIEW customer_order_summary AS SELECT ... (new query); -- PostgreSQL: can only change SELECT list, not restructure; use DROP+CREATE for structural changes -- Check view dependencies before altering a base table -- If you remove a column from Customers that customer_order_summary uses → view breaks -- Views become INVALID and produce errors on next query

When to use each view type:
  • Use regular views for security, abstraction, and code reuse — they're free (no storage)
  • Use materialized views when the underlying query is costly (joins multiple large tables, complex aggregations) and results can tolerate some staleness
  • Refresh materialized views based on acceptable staleness: nightly for reports, hourly for dashboards, on-commit for critical counts

Summary

  • Regular view: Stored query — virtual table, reexecuted on every query, no storage cost
  • Updatable view: Single-table, no aggregation/DISTINCT → INSERT/UPDATE/DELETE propagate to base table
  • WITH CHECK OPTION: Prevents DML that would remove the row from the view's visible set
  • Materialized view: Stored query results — faster reads, potential staleness, requires explicit (or automatic) refresh
  • Views provide security, abstraction, and denormalization depending on type and usage
