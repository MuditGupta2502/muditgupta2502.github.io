Database Normalization: From Anomalies to Clean Design

Database Design | 25 min read

Why Normalization Exists

The naive approach to storing data is to cram as many attributes as possible into a single table. This feels convenient at first, but once data volume grows and updates begin, the structural flaws become painful. Normalization is the formal process of decomposing tables to eliminate redundancy and the update anomalies it causes.

Consider a table called StudentCourse that tries to store everything:

StudentCourse (un-normalized): student_id | student_name | dept_id | dept_name | course_id | course_name | instructor | grade -----------|--------------|---------|-------------|-----------|-------------|------------|------ 101 | Mudit | D01 | Computer Sci| C001 | DBMS | Prof. Shah | A 101 | Mudit | D01 | Computer Sci| C002 | OS | Prof. Rao | B 102 | Aisha | D02 | Mathematics | C003 | Calculus | Prof. Iyer | A+ 102 | Aisha | D02 | Mathematics | C001 | DBMS | Prof. Shah | B+

The Three Anomalies

Insertion Anomaly

You cannot insert a new department until at least one student is enrolled. If the Electrical Engineering department exists but has no enrolled students yet, there is no row to put it in.

Update Anomaly

If Computer Science changes its name to "Computing & AI", every row for a CS student in every course must be updated. Miss one, and the data becomes inconsistent — some rows say "Computer Sci", others say "Computing & AI".

Deletion Anomaly

If student 102 drops all courses, and her row is deleted, we also lose the fact that the Mathematics department exists and that Prof. Iyer teaches Calculus.

Functional Dependency (FD)

Functional dependencies are the mathematical tool used to reason about normalization. X → Y (read as "X determines Y") means: if two tuples have the same value for X, they must have the same value for Y. X is called the determinant; Y is the dependent.

In StudentCourse: student_id → student_name (knowing id determines name) student_id → dept_id dept_id → dept_name (transitive: id → dept_id → dept_name) course_id → course_name course_id → instructor {student_id, course_id} → grade (grade depends on both)

Types of Functional Dependencies

  • Trivial FD: Y is a subset of X. (e.g., {A, B} → A is always true.)
  • Non-trivial FD: Y is not a subset of X. These are the meaningful ones.
  • Partial dependency: A non-key attribute depends on only part of a composite key.
  • Transitive dependency: A non-key attribute depends on another non-key attribute.
  • Multivalued dependency: An attribute independently determines a set of values for another attribute (relevant to 4NF).

Armstrong's Axioms

Armstrong's axioms provide a complete and sound set of inference rules for deriving all valid functional dependencies from a given set.

Primary Rules

  • Reflexivity: If Y ⊆ X, then X → Y. (Trivial FD.)
  • Augmentation: If X → Y, then XZ → YZ. (Adding attributes does not break the dependency.)
  • Transitivity: If X → Y and Y → Z, then X → Z.

Derived Rules (can be proved from above)

  • Union: If X → Y and X → Z, then X → YZ.
  • Decomposition: If X → YZ, then X → Y and X → Z.
  • Pseudotransitivity: If X → Y and WY → Z, then WX → Z.

Closure of Attributes

The closure of attribute set X under a set F of FDs, written X+, is the set of all attributes that can be determined from X using F. Closure is used to identify candidate keys and derive implied FDs.

FDs: A → B, B → C, A → D Find A+: Start with {A} Apply A → B → {A, B} Apply B → C → {A, B, C} Apply A → D → {A, B, C, D} A+ = {A, B, C, D} -- A is a candidate key if schema has only these 4 attrs

Minimal Cover (Canonical Cover)

A minimal cover of a set F of FDs is an equivalent set Fc where:

  1. Every FD in Fc has a single attribute on the right-hand side.
  2. No FD in Fc is redundant (removing it changes the closure).
  3. No attribute on the left-hand side of any FD is extraneous.

Minimal cover is important for producing the most compact normalized schema.

Normal Forms

First Normal Form (1NF)

A relation is in 1NF if and only if:

  • Every attribute contains only atomic (indivisible) values.
  • No column stores sets, arrays, or repeated groups.
  • Rows are uniquely identifiable.

Violation example:

Student(student_id, name, phone_numbers) Row: 101, Mudit, "9876543210, 9123456789" -- NOT 1NF (multi-valued cell)

Fix (1NF):

StudentPhones(student_id, phone) (101, 9876543210) (101, 9123456789)

Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and every non-key attribute is fully functionally dependent on the entire primary key. Partial dependencies must be eliminated. 2NF is only relevant when the primary key is composite.

Violation example:

Enrollment(student_id, course_id, student_name, course_name, grade) PK = {student_id, course_id} Partial dependencies (violate 2NF): student_id → student_name (depends on only part of PK) course_id → course_name (depends on only part of PK) Full dependency (correct for 2NF): {student_id, course_id} → grade

Fix (decompose into 2NF):

Students(student_id [PK], student_name) Courses(course_id [PK], course_name) Enrollment(student_id [PK, FK], course_id [PK, FK], grade)

Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and there are no transitive dependencies of non-key attributes on the primary key. Formally: for every non-trivial FD X → Y, either X is a super key or Y is part of a candidate key.

Violation example:

Student(student_id [PK], name, dept_id, dept_name) Transitive dependency: student_id → dept_id → dept_name dept_name depends on dept_id, not directly on student_id

Fix (decompose into 3NF):

Students(student_id [PK], name, dept_id [FK]) Departments(dept_id [PK], dept_name)

Boyce-Codd Normal Form (BCNF)

BCNF is a stricter version of 3NF. A relation is in BCNF if and only if for every non-trivial FD X → Y, X is a super key. BCNF eliminates all redundancy detectable by functional dependencies.

Key difference: 3NF allows Y to be part of a candidate key even if X is not a super key. BCNF does not allow this exception. Every 3NF relation is not necessarily in BCNF.

Classic BCNF violation example:

CourseTeacher(student_id, subject, teacher) Assumption: each teacher teaches only one subject; each student can have one teacher per subject. Candidate keys: {student_id, subject}, {student_id, teacher} FD: teacher → subject (teacher determines subject) teacher → subject violates BCNF because 'teacher' is NOT a super key, yet the dependency is non-trivial.

Fix (decompose into BCNF):

TeacherSubject(teacher [PK], subject) StudentTeacher(student_id [PK, FK], teacher [PK, FK])

Note: Decomposition into BCNF sometimes loses the ability to check a functional dependency without performing a join. This is the tradeoff: BCNF guarantees zero FD-based redundancy, but may not be dependency-preserving. 3NF always has a dependency-preserving lossless decomposition.

Fourth Normal Form (4NF)

4NF addresses multivalued dependencies. A multivalued dependency X ↠ Y exists when for a given X value, there is a set of Y values that is independent of all other attributes. A relation is in 4NF if it is in BCNF and has no non-trivial multivalued dependencies.

Employee(emp_id, skill, dependent_name) emp_id ↠ skill (employee has many skills) emp_id ↠ dependent_name (employee has many dependents) These are independent — skills have nothing to do with dependents. Storing them in one table creates spurious rows. Fix: EmployeeSkills(emp_id, skill) EmployeeDependents(emp_id, dependent_name)

Testing for Normal Form Violations

A practical checklist:

  1. List all FDs for the relation.
  2. Identify all candidate keys using attribute closure.
  3. Check 1NF: are all attribute values atomic?
  4. Check 2NF: does every non-key attribute depend on the whole PK (not just part of it)?
  5. Check 3NF: for every FD X → Y, is X a super key OR is Y part of a candidate key?
  6. Check BCNF: for every non-trivial FD X → Y, is X a super key?

Lossless Decomposition

When splitting a table into two, the decomposition is lossless if joining the resulting tables back always reproduces the original relation exactly — no spurious rows added.

A decomposition of R into R1 and R2 is lossless if either:

  • (R1 ∩ R2) → R1 attributes (the common attributes form a key of R1), or
  • (R1 ∩ R2) → R2 attributes (the common attributes form a key of R2).
Original: Employee(emp_id, dept_id, dept_name) FD: dept_id → dept_name Decompose to: E1(emp_id, dept_id) E2(dept_id, dept_name) E1 ∩ E2 = {dept_id}, which is PK of E2. Lossless. ✓

Dependency Preservation

A decomposition is dependency-preserving if every original FD can be checked by examining at most one of the result tables (without re-joining). If an FD spans multiple decomposed tables, enforcing it requires a join every time data changes — which is expensive.

BCNF always gives lossless decomposition but not always dependency preservation. 3NF always gives both.

Denormalization — When Normalization Goes Too Far

Normalization breaks data into many tables to minimize redundancy. But read-heavy analytical workloads may repeatedly join ten or more tables, which is expensive. Denormalization deliberately reintroduces controlled redundancy to reduce query complexity and improve read performance.

  • Pre-joining related tables into derived summary tables (data warehouses use star schema).
  • Storing a computed column derived from other columns (e.g., total_price = price * quantity).
  • Duplicating a column from a parent table into a child table to avoid a join.

The cost of denormalization is that update operations must maintain multiple copies of the same data. This is acceptable only when reads vastly outnumber writes and stale data is tolerable (or handled via triggers/application logic).

Step-by-Step Normalization Example

Start: OrderDetails(order_id, cust_id, cust_name, cust_city, product_id, product_name, qty, unit_price) FDs: order_id → {cust_id, cust_name, cust_city} (order belongs to one customer) cust_id → {cust_name, cust_city} (transitive through order_id) product_id → {product_name, unit_price} (product info) {order_id, product_id} → qty (qty specific to order-product pair) Candidate key: {order_id, product_id} --- Check 2NF --- Partial dependencies: order_id → cust_id, cust_name, cust_city (part of PK) product_id → product_name, unit_price (part of PK) => NOT in 2NF --- Fix: decompose --- Orders(order_id [PK], cust_id [FK]) Customers(cust_id [PK], cust_name, cust_city) Products(product_id [PK], product_name, unit_price) OrderItems(order_id [PK,FK], product_id [PK,FK], qty) --- Check 3NF --- In Customers: cust_id → cust_name, cust_city (cust_id is PK, no transitive FDs) In Orders: order_id → cust_id (cust_id is FK, no further transitive dependency within table) => All tables are in 3NF ✓ --- Check BCNF --- In each table, the only determinant is the PK, which is a super key. => All tables are in BCNF ✓

What You Should Remember

  • Normalization is driven by functional dependencies, found by analyzing what determines what.
  • The three anomalies — insertion, update, deletion — are the evidence that a table needs decomposition.
  • Armstrong's axioms let you derive all implied FDs from a given set.
  • Attribute closure X+ tells you which attributes a set X can determine, and helps find candidate keys.
  • 1NF removes multi-valued cells. 2NF removes partial dependencies. 3NF removes transitive dependencies. BCNF makes every determinant a super key.
  • Every lossless decomposition to BCNF is achievable; dependency preservation may require settling for 3NF.
  • Denormalization is a deliberate decision in read-heavy, performance-critical scenarios.
