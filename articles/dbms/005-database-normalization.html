Database Normalization: Eliminating Redundancy & Anomalies

Database Design | 12 min read

What is Normalization?

Database normalization is a systematic process of organizing data in a database to eliminate redundancy and improve data integrity. It involves breaking down larger tables into smaller, related tables and defining relationships between them.

Goals of Normalization:

  • Eliminate data redundancy (same data stored multiple times)
  • Prevent insertion, update, and deletion anomalies
  • Ensure data consistency and integrity
  • Simplify data maintenance

Problems Without Normalization

Insertion Anomaly: Difficulty inserting new data due to missing related data. Example: Cannot add a new course until a student enrolls in it.

Update Anomaly: Data inconsistency when updating redundant information. If a student's email is stored in multiple places and you update one, others become incorrect.

Deletion Anomaly: Unintended data loss when deleting records. Example: Deleting the last student enrolled in a course also deletes course information.

Data Redundancy: Same data repeated unnecessarily, wasting storage and creating consistency issues.

Functional Dependency

A functional dependency (FD) is a relationship between attributes. If attribute B depends on attribute A, then for each value of A, there is exactly one value of B.

Notation: A → B (read as "A determines B")

Example: In a Student table: 

student_id → name (one student_id has exactly one name)
student_id → email (one student_id has exactly one email)
email → student_id (one email belongs to exactly one student, assuming unique emails)

Keys and Functional Dependency

Prime Attribute: An attribute that is part of a candidate key.

Non-Prime Attribute: An attribute that is not part of any candidate key.

Functional dependencies are the basis for determining normal forms. If non-prime attributes are dependent only on the primary key (not on other attributes), the table is properly normalized.

Normal Forms

First Normal Form (1NF)

Rule: All attributes must contain atomic (indivisible) values. No repeating groups.

Violation Example: A Student table with a "phone_numbers" column containing multiple comma-separated values.

How to fix: Separate multi-valued attributes into a new table with a foreign key relationship. 

BEFORE (Not 1NF):
Students: student_id, name, phone_numbers ("123,456,789")

AFTER (1NF):
Students: student_id, name
Phones: student_id, phone_number

Second Normal Form (2NF)

Requirements: Must be in 1NF, and all non-prime attributes must be fully functionally dependent on the primary key (not just part of it).

Addresses partial dependency — where a non-prime attribute depends on only a part of a composite primary key.

Violation Example: 

StudentCourse (student_id, course_id, professor_name, professor_office)
Primary Key: (student_id, course_id)

Issue: professor_name depends only on course_id, not on the full key

How to fix: Separate into two tables: 

StudentCourse: (student_id, course_id)
Course: (course_id, professor_name, professor_office)

Third Normal Form (3NF)

Requirements: Must be in 2NF, and no non-prime attribute should depend on another non-prime attribute.

Addresses transitive dependency — where a non-prime attribute depends indirectly on the primary key through another non-prime attribute.

Violation Example: 

Employee (emp_id, emp_name, dept_id, dept_name, dept_location)
Primary Key: emp_id

Issue: dept_name and dept_location depend on dept_id, not directly on emp_id
(emp_id → dept_id → dept_name)

How to fix: 

Employee: (emp_id, emp_name, dept_id)
Department: (dept_id, dept_name, dept_location)

Boyce-Codd Normal Form (BCNF)

Requirements: 3NF with stricter conditions. Every determinant must be a candidate key.

BCNF handles edge cases where 3NF doesn't fully eliminate anomalies in tables with overlapping candidate keys.

Note: Most practical applications achieve 3NF, which provides good balance between normalization and query performance.

Denormalization: The Trade-off

While normalization reduces redundancy, excessive normalization can impact query performance due to multiple joins.

Denormalization is intentionally introducing redundancy for performance optimization. Used when:

  • Read performance is critical and write performance is less important
  • Complex joins on large tables create performance bottlenecks
  • Aggregate data needs to be frequently accessed

Example: Storing order_total in Orders table even though it can be calculated from order_items.

Normalization Process Summary

Unnormalized → 1NF (atomic values)
1NF → 2NF (remove partial dependencies)
2NF → 3NF (remove transitive dependencies)
3NF → BCNF (stricter candidate key requirement)

Summary

Database normalization is a crucial design principle that ensures data integrity and eliminates anomalies. By understanding functional dependencies and applying normal forms, you can design databases that are maintainable, consistent, and efficient. The goal is to find the right balance between purity (full normalization) and performance (denormalization where justified).
