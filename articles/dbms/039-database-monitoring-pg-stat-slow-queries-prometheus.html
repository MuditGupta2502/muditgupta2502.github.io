Database Monitoring and Alerting: pg_stat, Slow Queries and Prometheus

Operations | 22 min read

You cannot tune what you cannot see. A comprehensive monitoring strategy tells you what your database is doing right now, what it has been doing historically, and alerts you before small issues become outages. This article covers the key PostgreSQL system views, slow query identification, and metric collection.

Active Session Monitoring

-- pg_stat_activity: live view of all backend connections SELECT pid, usename, application_name, client_addr, state, wait_event_type, wait_event, LEFT(query, 100) AS query_snippet, NOW() - query_start AS query_duration, NOW() - xact_start AS transaction_duration FROM pg_stat_activity WHERE state != 'idle' ORDER BY query_duration DESC NULLS LAST; -- Important states: -- active: currently executing SQL -- idle: connected but no query running -- idle in transaction: opened a transaction but not doing anything (DANGEROUS) -- idle in transaction (aborted): transaction had an error; needs rollback -- fastpath function call: short-circuit API call -- Key wait events to watch: -- Lock → waiting for a lock (often a blocking query situation) -- LWLock:WALWrite → WAL writer is a bottleneck (high commit rate) -- IO:DataFileRead → I/O bound; shared_buffers too small or seq scans -- Client → waiting for client to send next query (normal for pooled apps) -- Find long-running transactions (idle in transaction > 5 minutes): SELECT pid, usename, state, wait_event, NOW() - xact_start AS stuck_duration, LEFT(query, 200) AS last_query FROM pg_stat_activity WHERE state = 'idle in transaction' AND NOW() - xact_start > INTERVAL '5 minutes' ORDER BY stuck_duration DESC; -- Terminate a stuck query: SELECT pg_cancel_backend(pid); -- sends SIGINT, graceful kill SELECT pg_terminate_backend(pid); -- sends SIGTERM, hard kill (use if cancel fails)

Lock Monitoring

-- Find blocking chains: who is blocking whom WITH RECURSIVE lock_tree AS ( -- Blocked backends: SELECT blocked.pid AS blocked_pid, blocked.usename AS blocked_user, blocking.pid AS blocking_pid, blocking.usename AS blocking_user, blocked_locks.locktype, blocked_locks.relation::regclass AS locked_table, NOW() - blocked.query_start AS blocked_duration, LEFT(blocked.query, 100) AS blocked_query FROM pg_locks blocked_locks JOIN pg_stat_activity blocked ON blocked.pid = blocked_locks.pid JOIN pg_locks blocking_locks ON ( blocking_locks.locktype = blocked_locks.locktype AND blocking_locks.relation = blocked_locks.relation AND blocking_locks.granted = true AND blocked_locks.granted = false AND blocking_locks.pid != blocked_locks.pid ) JOIN pg_stat_activity blocking ON blocking.pid = blocking_locks.pid ) SELECT * FROM lock_tree ORDER BY blocked_duration DESC; -- Set lock_timeout to prevent indefinite blocking (globally or per connection): SET lock_timeout = '5s'; -- individual session -- In postgresql.conf: lock_timeout = '30s' -- global default -- If lock cannot be acquired within timeout: ERROR 55P03: lock timeout -- Statement timeout to kill runaway queries: SET statement_timeout = '30s'; -- kill individual session -- In postgresql.conf: statement_timeout = '60s' -- global guard

Table and Index Statistics

-- pg_stat_user_tables: read/write activity and bloat indicators SELECT schemaname, relname AS table, seq_scan, -- full table scans (high = missing index) seq_tup_read, -- rows read by seq scans idx_scan, -- index scans n_live_tup, -- estimated live rows n_dead_tup, -- dead rows (bloat, needs VACUUM) last_autovacuum, -- when last autovacuum ran last_autoanalyze -- when last autoanalyze ran FROM pg_stat_user_tables WHERE schemaname = 'public' ORDER BY seq_scan DESC; -- Tables with many seq_scans and large seq_tup_read are candidates for new indexes -- Check dead tuple ratio (bloat warning): SELECT relname AS table, n_live_tup, n_dead_tup, ROUND(100 * n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct FROM pg_stat_user_tables WHERE n_dead_tup > 1000 ORDER BY dead_pct DESC; -- Dead tuple % > 10% → consider VACUUM ANALYZE -- Index scan vs seq scan ratio: SELECT relname, CASE WHEN seq_scan + idx_scan = 0 THEN 0 ELSE ROUND(100 * idx_scan::numeric / (seq_scan + idx_scan), 1) END AS index_pct FROM pg_stat_user_tables WHERE schemaname = 'public' ORDER BY index_pct; -- Low index_pct on large tables = indexes missing or not being used

pg_stat_statements: Query Performance Analysis

-- pg_stat_statements extension: tracks statistics for all queries executed -- Must be loaded at server start: -- In postgresql.conf: shared_preload_libraries = 'pg_stat_statements' CREATE EXTENSION pg_stat_statements; -- Top 20 slowest queries by total time: SELECT calls, ROUND(total_exec_time::numeric / 1000, 2) AS total_sec, ROUND(mean_exec_time::numeric, 2) AS mean_ms, ROUND(stddev_exec_time::numeric, 2) AS stddev_ms, rows, ROUND(rows::numeric / NULLIF(calls, 0), 2) AS rows_per_call, LEFT(query, 120) AS query FROM pg_stat_statements WHERE calls > 100 ORDER BY total_exec_time DESC LIMIT 20; -- High-call, low-mean queries (look for OLTP optimization): SELECT calls, ROUND(mean_exec_time::numeric, 2) AS mean_ms, ROUND(total_exec_time::numeric / 1000, 2) AS total_sec, LEFT(query, 120) AS query FROM pg_stat_statements ORDER BY calls DESC LIMIT 20; -- Queries with high buffer reads (cache misses): SELECT calls, shared_blks_hit, shared_blks_read, ROUND(100 * shared_blks_hit::numeric / NULLIF(shared_blks_hit + shared_blks_read, 0), 1) AS cache_hit_pct, LEFT(query, 100) AS query FROM pg_stat_statements WHERE shared_blks_hit + shared_blks_read > 10000 ORDER BY cache_hit_pct; -- cache_hit_pct < 90% suggests: index needed, or shared_buffers too small -- Reset stats to start fresh: SELECT pg_stat_statements_reset();

Slow Query Logging

-- postgresql.conf settings for slow query capture: log_min_duration_statement = 1000 -- log queries taking > 1 second (1000ms) log_duration = off -- do not log duration of ALL queries (too noisy) log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h ' log_checkpoints = on -- log checkpoint activity (useful for tuning) log_lock_waits = on -- log queries waiting for locks > deadlock_timeout log_temp_files = 0 -- log temp file creation (spills to disk) -- auto_explain: log execution plans for slow queries automatically: -- shared_preload_libraries = 'auto_explain' auto_explain.log_min_duration = 5000 -- log plan if query takes > 5s auto_explain.log_analyze = on -- include ANALYZE (actual rows, timing) auto_explain.log_buffers = on -- include buffer stats auto_explain.log_nested_statements = on -- also log sub-queries -- Reading the PostgreSQL log (typical pattern): -- 2024-11-04 09:12:34 UTC [17234]: user=app,db=prod duration: 4567.123 ms -- statement: SELECT * FROM orders WHERE status = 'pending' ORDER BY created_at -- Log analysis tools: -- pgBadger: parses PostgreSQL logs, generates HTML reports with top slow queries -- pganalyze: SaaS monitoring for PostgreSQL (commercial) -- pg_activity: top-like tool for PostgreSQL (real-time monitoring)

Checkpoint and WAL Monitoring

-- pg_stat_bgwriter: checkpoint and background writer stats SELECT checkpoints_timed, -- checkpoints due to timeout (normal) checkpoints_req, -- checkpoints due to WAL size (potential issue) checkpoint_write_time, checkpoint_sync_time, buffers_checkpoint, -- buffers written at checkpoint buffers_clean, -- buffers written by background writer buffers_backend, -- buffers written directly by backend (BAD - stalls queries) buffers_alloc, -- new buffers allocated maxwritten_clean -- times bgwriter stopped due to max_bgwriter_lru_maxpages FROM pg_stat_bgwriter; -- buffers_backend > 0 means: queries are stalling waiting to flush dirty pages -- This indicates: shared_buffers too small, or bgwriter too slow -- Remediation: increase shared_buffers, tune bgwriter_lru_maxpages -- WAL generation rate: SELECT pg_size_pretty( (SELECT setting::bigint FROM pg_settings WHERE name = 'wal_buffers') + pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0') ) AS total_wal_generated; -- checkpoints_req high relative to checkpoints_timed means: -- WAL fills up faster than checkpoint interval → increase max_wal_size

Prometheus and postgres_exporter

-- postgres_exporter: exposes PostgreSQL metrics to Prometheus -- Install and configure: -- docker run -e DATA_SOURCE_NAME="postgresql://user:pass@localhost/postgres?sslmode=disable" \ -- quay.io/prometheuscommunity/postgres-exporter -- Key metrics exposed (sample Prometheus queries): -- Connection count by state: -- pg_stat_activity_count{datname="mydb"} -- Cache hit ratio: -- rate(pg_stat_database_blks_hit[5m]) / -- (rate(pg_stat_database_blks_hit[5m]) + rate(pg_stat_database_blks_read[5m])) -- Replication lag (read replica is behind by N bytes): -- pg_replication_slots_confirmed_flush_lsn - pg_current_wal_lsn() -- Transaction rate: -- rate(pg_stat_database_xact_commit[1m]) + rate(pg_stat_database_xact_rollback[1m]) -- Alerting thresholds (Alertmanager / Grafana alerts): -- Cache hit ratio < 95% → shared_buffers likely too small -- Connection count > 80% → connection pool approaching limit -- Replication lag > 30 seconds → replica falling behind (data freshness risk) -- Lock wait count > 0 → blocking detected (investigate pg_stat_activity) -- Dead tuple ratio > 10% → autovacuum not keeping up (tune autovacuum) -- Disk I/O wait > 20ms avg → I/O subsystem bottleneck -- Grafana dashboards for PostgreSQL: -- Dashboard ID 9628 (postgres_exporter): comprehensive, community-maintained -- Includes: connections, cache hit, checkpoints, replication, table stats
Monitoring checklist — apply to every production PostgreSQL instance:
  • Enable pg_stat_statements — no overhead when not queried; invaluable for slow query analysis
  • Set log_min_duration_statement = 1000 — catch any query over 1s
  • Enable log_lock_waits — know when lock contention happens
  • Set idle_in_transaction_session_timeout = 5min — kill stuck transactions automatically
  • Deploy postgres_exporter + Prometheus + Grafana — baseline dashboards in minutes
  • Alert on: cache hit < 95%, replication lag > 30s, connection usage > 80%, dead tuple ratio > 10%

Summary

  • pg_stat_activity: real-time sessions; look for lock waits, long-running transactions, idle-in-transaction
  • pg_stat_statements: aggregated query statistics; find top time consumers and high-frequency queries
  • pg_stat_user_tables: seq_scan counts and dead tuple ratios guide indexing and vacuum decisions
  • Slow query logging with log_min_duration_statement and auto_explain capture actual execution plans automatically
  • Prometheus + postgres_exporter enables time-series monitoring and threshold alerting
  • Key alerts: cache hit ratio, connection saturation, replication lag, lock contention, dead tuple ratio
