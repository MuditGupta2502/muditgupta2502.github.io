Query Optimization: From SQL String to Efficient Execution

Performance & Internals | 22 min read

Why Query Optimization Matters

The same SQL query can be answered in dozens of physically different ways. A naive plan might scan millions of rows; an optimized plan might resolve the same result by reading a few hundred index entries. On large tables the difference is not 10% faster — it can be the difference between a 0.5 ms response and a 30-second timeout.

Understanding optimization does not require reading DBMS source code. It requires understanding how the DBMS reasons about SQL, what execution plan choices exist, and what signals it uses to choose between them.

The Query Processing Pipeline

  1. Parsing: The SQL string is tokenized and validated against grammar rules. A parse tree is produced.
  2. Semantic analysis: Table names, column names, and types are resolved against the catalog. Errors like "no such column" are caught here.
  3. Translation to logical plan: The parse tree is converted to a relational algebra expression tree.
  4. Optimization: The logical plan is transformed into the lowest-cost physical execution plan by the query optimizer.
  5. Code generation / compilation: The physical plan is compiled into executable operators or bytecode.
  6. Execution: Operators are executed, data retrieved, and results returned to the client.

Logical vs Physical Plan

The logical plan describes what operations to perform (select, project, join) without specifying how. The physical plan specifies the exact algorithm for each operation: which index to use, which join algorithm to apply, and in what order to execute operations.

SQL: SELECT s.name FROM Students s JOIN Departments d ON s.dept_id = d.dept_id WHERE d.dept_name = 'Computer Science'; Logical plan: Project[name] → Join[s.dept_id = d.dept_id] → Scan(Students) → Select[dept_name = 'CS'](Scan(Departments)) Physical plan option A (poor): Hash Join → SeqScan(Students) → SeqScan(Departments) filtered dept_name='CS' Physical plan option B (better, with index): Nested Loop → IndexScan(Departments, dept_name = 'CS') -- few rows → IndexScan(Students, dept_id = matched_ids) -- use FK index

Rule-Based Optimization (RBO)

Early optimizers used only fixed transformation rules:

  • Push selections (WHERE clauses) as low as possible in the plan tree — reduces intermediate result size early.
  • Push projections down to eliminate unused columns early.
  • Reorder joins if one side is significantly smaller.
  • Replace subqueries with joins when equivalent.
  • Avoid Cartesian products when a join condition exists.

RBO is deterministic — the same SQL always produces the same plan. The limitation is that it ignores actual data statistics. A rule might say "always use the index," but on a highly duplicated column an index lookup may be slower than a full table scan.

Cost-Based Optimization (CBO)

Modern DBMS engines (PostgreSQL, MySQL 8+, SQL Server, Oracle) are primarily cost-based. The optimizer assigns a numerical cost to each candidate plan using statistics stored in the system catalog, then selects the minimum-cost plan.

Statistics the Optimizer Uses

  • Table cardinality: Number of rows in the table.
  • Column cardinality: Number of distinct values. High cardinality = more selective.
  • Histograms: Distribution of values (are values evenly spread, or does one value appear 90% of the time?).
  • Index type and depth: B-tree height, covering columns, selectivity.
  • Average row size: Affects memory requirements for sort and hash operations.
  • Correlation between columns: Rarely tracked, but important for multi-column predicates.
-- Update statistics in PostgreSQL ANALYZE Students; -- Update statistics in MySQL ANALYZE TABLE Students; -- SQL Server UPDATE STATISTICS Students;
Stale statistics are one of the most common causes of bad query plans. Re-analyze tables after bulk loads or significant data changes.

Execution Plan Access Methods

Sequential Scan (Full Table Scan)

Reads every block of the table in order. Cost is proportional to table size. Best when retrieving a large fraction of rows or when no useful index exists.

Index Scan

Traverses a B-tree index to find qualifying rows, then fetches the actual row from the heap. Efficient when selecting a small fraction of rows (high-selectivity predicate).

Index-Only Scan

When all required columns are contained in the index itself, the heap access step is skipped entirely. This is called a "covering index" scenario and is very fast.

-- Covering index: query only touches index, never heap CREATE INDEX idx_students_dept_gpa ON Students(dept_id, gpa); SELECT dept_id, gpa FROM Students WHERE dept_id = 3; -- Both dept_id and gpa are in the index. Index-only scan possible.

Bitmap Index Scan

Used when OR combines two index conditions. Builds bitmaps from two separate indexes, ORs them together, then fetches matching rows in heap order.

Join Algorithms

Nested Loop Join

For every row in the outer table, scan the inner table. Cost: O(|outer| × |inner|). Fast when the inner table is small or has a supporting index.

-- Effective when inner side is small or accessed via index For each row r in Students: -- outer For each row t in Enrollments -- inner where t.student_id = r.student_id: output (r, t)

Hash Join

Builds a hash table from the smaller relation (build phase), then probes it with each row from the larger relation (probe phase). Cost: O(|build| + |probe|). Excellent for large tables without supporting index. Requires memory to hold the hash table; spills to disk if memory is exhausted.

Sort-Merge Join

Both relations are sorted on the join key, then merged in a single pass. Cost: O(|R| log|R| + |S| log|S| + |R| + |S|). Particularly efficient when both sides are already sorted (e.g., already using a B-tree index on the join column). Good for inequality joins (<, >, BETWEEN) where hash join cannot be used.

Reading Execution Plans

Most DBMS provide an EXPLAIN command that shows the chosen execution plan without executing the query.

-- PostgreSQL EXPLAIN SELECT s.name FROM Students s JOIN Departments d ON s.dept_id = d.dept_id WHERE d.dept_name = 'Computer Science'; -- Sample output: Nested Loop (cost=0.28..19.41 rows=5 width=50) -> Index Scan using departments_name_idx on departments d Index Cond: ((dept_name)::text = 'Computer Science'::text) -> Index Scan using students_dept_idx on students s Index Cond: (dept_id = d.dept_id) -- EXPLAIN ANALYZE actually runs the query and shows real timing EXPLAIN ANALYZE SELECT ...;

Key fields to read:

  • cost=startup..total — Estimated cost units (not milliseconds). Lower is better.
  • rows — Estimated number of rows returned by this node.
  • actual time (ANALYZE only) — Real execution time in milliseconds.
  • loops (ANALYZE only) — How many times this node was executed.
  • Scan type — SeqScan means no index used; IndexScan means an index was used.

When Does the Optimizer NOT Use an Index?

  • Low selectivity: Column has very few distinct values (e.g., a boolean column). Scanning the whole table is faster than random-access index reads.
  • Function on the indexed column: WHERE LOWER(name) = 'mudit' prevents index use on name. Fix: create a functional index.
  • Leading wildcard LIKE: WHERE name LIKE '%udit' cannot use a B-tree prefix scan. Fix: use full-text search.
  • Implicit type cast: WHERE student_id = '101' (string vs int comparison) may prevent index use.
  • OR conditions: WHERE dept_id = 1 OR dept_id = 2 may lead to separate index scans + bitmap merge, or a full scan. Use IN (1, 2).
  • Negation: WHERE dept_id != 3 has low selectivity in most cases. Index rarely helps.

Query Rewriting Techniques

Subquery to JOIN

-- Subquery version (may execute once per outer row) SELECT name FROM Students WHERE dept_id IN (SELECT dept_id FROM Departments WHERE dept_name = 'CS'); -- JOIN version (usually more efficient; optimizer can choose join algorithm) SELECT s.name FROM Students s JOIN Departments d ON s.dept_id = d.dept_id WHERE d.dept_name = 'CS';

EXISTS over COUNT

-- Slow: scans entire match set SELECT name FROM Students s WHERE (SELECT COUNT(*) FROM Enrollments e WHERE e.student_id = s.student_id) > 0; -- Fast: stops at first match SELECT name FROM Students s WHERE EXISTS (SELECT 1 FROM Enrollments e WHERE e.student_id = s.student_id);

Avoid SELECT *

-- Fetches all columns, including those never used. Defeats index-only scans. SELECT * FROM Students WHERE dept_id = 3; -- Better: fetch only what you need SELECT student_id, name FROM Students WHERE dept_id = 3;

Avoid Functions on Indexed Columns in WHERE

-- Prevents index use on 'name' SELECT * FROM Students WHERE YEAR(dob) = 2002; -- Better: range predicate the index can satisfy SELECT * FROM Students WHERE dob BETWEEN '2002-01-01' AND '2002-12-31';

Relational Algebra Equivalences Used by Optimizer

  • Selection cascading: σc1 AND c2(R) = σc1(σc2(R))
  • Selection commutativity: σc1(σc2(R)) = σc2(σc1(R))
  • Projection cascading: πL1(πL2(R)) = πL1(R) when L1 ⊆ L2
  • Pushing selection below join: σc(R ⋈ S) = σc(R) ⋈ S where c only references R's attributes
  • Join commutativity: R ⋈ S = S ⋈ R
  • Join associativity: (R ⋈ S) ⋈ T = R ⋈ (S ⋈ T)

Practical Optimization Checklist

  1. Run EXPLAIN (ANALYZE) first. Understand the current plan before changing anything.
  2. Identify the most expensive node in the plan tree (highest Cost or Actual Time).
  3. Check if a useful index exists on filtering and joining columns.
  4. Ensure statistics are up to date with ANALYZE.
  5. Remove functions from indexed columns in WHERE. Rewrite as range conditions.
  6. Replace correlated subqueries with JOINs where possible.
  7. Limit result sets early with WHERE conditions, not late with outer filtering.
  8. Use covering indexes for hot read paths.
  9. Avoid SELECT * — name exactly the columns needed.
  10. Watch query frequency: a moderately slow query executed 10,000 times/sec matters more than a slow one executed once/day.

What You Should Remember

  • SQL is declarative — the optimizer decides the physical execution strategy.
  • Cost-based optimization uses statistics (cardinality, histograms, index depth) to estimate and minimize plan cost.
  • The three main join algorithms are Nested Loop (good for small/indexed inner), Hash Join (good for big unsorted equi-joins), and Sort-Merge (good for sorted or inequality joins).
  • EXPLAIN shows the chosen plan. EXPLAIN ANALYZE runs it and shows actual timings — use this to validate optimizer assumptions.
  • Indexes are not always used: low selectivity, functions on columns, leading wildcards, and implicit casts all prevent index use.
  • Rewrite subqueries as JOINs where semantically equivalent; use EXISTS instead of COUNT for existence checks.
  • Stale statistics cause bad plans. Analyze after bulk operations.
