SQL Subqueries Deep Dive: Scalar, Correlated, EXISTS & CTEs

SQL Advanced | 28 min read

Subqueries are queries nested within another query. They're one of the most expressive SQL features — enabling you to filter by aggregated results, compare rows to their peers, check for existence, and build complex multi-step data pipelines. This article covers every subquery type with practical examples and explains when to use a subquery vs a join vs a CTE.

Classification of Subqueries

CategoryLocation in QueryReturns
Scalar subquerySELECT list or WHERE comparing with =, <, >Exactly one row, one column
Column subqueryWHERE col IN (...) or WHERE col = ANY(...)One column, multiple rows
Table/Row subqueryFROM clause (derived table)Full result set used as a table
Correlated subqueryWHERE or HAVING; references outer queryEvaluated once per outer row
EXISTS subqueryWHERE EXISTS / NOT EXISTSBoolean (row found or not)

Scalar Subqueries

A scalar subquery returns exactly one value (one row, one column). It can be used anywhere a single value is expected: in the SELECT list, in WHERE, or in CASE.

-- SELECT list: show each customer's total alongside the grand average SELECT customer_id, name, (SELECT SUM(total) FROM Orders o WHERE o.customer_id = c.customer_id) AS my_total, (SELECT AVG(total) FROM Orders) AS avg_order -- same value for all rows (non-correlated) FROM Customers c; -- WHERE clause: products priced above the category average SELECT product_id, name, price, category FROM Products p WHERE price > ( SELECT AVG(price) FROM Products WHERE category = p.category -- correlated: references outer p.category ); -- CASE with scalar subquery SELECT order_id, total, CASE WHEN total > (SELECT PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY total) FROM Orders) THEN 'high-value' ELSE 'standard' END AS order_tier FROM Orders;
Scalar subquery contract: If a scalar subquery returns more than one row, it raises a runtime error. If it returns zero rows, it returns NULL. Design with these in mind.

Column Subqueries: IN and ANY/ALL

-- IN: matches any value in the list SELECT name, email FROM Customers WHERE customer_id IN ( SELECT DISTINCT customer_id FROM Orders WHERE order_date >= CURRENT_DATE - INTERVAL '30 days' ); -- NOT IN (careful with NULLs!) SELECT name FROM Products WHERE product_id NOT IN ( SELECT product_id FROM OrderItems -- if ANY product_id is NULL here, whole result is empty! ); -- Safer: NOT EXISTS handles NULLs correctly SELECT name FROM Products p WHERE NOT EXISTS ( SELECT 1 FROM OrderItems oi WHERE oi.product_id = p.product_id ); -- ANY: at least one value matches SELECT name, price FROM Products WHERE price > ANY (SELECT price FROM Products WHERE category = 'Budget'); -- Equivalent to: price > MIN(budget category prices) -- ALL: must be greater than every value SELECT name, price FROM Products WHERE price > ALL (SELECT price FROM Products WHERE category = 'Budget'); -- Equivalent to: price > MAX(budget category prices)

Derived Tables (FROM Subqueries)

A subquery in the FROM clause creates a temporary result set, called a derived table. It must be given an alias.

-- Average revenue per customer segment SELECT segment_data.loyalty_tier, AVG(segment_data.customer_revenue) AS avg_revenue_per_customer FROM ( SELECT c.loyalty_tier, SUM(o.total) AS customer_revenue FROM Customers c JOIN Orders o ON c.customer_id = o.customer_id GROUP BY c.customer_id, c.loyalty_tier ) AS segment_data GROUP BY segment_data.loyalty_tier; -- Top 5 products by revenue in each category SELECT * FROM ( SELECT product_id, name, category, SUM(oi.quantity * oi.unit_price) AS revenue, RANK() OVER (PARTITION BY category ORDER BY SUM(oi.quantity * oi.unit_price) DESC) AS rnk FROM Products p JOIN OrderItems oi USING(product_id) GROUP BY product_id, name, category ) ranked WHERE rnk <= 5;

Correlated Subqueries

A correlated subquery references columns from the outer query. It is evaluated once for each row processed by the outer query — making it potentially O(n) or O(n²) in complexity. Use them carefully on large tables; a JOIN or window function is often faster.

-- Find each employee's salary compared to department average SELECT e.employee_id, e.name, e.salary, e.department, (SELECT AVG(e2.salary) FROM Employees e2 WHERE e2.department = e.department) AS dept_avg FROM Employees e; -- Correlated: inner query references e.department from outer query -- Runs once per employee row → potentially slow on large tables -- Better alternative: window function (runs in one pass) SELECT employee_id, name, salary, department, AVG(salary) OVER (PARTITION BY department) AS dept_avg FROM Employees; -- Correlated UPDATE: update each customer's last_order_date UPDATE Customers c SET last_order_date = ( SELECT MAX(order_date) FROM Orders o WHERE o.customer_id = c.customer_id ); -- This is fine for a one-time fix; for frequent updates use triggers or denormalization

EXISTS and NOT EXISTS

EXISTS returns TRUE if the subquery returns at least one row; FALSE otherwise. It's a semi-join: it stops at the first match (short-circuits). Unlike IN, it handles NULLs correctly and doesn't care about the subquery's column values — just row existence.

-- Customers who have made at least one order (semi-join) SELECT c.customer_id, c.name FROM Customers c WHERE EXISTS ( SELECT 1 FROM Orders o WHERE o.customer_id = c.customer_id ); -- SELECT 1: we don't care about values, just existence — SELECT * would also work -- The subquery short-circuits at the first matching order -- Customers who have placed an order in EVERY product category -- (using NOT EXISTS double negation — equivalent to relational algebra division) SELECT c.customer_id, c.name FROM Customers c WHERE NOT EXISTS ( SELECT 1 FROM Products p WHERE NOT EXISTS ( SELECT 1 FROM Orders o JOIN OrderItems oi ON o.order_id = oi.order_id WHERE o.customer_id = c.customer_id AND oi.product_id = p.product_id ) ); -- Read: there is no product for which this customer does NOT have an order -- EXISTS in WITH CHECK OPTION enforcement (automatically handled by DB) -- EXISTS for duplicate prevention SELECT p.name FROM Products p WHERE EXISTS ( SELECT 1 FROM OrderItems oi WHERE oi.product_id = p.product_id AND EXISTS (SELECT 1 FROM Orders o WHERE o.order_id = oi.order_id AND o.status = 'cancelled') );

Common Table Expressions (CTEs)

A CTE defines a named result set at the top of a query using WITH. It can be referenced multiple times in the main query. CTEs improve readability and organization without creating database objects.

-- CTE: customer revenue ranking WITH customer_revenue AS ( SELECT customer_id, SUM(total) AS total_revenue FROM Orders WHERE status = 'delivered' GROUP BY customer_id ), revenue_ranked AS ( SELECT customer_id, total_revenue, RANK() OVER (ORDER BY total_revenue DESC) AS revenue_rank FROM customer_revenue ) SELECT c.name, c.email, rr.total_revenue, rr.revenue_rank FROM Customers c JOIN revenue_ranked rr USING(customer_id) WHERE rr.revenue_rank <= 10; -- Top 10 customers by revenue -- Multiple CTEs (separated by commas) WITH active_products AS ( SELECT product_id, name, price FROM Products WHERE is_active = TRUE ), recent_sales AS ( SELECT product_id, SUM(quantity) AS units_sold FROM OrderItems oi JOIN Orders o USING(order_id) WHERE o.order_date >= NOW() - INTERVAL '90 days' GROUP BY product_id ) SELECT ap.name, ap.price, COALESCE(rs.units_sold, 0) AS recent_units FROM active_products ap LEFT JOIN recent_sales rs USING(product_id) ORDER BY recent_units DESC;

Recursive CTEs

Recursive CTEs traverse hierarchical structures like org charts, folder trees, category hierarchies, or graph-like BOM (bill of materials) relationships. They have two parts: the base case (anchor) and the recursive step.

-- Org chart: find all reports under a given manager (any depth) WITH RECURSIVE org_hierarchy AS ( -- Anchor: start with the target manager SELECT employee_id, name, manager_id, 1 AS depth FROM Employees WHERE employee_id = 5 -- CEO or target manager's ID UNION ALL -- Recursive step: find direct reports of employees already in result SELECT e.employee_id, e.name, e.manager_id, oh.depth + 1 FROM Employees e INNER JOIN org_hierarchy oh ON e.manager_id = oh.employee_id ) SELECT employee_id, name, depth FROM org_hierarchy ORDER BY depth, name; -- Category path: get full breadcrumb path for each category WITH RECURSIVE category_path AS ( -- Anchor: top-level categories (no parent) SELECT category_id, name, parent_id, name::TEXT AS full_path FROM Categories WHERE parent_id IS NULL UNION ALL SELECT c.category_id, c.name, c.parent_id, cp.full_path || ' > ' || c.name FROM Categories c JOIN category_path cp ON c.parent_id = cp.category_id ) SELECT * FROM category_path ORDER BY full_path; -- Results: "Electronics", "Electronics > Computers", "Electronics > Computers > Laptops" -- Safety: always add a depth limit to prevent infinite loops from bad data WITH RECURSIVE tree AS ( SELECT id, parent_id, 0 AS depth FROM nodes WHERE parent_id IS NULL UNION ALL SELECT n.id, n.parent_id, t.depth + 1 FROM nodes n JOIN tree t ON n.parent_id = t.id WHERE t.depth < 20 -- max depth guard ) SELECT * FROM tree;
CTE materialization: In PostgreSQL 12+, CTEs are inlined by default (the optimizer can push predicates down). In PostgreSQL <12 and when declared with WITH ... AS MATERIALIZED, the CTE is evaluated once and cached. Use AS MATERIALIZED when you reference the CTE multiple times or want to prevent re-evaluation.

Subquery vs JOIN vs CTE: When to Use Each

Use CaseBest ChoiceWhy
Filter by existenceEXISTS subqueryShort-circuits, handles NULLs
Combine rows from two tablesJOINSet-based, optimizer-friendly
Complex multi-step queryCTEReadable, referenceable multiple times
Hierarchical traversalRecursive CTEOnly clean solution for unbounded depth
Compare row to its group aggregateWindow functionSingle pass, far faster than correlated subquery
Top-N per groupWindow function or LATERALEfficient; correlated subquery is O(n²)

Summary

  • Scalar subquery: Returns one value — usable in SELECT list, WHERE, CASE
  • IN / ANY / ALL: Column comparisons — use NOT EXISTS instead of NOT IN to avoid NULL traps
  • Derived tables: FROM subquery — useful for pre-aggregating before joining
  • Correlated subquery: References outer row — powerful but can be O(n) per row; prefer window functions
  • EXISTS: Semi-join pattern — short-circuits, NULL-safe, best for existence checks
  • CTE: Named temporary result set — improves readability, reusable in query
  • Recursive CTE: Hierarchical traversal — org charts, trees, paths — add depth limit as guard
