Data Warehousing: Star Schema, Snowflake, OLAP, Kimball and Inmon

Analytics | 20 min read

Operational databases (OLTP) are optimized for frequent small transactions. Analytical workloads (OLAP) need large scans and aggregations over historical data. A data warehouse is designed for this second category.

Fact and Dimension Tables

-- Fact table: measurable events (sales, clicks, payments) -- Dimension table: descriptive context (customer, product, date, region) CREATE TABLE fact_sales ( date_key INT, product_key INT, customer_key INT, store_key INT, quantity INT, revenue NUMERIC(12,2), discount NUMERIC(12,2) ); CREATE TABLE dim_product ( product_key INT PRIMARY KEY, sku TEXT, product_name TEXT, category TEXT, brand TEXT ); CREATE TABLE dim_date ( date_key INT PRIMARY KEY, full_date DATE, day_name TEXT, month INT, quarter INT, year INT );

Star vs Snowflake

ModelDesignProsCons
StarDenormalized dimensionsFast joins, easy BI queriesSome redundancy
SnowflakeNormalized dimensionsLess redundancyMore joins, more complexity

OLAP Operations

-- Roll-up: aggregate to higher level SELECT d.year, p.category, SUM(f.revenue) FROM fact_sales f JOIN dim_date d ON f.date_key = d.date_key JOIN dim_product p ON f.product_key = p.product_key GROUP BY d.year, p.category; -- Drill-down: move from year -> month -> day -- Slice: filter one dimension (e.g., year=2024) -- Dice: filter multiple dimensions (year=2024, category='Electronics')

Kimball vs Inmon

ApproachDirectionStrength
KimballBottom-up (data marts first)Fast business value, practical for product teams
InmonTop-down (enterprise DW first)Strong enterprise consistency/governance

Modern Warehouse Stack

-- Typical pipeline: -- Source DB -> CDC/ETL -> staging -> warehouse model -> BI dashboards -- Popular tools: -- Storage/compute: Snowflake, BigQuery, Redshift -- Transform: dbt -- Ingestion: Fivetran, Airbyte, custom CDC -- BI: Power BI, Tableau, Looker
Practical rule: Start with star schema + Kimball-style marts for quick delivery. Introduce stricter enterprise modeling later when governance and cross-domain consistency become bottlenecks.

Summary

  • Warehouses are built for analytical scans and aggregations, not transactional writes.
  • Fact tables hold events; dimensions provide context.
  • Star schema is usually the best first choice for analytics speed and simplicity.
  • Kimball gives faster iterative delivery; Inmon gives stronger enterprise consistency.
