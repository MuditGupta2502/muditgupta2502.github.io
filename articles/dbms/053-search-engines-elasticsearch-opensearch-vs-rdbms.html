Search Engines vs Databases: Elasticsearch/OpenSearch and RDBMS Integration

Search | 17 min read

Relational databases are optimized for exact lookup and transactional correctness. Search engines are optimized for relevance ranking, fuzzy matching, and text analytics at scale.

When PostgreSQL FTS Is Enough

ALTER TABLE articles ADD COLUMN search_vec tsvector; UPDATE articles SET search_vec = to_tsvector('english', title || ' ' || body); CREATE INDEX idx_articles_search ON articles USING GIN(search_vec); SELECT id, title FROM articles WHERE search_vec @@ plainto_tsquery('db indexing') ORDER BY ts_rank(search_vec, plainto_tsquery('db indexing')) DESC;

When to Add Elasticsearch/OpenSearch

  • Need typo tolerance, synonyms, stemming, language analyzers.
  • Need relevance tuning and faceted filtering at high scale.
  • Need near real-time dashboards over logs and documents.
-- Common architecture: -- PostgreSQL = source of truth -- CDC/Outbox -> Kafka -> Elasticsearch indexer -- Search API reads from Elasticsearch -- Mutations still go to PostgreSQL

Summary

  • Keep transactional truth in RDBMS; use search index as a read model.
  • Plan for eventual consistency between DB and search index.
  • Use CDC/outbox for reliable indexing pipelines.
