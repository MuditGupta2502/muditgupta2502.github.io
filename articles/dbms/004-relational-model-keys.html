The Relational Model & Keys: Unique Identification in Databases

Database Design | 11 min read

Introduction to the Relational Model

The Relational Model, proposed by E.F. Codd in 1970, is the foundation of modern database systems. It represents data as a collection of tables (relations) with rows (tuples) and columns (attributes).

Key features of the Relational Model:

  • Table Structure: Data organized in rows and columns
  • Atomic Values: Each attribute contains a single, indivisible value
  • Unique Rows: No duplicate rows in a table
  • Column Independence: Columns can be reordered without changing meaning
  • SQL Support: Standardized query language for manipulation

Terminology

Understanding Keys

A key is an attribute or combination of attributes that uniquely identifies each tuple (row) in a relation. Keys are crucial for maintaining data integrity and establishing relationships between tables.

Types of Keys

Candidate Key: An attribute or minimal combination of attributes that can uniquely identify each tuple. A relation can have multiple candidate keys.

  • Minimal: Cannot be reduced further
  • Unique: No two tuples have the same value
  • Example: In a Student table, both SSN (Social Security Number) and email could be candidate keys

Primary Key: The candidate key chosen as the main identifier for a tuple. Only one primary key per relation.

  • Must be unique
  • Cannot be NULL
  • Usually immutable (doesn't change)
  • Example: student_id is chosen as primary key over email

Alternate Key: A candidate key that is not chosen as the primary key.

  • Also provides unique identification
  • Example: email is an alternate key if student_id is the primary key

Foreign Key: An attribute that refers to the primary key of another table. Establishes relationships between tables.

  • Creates referential integrity constraint
  • Can be NULL (in many-to-one relationships)
  • Example: In an Enrollment table, student_id is a foreign key referencing Student table

Composite Key: A primary key composed of two or more attributes.

  • Used when a single attribute cannot uniquely identify
  • Example: In CourseEnrollment table, (student_id, course_id) together form the composite key

Super Key: Any attribute or combination of attributes that can uniquely identify a tuple. Includes candidate and primary keys.

  • Not minimal (may contain unnecessary attributes)
  • Example: (student_id, name, email) is a super key but not minimal

Key Selection Example

Consider a Customer table with attributes: customer_id, email, phone, SSN, name. 

Candidate Keys: customer_id, email, SSN
Primary Key:    customer_id (chosen)
Alternate Keys: email, SSN
Super Key:      (customer_id, email, phone, name)

Referential Integrity & Foreign Keys

Foreign keys enforce referential integrity, ensuring that references to data in other tables are valid.

Rules:

  • A foreign key value must either exist in the referenced table or be NULL
  • Cannot reference a non-existent primary key
  • Prevents orphaned records (records with invalid references)

Actions on foreign key violation:

  • RESTRICT: Prevent the operation (delete parent will fail if children exist)
  • CASCADE: Propagate the change (delete parent, children are also deleted)
  • SET NULL: Set foreign key to NULL
  • SET DEFAULT: Set foreign key to default value

Practical Example: University Database

Students Table:
- student_id (PRIMARY KEY)
- email (UNIQUE, ALTERNATE KEY)
- name
- date_of_birth

Courses Table:
- course_id (PRIMARY KEY)
- course_name
- department_id (FOREIGN KEY → Department)

Enrollment Table:
- student_id (COMPOSITE PRIMARY KEY, FOREIGN KEY → Students)
- course_id (COMPOSITE PRIMARY KEY, FOREIGN KEY → Courses)
- enrollment_date

Key Constraints

Uniqueness Constraint: Ensures no two tuples have the same value for a key.

Not-Null Constraint: Primary keys must have values; they cannot be NULL.

Domain Constraint: Attribute values must be within the defined domain (e.g., age ≥ 0).

Choosing a Primary Key

Guidelines for selecting a good primary key:

  • Uniqueness: Must uniquely identify each record
  • Minimal: Should not contain unnecessary attributes
  • Immutable: Should not change over time (avoid using names or addresses)
  • Simple: Preferably single attribute for performance
  • Natural or Surrogate: Use surrogate keys (auto-increment) when natural keys are unreliable

Summary

Keys are fundamental to relational database design. They ensure data integrity, establish relationships between tables, and enable efficient retrieval. Understanding candidate keys, primary keys, foreign keys, and their roles is essential for designing robust databases.
