The Relational Model & Keys: Deep Dive

Database Design | 20 min read

Foundation: The Relational Model

E.F. Codd introduced the relational model in his landmark 1970 paper A Relational Model of Data for Large Shared Data Banks. The model is built on set theory and first-order predicate logic. Data is represented as relations (tables), and operations on data are expressed in relational algebra or relational calculus. The DBMS determines the best physical strategy to answer such requests.

The key innovation was separating what data users want from how it is physically stored. This logical independence is why the relational model still dominates transactional software more than fifty years later.

Properties of a Relation

A valid relation must satisfy all of the following:

  • Each cell is atomic: Every attribute value is a single, indivisible unit. Multi-valued or nested values violate 1NF.
  • Each row is unique: No two tuples are identical. Enforced by the primary key.
  • Column names are distinct: Every attribute has a unique name within the relation.
  • Row order is immaterial: Relations are sets. Physical ordering has no meaning at the logical level.
  • Column order is immaterial: Columns are identified by name.
  • All values in a column share the same domain.

What Is a Key?

A key is an attribute or minimal set of attributes whose values uniquely identify every tuple in a relation. Keys are the mechanism by which rows are identified without ambiguity, relationships between tables are expressed, and integrity constraints are enforced.

Types of Keys

1. Super Key

A super key is any set of one or more attributes that can uniquely identify each tuple. Super keys are not required to be minimal.

Students(student_id, name, email, phone, dob) Super keys (all uniquely identify a student): {student_id} {email} {student_id, name} -- redundant but still unique {student_id, email, phone} -- also redundant

Every candidate key is a super key. Not every super key is a candidate key.

2. Candidate Key

A candidate key is a minimal super key — no attribute can be removed while still guaranteeing uniqueness. A relation may have more than one candidate key.

Students(student_id, email, phone, name) Candidate keys (minimal unique identifiers): {student_id} {email} {student_id, email} is a super key but NOT a candidate key because removing either attribute still leaves a unique identifier.
A candidate key cannot contain a NULL value and must remain unique across all current and future tuples.

3. Primary Key

The primary key is the candidate key chosen by the designer as the official row identifier.

  • Unique: no two rows share the same value.
  • NOT NULL: every row must have a value.
  • Stable: should rarely or never change after being set.
  • Minimal: single-column where possible for join and index performance.
CREATE TABLE Students ( student_id INT PRIMARY KEY, email VARCHAR(100) UNIQUE NOT NULL, -- alternate key name VARCHAR(100) NOT NULL, dob DATE );

4. Alternate Key

Any candidate key not chosen as the primary key. Still unique and non-null; enforced via a UNIQUE constraint. In the Students example above, email is an alternate key.

5. Composite Key

A composite key is a primary (or candidate) key formed from two or more attributes. No single attribute is sufficient for uniqueness; only the combination is unique and minimal.

-- A student can enroll in many courses; a course has many students. -- Each (student, course, semester) combination is unique. CREATE TABLE Enrollments ( student_id INT NOT NULL, course_id INT NOT NULL, semester CHAR(6) NOT NULL, -- e.g., 2026S1 grade CHAR(2), PRIMARY KEY (student_id, course_id, semester), FOREIGN KEY (student_id) REFERENCES Students(student_id), FOREIGN KEY (course_id) REFERENCES Courses(course_id) );

6. Foreign Key

A foreign key is an attribute (or set of attributes) in one relation that references the primary key (or unique key) of another relation. It expresses a referential relationship and enforces referential integrity.

-- dept_id in Students is a foreign key referencing Departments Students (student_id [PK], name, email, dept_id [FK → Departments.dept_id]) Departments (dept_id [PK], dept_name, hod)
  • A FK value must match an existing PK value in the referenced table, or be NULL.
  • A FK does not have to be unique — many students can belong to the same department.

Most tables use surrogate primary keys and enforce natural uniqueness with UNIQUE constraints on the relevant columns.

Integrity Constraints

Entity Integrity

No attribute of a primary key may be NULL. A row without a primary key cannot be uniquely identified.

Referential Integrity

Every non-null foreign key value must match an existing primary key in the referenced table. This prevents orphan records — rows that reference something that does not exist.

-- Rejected if dept_id 99 does not exist in Departments INSERT INTO Students (student_id, name, dept_id) VALUES (201, 'Aisha', 99); -- ERROR: foreign key constraint violation

Domain Constraint

Every value must belong to the defined domain of its attribute.

age INT CHECK (age >= 0 AND age <= 150), email VARCHAR(255) CHECK (email LIKE '%@%')

NOT NULL Constraint

Attributes that must always have values are declared NOT NULL. Unlike entity integrity (which applies only to PK attributes), NOT NULL can apply to any attribute.

Referential Actions

When a referenced row is deleted or updated, the DBMS decides what to do with dependent rows based on the declared action.

CREATE TABLE Enrollments ( student_id INT, course_id INT, FOREIGN KEY (student_id) REFERENCES Students(student_id) ON DELETE CASCADE ON UPDATE CASCADE, FOREIGN KEY (course_id) REFERENCES Courses(course_id) ON DELETE RESTRICT );

Relational Algebra — The Formal Query Language

Relational algebra defines operations on relations that return new relations. SQL is built on top of these operations.

Selection (σ)

Filters rows satisfying a condition. σcondition(R)

σ age>20 (Students) -- SQL: SELECT * FROM Students WHERE age > 20;

Projection (π)

Returns only specified columns, removing duplicates. πattrs(R)

π name, email (Students) -- SQL: SELECT DISTINCT name, email FROM Students;

Cartesian Product (×)

Pairs every row of A with every row of B. If A has m rows and B has n rows, result has m×n rows. The raw foundation of joins.

Natural Join (⋈)

Combines tuples from two relations that have equal values on all shared attributes, then removes duplicate columns.

Students ⋈ Enrollments -- SQL: SELECT * FROM Students NATURAL JOIN Enrollments;

Set Operations (∪, ∩, −)

Require union-compatible schemas (same number of attributes, compatible domains).

-- All names appearing in Students OR in Alumni π name(Students) ∪ π name(Alumni) -- Students enrolled in BOTH Maths AND Physics EnrolledMaths ∩ EnrolledPhysics

Rename (ρ)

Renames a relation or individual attributes. ρNewName(R). Used to avoid ambiguity in self-joins.

Introduction to Functional Dependency

A functional dependency X → Y means: knowing the value of X is enough to determine the value of Y. If two tuples have the same X value, they must also have the same Y value.

In Students(student_id, name, email, dept_id): student_id → name (id determines name) student_id → email student_id → dept_id dept_id → dept_name (if dept_name were stored here) In Enrollments(student_id, course_id, semester, grade): {student_id, course_id, semester} → grade (PK → non-key attribute)

Functional dependencies are the formal basis of normalization. When non-key attributes depend on only part of the key (partial dependency) or on other non-key attributes (transitive dependency), the table should be decomposed. This is covered fully in the normalization article.

Complete University Schema

CREATE TABLE Departments ( dept_id INT PRIMARY KEY, dept_name VARCHAR(100) UNIQUE NOT NULL ); CREATE TABLE Students ( student_id INT PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(255) UNIQUE NOT NULL, dob DATE, dept_id INT, FOREIGN KEY (dept_id) REFERENCES Departments(dept_id) ON DELETE SET NULL ); CREATE TABLE Courses ( course_id INT PRIMARY KEY, course_name VARCHAR(150) NOT NULL, credits INT NOT NULL CHECK (credits BETWEEN 1 AND 6), dept_id INT, FOREIGN KEY (dept_id) REFERENCES Departments(dept_id) ); CREATE TABLE Enrollments ( student_id INT NOT NULL, course_id INT NOT NULL, semester CHAR(6) NOT NULL, grade CHAR(2), PRIMARY KEY (student_id, course_id, semester), FOREIGN KEY (student_id) REFERENCES Students(student_id) ON DELETE CASCADE, FOREIGN KEY (course_id) REFERENCES Courses(course_id) ON DELETE CASCADE );

What You Should Remember

  • Every candidate key is a super key; not every super key is a candidate key.
  • One candidate key is chosen as the primary key — unique, non-null, stable.
  • Foreign keys express inter-table relationships and enforce referential integrity.
  • Composite keys are used when no single attribute alone is unique.
  • Referential actions (CASCADE, SET NULL, RESTRICT) govern how the DBMS responds to changes in referenced rows.
  • Functional dependencies describe which attributes determine others — the formal foundation of normalization.
