Isolation Levels: Phenomena, MVCC Behavior and Real-World Usage

Transactions | 26 min read

SQL defines four isolation levels — each preventing a specific set of read anomalies. But the SQL-92 specification was written with locking in mind; MVCC databases like PostgreSQL implement these levels differently, sometimes offering stronger guarantees than the standard requires. Understanding what your database actually does at each level is more important than memorizing the standard definition.

READ UNCOMMITTED

-- Allows dirty reads: can see uncommitted changes from other transactions -- SQL standard: the weakest level; allows dirty reads -- Example: -- T1 begins, updates salary = 100000 (not yet committed) -- T2 (READ UNCOMMITTED): SELECT salary → reads 100000 even though T1 hasn't committed -- T1 rolls back → salary was never 100000 in reality -- T2 made a decision based on data that never existed → incorrect -- PostgreSQL: READ UNCOMMITTED behaves identically to READ COMMITTED -- PostgreSQL never allows dirty reads — its MVCC model makes it structurally impossible -- The lowest practical level in PostgreSQL is READ COMMITTED -- When useful in other databases (MySQL READ UNCOMMITTED): -- Approximate COUNT(*) on huge tables without blocking -- Analytics on data where rough approximations are acceptable -- Generally avoided in production

READ COMMITTED (Default in PostgreSQL)

-- Only sees committed data; no dirty reads -- Each STATEMENT gets a fresh snapshot — NOT the transaction start -- PostgreSQL READ COMMITTED: -- Snapshot is taken at the START OF EACH SQL STATEMENT (not transaction start) -- This means: within one transaction, two identical SELECT statements may return different rows -- Demonstrating non-repeatable read in READ COMMITTED: -- Session 1 (READ COMMITTED): BEGIN; SELECT salary FROM employees WHERE id = 1; -- returns 80000 (snapshot for this statement) -- (session 2 updates and commits here) SELECT salary FROM employees WHERE id = 1; -- returns 90000 ← different! (new snapshot) COMMIT; -- Demonstrating phantom read: -- Session 1: BEGIN; SELECT COUNT(*) FROM orders WHERE status = 'pending'; -- returns 5 -- (session 2 inserts a new pending order and commits here) SELECT COUNT(*) FROM orders WHERE status = 'pending'; -- returns 6 ← phantom row COMMIT; -- Practical implication: READ COMMITTED can cause issues in multi-statement transactions -- where your code reads data, makes a decision, then writes based on that decision -- Another transaction may have changed the data between your READ and WRITE

When READ COMMITTED Causes Bugs

-- Bug pattern: read-before-write -- Goal: "transfer items from category A to category B" BEGIN; -- Step 1: Read count of A items SELECT COUNT(*) FROM products WHERE category = 'A'; -- returns 100 -- (Another transaction updates some A products to category C here and commits) -- Step 2: Move A items to B (now there are only 80 A items, not 100) UPDATE products SET category = 'B' WHERE category = 'A'; -- The UPDATE is correct (updates current 80 rows) -- But any subsequent SELECT within this transaction WILL see the new 80, not 100 -- If you used the count 100 to allocate output space, you have a bug COMMIT; -- Solution: Use REPEATABLE READ so all reads in the transaction see the same snapshot

REPEATABLE READ

-- Snapshot taken at TRANSACTION START (not per-statement) -- All reads within the transaction see the same consistent snapshot -- Prevents: dirty reads, non-repeatable reads -- SQL-92 standard allows phantoms; PostgreSQL REPEATABLE READ also prevents phantoms -- PostgreSQL REPEATABLE READ is actually Snapshot Isolation: -- One snapshot for the entire transaction -- Other transactions' commits are invisible to you, no matter how long you wait BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT salary FROM employees WHERE id = 1; -- 80000 (snapshot taken here) -- (Session 2 updates salary to 90000 and commits) SELECT salary FROM employees WHERE id = 1; -- still 80000 ← same snapshot! -- (Session 2 inserts a new employee) SELECT COUNT(*) FROM employees; -- still same count as first read ← no phantom COMMIT; -- Write behavior: "first committer wins" -- If T1 and T2 both read the same row at snapshot start -- T1 updates and commits → T1 wins -- T2 tries to update → PostgreSQL checks: has this row been modified since T2's snapshot? -- YES (T1 updated it) → PostgreSQL uses "first committer wins": -- At REPEATABLE READ: T2's UPDATE proceeds (updates on top of T1's committed row) -- This can cause write skew anomalies

Write Skew at REPEATABLE READ

-- Classic write skew example: meetup RSVP system -- Rule: at most 100 attendees -- Current count: 99 -- T1 (REPEATABLE READ): BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT COUNT(*) FROM rsvps WHERE event_id = 1; -- returns 99 (< 100, ok to add) -- (T2 runs now before T1 finishes) -- T2 (REPEATABLE READ): BEGIN ISOLATION LEVEL REPEATABLE READ; SELECT COUNT(*) FROM rsvps WHERE event_id = 1; -- returns 99 (same, ok to add) INSERT INTO rsvps (event_id, user_id) VALUES (1, 2002); COMMIT; -- now count = 100 -- T1 continues: INSERT INTO rsvps (event_id, user_id) VALUES (1, 1001); COMMIT; -- THIS SUCCEEDS at REPEATABLE READ — now count = 101! -- T1's snapshot still showed count=99; its INSERT was not conflicting with T2's INSERT -- Rule violated: 101 attendees when max is 100 -- Fix: Use SERIALIZABLE level, or use SELECT FOR UPDATE (pessimistic) to serialize

SERIALIZABLE

-- Equivalent to serial execution — transactions appear to run one after another -- Prevents all anomalies: dirty reads, non-repeatable reads, phantoms, write skew -- PostgreSQL uses SSI (Serializable Snapshot Isolation) — MVCC-based, not lock-based -- Same RSVP example with SERIALIZABLE: BEGIN ISOLATION LEVEL SERIALIZABLE; SELECT COUNT(*) FROM rsvps WHERE event_id = 1; -- 99 INSERT INTO rsvps (event_id, user_id) VALUES (1, 1001); COMMIT; -- If T2 committed a conflicting INSERT while T1 was running, T1 gets: -- ERROR: could not serialize access due to read/write dependencies among transactions -- Application must catch this error and retry the transaction -- SSI tracks "anti-dependencies": -- T1 reads something that T2 wrote (rw-anti-dep: T1 depends on T2's write) -- T2 reads something that T1 wrote (rw-anti-dep: T2 depends on T1's write) -- Cycle detected → one transaction aborted -- How often does serialization failure happen? -- Less often than you think for well-structured applications -- SELECT FOR UPDATE still works inside SERIALIZABLE for explicit locking -- READ ONLY transactions in SERIALIZABLE: no anti-dependency tracking (no overhead)

Practical Guidelines

Use CaseRecommended LevelReason
Simple read queries, dashboardsREAD COMMITTEDFast; stale data by milliseconds is fine
Report generation (multi-query)REPEATABLE READConsistent snapshot across all report queries
Financial transactionsSERIALIZABLE or RR + SELECT FOR UPDATEPrevent double-spend, write skew anomalies
Inventory/booking systemsSERIALIZABLE or pessimistic lockingMust prevent over-booking (write skew)
Idempotent background jobsREAD COMMITTEDPerformance acceptable; can retry safely
Long-running batch jobsREPEATABLE READConsistent view; no surprises mid-batch
Common production bug — READ COMMITTED + multi-step logic: If your code does read, compute, then write in READ COMMITTED, the data you read might be stale by the time you write. Upgrade to REPEATABLE READ or use SELECT ... FOR UPDATE to lock the rows you'll update.

Summary

  • READ COMMITTED: fresh snapshot per-statement; most common default; allows non-repeatable reads and phantoms
  • REPEATABLE READ: single snapshot per-transaction; PostgreSQL also prevents phantoms (stronger than standard)
  • SERIALIZABLE (SSI): prevents all anomalies including write skew; may abort with serialization error; applications must retry
  • PostgreSQL's MVCC means readers never block writers at any isolation level
  • Write skew requires SERIALIZABLE or explicit pessimistic locking (SELECT FOR UPDATE)
  • Use REPEATABLE READ for report generation; SERIALIZABLE for financial/booking critical paths
