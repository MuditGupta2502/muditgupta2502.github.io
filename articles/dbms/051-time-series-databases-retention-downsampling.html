Time-Series Databases: Retention, Downsampling and High-Write Design

Specialized DBs | 17 min read

Time-series workloads are append-heavy and query by time ranges. The schema and retention strategy matter more than complex normalization.

Schema Pattern

CREATE TABLE metrics ( ts TIMESTAMPTZ NOT NULL, service TEXT NOT NULL, metric_name TEXT NOT NULL, value DOUBLE PRECISION NOT NULL, tags JSONB ); CREATE INDEX idx_metrics_service_name_ts ON metrics (service, metric_name, ts DESC);

Retention and Downsampling

-- Keep raw 10-second metrics for 30 days DELETE FROM metrics WHERE ts < NOW() - INTERVAL '30 days'; -- Store 5-minute rollups for long-term trend analysis INSERT INTO metrics_rollup_5m(service, metric_name, bucket, avg_value, p95_value) SELECT service, metric_name, date_trunc('minute', ts) - (EXTRACT(minute FROM ts)::int % 5) * INTERVAL '1 minute', AVG(value), percentile_cont(0.95) WITHIN GROUP (ORDER BY value) FROM metrics WHERE ts >= NOW() - INTERVAL '1 day' GROUP BY service, metric_name, 3;

Summary

  • Design for append + range scans: index by dimension columns then timestamp.
  • Use retention windows to control cost and table size.
  • Downsample old data for long-term analytics without keeping full raw detail.
