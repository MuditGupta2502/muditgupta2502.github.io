Data Models & Entity-Relationship (ER) Model

Database Design | 10 min read

What is a Data Model?

A data model is a blueprint that describes how data is organized, structured, and related in a database. It serves as a communication tool between stakeholders, developers, and database designers.

Data models exist at different levels of abstraction:

  • Conceptual Data Model: High-level, technology-independent view (ER diagrams)
  • Logical Data Model: Intermediate level showing relational structure (Relational Model)
  • Physical Data Model: Low-level, implementation-specific (actual database schema)

Types of Data Models

1. Relational Model — Data organized in tables (relations) with rows and columns. Most common in business applications (MySQL, PostgreSQL, Oracle).

2. Hierarchical Model — Data organized in a tree structure with parent-child relationships. Limited and rarely used today.

3. Network Model — Allows multiple parents for a node, providing more flexibility than hierarchical. Complex and outdated.

4. Entity-Relationship Model — Conceptual model focusing on entities and their relationships, providing a visual representation before implementation.

5. NoSQL Models — Document model (MongoDB), Key-value model (Redis), Graph model (Neo4j), Wide-column model (Cassandra).

What is Entity-Relationship (ER) Model?

The Entity-Relationship model, introduced by Peter Chen in 1976, is a conceptual data model used to design databases before physical implementation. It represents entities, attributes, and their relationships in an easy-to-understand visual format.

The ER model is widely used because it:

  • Provides a clear, visual representation of data structure
  • Is independent of any database system
  • Facilitates communication between technical and non-technical stakeholders
  • Helps eliminate data redundancy through normalization

Components of ER Model

Entity: A real-world object or concept that has independent existence. Examples: Student, Course, Employee, Product.

  • Represented as a rectangle in ER diagrams
  • Strong Entity: Can exist independently (Student)
  • Weak Entity: Depends on another entity for existence (Enrollment depends on Student and Course)

Attribute: Properties or characteristics of an entity. Examples: Student has name, roll_number, email, phone.

  • Represented as an oval in ER diagrams
  • Simple Attribute: Cannot be broken down (age)
  • Composite Attribute: Can be decomposed (address = city + street + zip code)
  • Derived Attribute: Calculated from other attributes (age = current_year - birth_year)
  • Multi-valued Attribute: Can have multiple values (phone_numbers: ["123", "456", "789"])

Relationship: Association between two or more entities. Examples: Student enrolls in Course, Employee works in Department.

  • Represented as a diamond in ER diagrams
  • Describes how entities interact and relate

Types of Relationships

One-to-One (1:1) — Each entity of one type relates to at most one entity of another type. Example: A person has one passport, a passport belongs to one person.

One-to-Many (1:N) — One entity relates to many entities of another type. Example: One department has many employees, but each employee belongs to one department.

Many-to-Many (M:N) — Many entities relate to many entities of another type. Example: Students enroll in many courses, and each course has many students. Requires an intermediate junction table.

Self-Referencing: Entity relates to itself. Example: Employee has a manager, which is also an employee.

Cardinality and Participation

Cardinality: Specifies the number of entities that can participate in a relationship. Shown as 1:1, 1:M, or M:N.

Participation: Specifies whether an entity must participate in a relationship (total participation) or may optionally participate (partial participation).

  • Total (Mandatory): Shown with double line — every instance must participate
  • Partial (Optional): Shown with single line — instances may not participate

Example: In a University database, every Student must enroll in at least one Course (total participation), but a Course may exist with zero enrolled students (partial participation).

ER Diagram Symbols

Entity          → Rectangle
Attribute       → Oval
Relationship    → Diamond
Primary Key     → Underlined attribute
One-to-One      → 1:1 label
One-to-Many     → 1:N label
Many-to-Many    → M:N label
Weak Entity     → Double rectangle
Derived Attr    → Dashed oval

Extended ER (EER) Model

The Extended ER model adds advanced features:

  • Generalization: Grouping similar entities (Vehicle is generalization of Car, Bike, Truck)
  • Specialization: Breaking down generic entity into specific types
  • Inheritance: Child entities inherit attributes from parent entities
  • Aggregation: Treating relationship as an entity for further relationships

ER to Relational Conversion

Once an ER diagram is finalized, it's converted to a relational schema:

  • Each entity becomes a table
  • Each attribute becomes a column
  • Primary key of an entity becomes the primary key of the table
  • Relationships are represented using foreign keys
  • Many-to-Many relationships create junction tables

Summary

The ER model is a powerful tool for conceptual database design. It provides a clear, visual way to understand data structure and relationships before implementation. By understanding entities, attributes, relationships, and their types, you can design databases that are normalized, efficient, and maintainable.
