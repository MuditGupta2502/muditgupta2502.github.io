Database Migration and Event-Driven Architecture: CDC, Kafka, Outbox and Debezium

Architecture | 24 min read

Production schema changes are risky because applications and data evolve together. The safest migrations are backward-compatible, staged, and observable.

Expand and Contract Migration Pattern

-- Goal: rename users.full_name -> users.display_name without downtime -- Phase 1 (Expand): add new column, keep old column ALTER TABLE users ADD COLUMN display_name TEXT; -- Phase 2: dual-write from application -- INSERT/UPDATE writes both full_name and display_name -- Phase 3: backfill historical rows in batches UPDATE users SET display_name = full_name WHERE display_name IS NULL LIMIT 10000; -- loop in app/job until complete -- Phase 4: read-switch -- application reads display_name, still writes both for safety window -- Phase 5 (Contract): remove old column when safe ALTER TABLE users DROP COLUMN full_name;

CDC (Change Data Capture)

-- CDC captures row changes from DB logs (WAL/binlog) and streams them to consumers -- Typical flow: PostgreSQL WAL -> Debezium -> Kafka -> downstream services -- Why CDC: -- 1. Build read models/search indexes asynchronously -- 2. Feed analytics warehouse in near real-time -- 3. Integrate microservices without direct DB coupling -- Debezium emits events like: -- {"op":"c","before":null,"after":{"id":123,"status":"paid"}} -- {"op":"u","before":{"status":"pending"},"after":{"status":"paid"}}

Outbox Pattern

-- Problem: DB commit succeeds but Kafka publish fails -> data/event inconsistency -- Solution: write business row and outbox row in SAME DB transaction CREATE TABLE outbox_events ( id BIGSERIAL PRIMARY KEY, aggregate_type TEXT NOT NULL, aggregate_id TEXT NOT NULL, event_type TEXT NOT NULL, payload JSONB NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW(), published BOOLEAN DEFAULT FALSE ); BEGIN; INSERT INTO orders (id, customer_id, status, total) VALUES (1001, 42, 'created', 999.00); INSERT INTO outbox_events (aggregate_type, aggregate_id, event_type, payload) VALUES ('order', '1001', 'OrderCreated', '{"order_id":1001,"total":999.00}'); COMMIT; -- Publisher service reads unpublished outbox rows and sends to Kafka -- Then marks published=true (idempotent with unique event key)

Migration Tooling

-- Flyway migration naming: -- V1__create_users.sql -- V2__add_display_name.sql -- V3__backfill_display_name.sql -- Liquibase supports declarative XML/YAML/SQL changelogs with rollback blocks -- Operational guidance: -- 1. Keep each migration small and reversible -- 2. Run in off-peak windows if it touches large tables -- 3. Monitor lock waits and query latency during rollout
Anti-pattern: deploy app code that assumes new column exists before migration runs. Always deploy backward-compatible code first, then run migration, then switch reads/writes.

Summary

  • Use expand/contract for zero-downtime schema evolution.
  • CDC streams committed changes from transaction logs to downstream systems.
  • Outbox pattern gives reliable event publishing without distributed transactions.
  • Use Flyway/Liquibase to version and automate migrations safely.
