DBMS Architecture: Three-Tier Model & Data Independence

Architecture & Design | 9 min read

Overview of DBMS Architecture

DBMS architecture is based on a three-tier model also known as ANSI-SPARC architecture. This design separates the system into three independent layers: External, Conceptual, and Internal layers. Each layer has its own schema and provides a different view of the database.

The key benefit of this architecture is data independence — changes at one layer don't affect other layers.

Layer 1: External Layer (User View)

The external layer represents how individual users and applications perceive the database. Different users have different views based on their roles and needs.

  • Multiple Views: Different departments see different subsets of data
  • User-Specific: A customer service user sees only customer records; an analyst sees aggregated reports
  • Application-Specific: A web application may see a subset of tables optimized for its queries
  • Security: Users access only the data their role allows

Example: In a hospital database, a doctor sees patient medical history, while billing staff sees only payment information.

Layer 2: Conceptual Layer (Logical View)

The conceptual layer defines the logical structure of the entire database independent of physical storage. It represents what data exists and how it's logically organized.

  • Database Schema: Complete definition of all tables, columns, and relationships
  • Relationships: How entities relate to each other (one-to-many, many-to-many)
  • Constraints: Primary keys, foreign keys, unique constraints, check constraints
  • Unified View: Single source of truth for all data structure

What it looks like: 

Users Table (user_id, username, email, role)
Products Table (product_id, name, price, category_id)
Orders Table (order_id, user_id, product_id, quantity, order_date)

Changes to the conceptual schema (like adding a new column) are made once here, and all views can adapt.

Layer 3: Internal Layer (Physical View)

The internal layer describes how data is physically stored on disk. It deals with storage structures, file organization, and optimization.

  • File Organization: How tables are stored as files and blocks
  • Indexing Strategy: Which columns are indexed for faster retrieval
  • Storage Allocation: How disk space is allocated to tables
  • Compression: Data compression techniques for saving space
  • Clustering: How related data is grouped together physically

Example: An application doesn't need to know that User data is stored in a B-tree index or that it's split across multiple disk blocks.

Logical Data Independence

Logical data independence is the ability to modify the conceptual schema without changing external views or application code.

Scenario: Adding a new column "phone_number" to the Users table. Existing applications and views continue working without modification.

  • Flexible schema evolution
  • New applications can use new columns
  • Old applications ignore the new column

Physical Data Independence

Physical data independence is the ability to modify the internal schema (storage structures) without affecting the conceptual or external schemas.

Scenario: The DBA decides to add indexes on frequently queried columns or move data to faster storage. Applications continue running without changes.

  • Optimization without code changes
  • Storage migration transparency
  • Index changes don't affect queries

Database Users

Different users interact with the DBMS at different layers:

  • End Users: Access the Database Management System through applications; interact with the external layer
  • Application Developers: Write code to query the conceptual layer using SQL
  • Database Administrators (DBAs): Manage the internal layer, optimization, and security
  • Database Designers: Design the conceptual schema and relationships

DBMS Interfaces

Users interact with DBMS through various interfaces:

  • Query Interface (SQL): Write SELECT, INSERT, UPDATE, DELETE statements
  • Forms Interface: GUI-based data entry forms
  • Report Interface: Generate formatted reports
  • API Interface: Application programming interfaces for developers
  • Command-Line Interface (CLI): Direct database interaction

Data Model

The data model defines the structure and relationships of data. Common types include:

  • Relational Model: Tables with rows and columns (SQL databases)
  • Document Model: JSON/BSON documents (MongoDB)
  • Key-Value Model: Simple key-value pairs (Redis)
  • Graph Model: Nodes and edges representing relationships

Summary

The three-tier DBMS architecture provides separation of concerns:

  • External Layer: User-specific views
  • Conceptual Layer: Logical database design
  • Internal Layer: Physical storage and optimization

This separation ensures data independence, allowing changes at one layer without affecting others. Understanding this architecture is crucial for designing scalable, maintainable databases and building robust applications.
