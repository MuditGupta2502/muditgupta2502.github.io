DBMS Architecture: Three-Tier Model & Data Independence

Architecture & Design | 16 min read

Architecture Exists to Manage Complexity

A DBMS is not just a storage engine. It is a layered system that must satisfy many competing goals at once: user-friendly access, logical correctness, physical efficiency, security, recovery, and scalability. If all of these concerns were mixed together, even a small schema change would break applications constantly.

Database architecture solves this by separating the concerns of what users see, what data logically exists, and how data is physically stored. The standard academic explanation of this idea is the ANSI-SPARC three-schema architecture.

The Three Levels of DBMS Architecture

  • External level: User-specific views of data
  • Conceptual level: Overall logical structure of the database
  • Internal level: Physical storage details

These are often called schemas because each level has its own way of describing the same database.

1. External Level

The external level describes how different users or applications see the database. A cashier, an HR executive, and a data analyst may all access the same underlying data but require different subsets, formats, and permissions.

Why Multiple External Views Are Useful

  • They reduce complexity by showing only relevant data.
  • They enforce security by hiding unauthorized fields.
  • They let different teams work with the same database in different ways.
Hospital database:
- Doctor view: patient name, diagnosis, treatment history
- Billing view: patient name, bill amount, payment status
- Reception view: patient name, appointment time, assigned doctor

All three views may refer to the same patient record, but none of them needs to see everything.

2. Conceptual Level

The conceptual level is the logical blueprint of the entire database. It defines what entities exist, which attributes belong to them, what relationships connect them, and what rules maintain validity.

What the Conceptual Schema Includes

  • Table and relation definitions
  • Attributes and data types
  • Primary and foreign keys
  • Constraints and business rules
  • Relationship definitions
Students(student_id, name, email, dept_id)
Departments(dept_id, dept_name)
Courses(course_id, course_name, credits)
Enrollments(student_id, course_id, semester)

Notice that the conceptual level says nothing about disk pages, file blocks, or B-trees. It focuses on meaning, not storage mechanics.

3. Internal Level

The internal level describes how data is actually stored inside the system. This is where the DBMS cares about pages, blocks, record formats, compression, indexes, partitioning, clustering, and access paths.

Typical Internal-Level Concerns

  • Whether a table is stored row-wise or column-wise
  • Which indexes exist and on which columns
  • Whether data is partitioned across files or machines
  • How free space is tracked and reused
  • How logs are maintained for recovery

Application developers usually do not interact with this level directly, but performance and reliability depend on it heavily.

Mappings Between Levels

The three levels are useful only if the DBMS can translate between them. Two important mappings make that possible:

  • External-conceptual mapping: Connects user views to the logical schema
  • Conceptual-internal mapping: Connects logical definitions to physical storage

These mappings are the technical reason data independence is possible. Without them, every schema change would cascade directly into user code.

Data Independence

Data independence is the ability to change one level of the database without forcing changes in the next higher level. It is one of the strongest arguments for layered DBMS design.

Logical Data Independence

Logical data independence means the conceptual schema can change without forcing all user views or applications to change.

  • Adding a column to a table
  • Splitting a table into two related tables
  • Creating a new relationship for future features

This is harder to achieve than physical data independence because application queries often depend on logical structure.

Physical Data Independence

Physical data independence means internal storage can change without changing the conceptual schema.

  • Adding or dropping indexes
  • Changing storage engine parameters
  • Moving data to faster disks
  • Reorganizing partitions

This is more common in practice. A DBA can often improve performance significantly without any application code changes.

Centralized vs Client-Server vs Three-Tier Deployment

Students often confuse schema architecture with deployment architecture. They are related but different.

Centralized DBMS

Database, DBMS, and users are effectively tied to one central machine. This is simple but limited in scale and availability.

Client-Server Architecture

Clients send requests to a separate database server. This is common in web applications and enterprise systems.

Three-Tier Application Architecture

Presentation layer -> Application layer -> Database layer

This is not the same as external, conceptual, and internal schemas. One is about application deployment; the other is about database abstraction.

Major DBMS Components Behind the Architecture

  • Query processor: Parses, validates, rewrites, and optimizes queries.
  • Storage manager: Handles files, pages, indexes, and record layout.
  • Transaction manager: Preserves consistency under concurrent access.
  • Recovery manager: Restores the database after failure.
  • Authorization and integrity manager: Enforces permissions and rules.
  • Data dictionary: Stores metadata about tables, constraints, indexes, and users.

Database Users and Their Perspective

  • Naive users: Use forms and applications, not direct SQL.
  • Application developers: Work mostly at the external and conceptual levels.
  • Database designers: Focus on conceptual modeling.
  • DBAs: Work across conceptual and internal levels for performance, security, and maintenance.

Why This Architecture Matters in Practice

Suppose an e-commerce company adds a new feature for discount campaigns. The conceptual schema may get a new Promotions table. The internal layer may receive new indexes on product and campaign columns. The customer-facing application might still expose the same product search screens with minor UI changes only. That is architecture doing its job: local changes stay local.

Common Exam and Interview Confusions

  • External schema is not the same as frontend UI. It is the user-level database view.
  • Conceptual schema is not physical storage. It is the logical design.
  • Physical data independence is usually easier to achieve than logical data independence.
  • Three-schema architecture is different from three-tier web architecture.

Summary

DBMS architecture separates the database into user views, logical structure, and physical storage. That separation creates clarity, security, and flexibility. More importantly, it enables data independence so the system can evolve without breaking everything above it. Once this layered view is clear, data models and schema design become much easier to understand.
