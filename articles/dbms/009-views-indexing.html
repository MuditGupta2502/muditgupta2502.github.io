Views & Indexing: Abstraction and Performance

Database Design & Performance | 9 min read

Database Views

A view is a virtual table created from a query result. It doesn't store data itself but displays data from underlying base tables. Views provide abstraction, security, and simplified data access.

Types of Views

Simple View

Created from a single table without aggregation or joins. 

CREATE VIEW ActiveStudents AS
SELECT student_id, name, email, enrollment_date
FROM Students
WHERE status = 'active';

Complex View

Created from multiple tables using joins, aggregations, or subqueries. 

CREATE VIEW StudentCourseStats AS
SELECT 
  s.student_id,
  s.name,
  COUNT(e.course_id) as courses_enrolled,
  AVG(e.grade) as average_grade
FROM Students s
LEFT JOIN Enrollments e ON s.student_id = e.student_id
GROUP BY s.student_id, s.name;

Indexed View (Materialized View)

A view with an index that physically stores data. Used for frequently accessed complex queries. 

CREATE VIEW StudentCourseSummary AS
SELECT 
  s.student_id,
  s.name,
  COUNT(e.course_id) as total_courses
FROM Students s
LEFT JOIN Enrollments e ON s.student_id = e.student_id
GROUP BY s.student_id, s.name;

-- Create index on view (SQL Server/PostgreSQL)
CREATE INDEX idx_studentcoursesummary 
ON StudentCourseSummary(student_id);

Advantages of Views

  • Abstraction: Hide database complexity from users
  • Security: Show only relevant columns/rows to users
  • Simplification: Pre-join tables for common queries
  • Maintainability: Update the view when underlying schema changes
  • Consistency: Multiple applications use the same business logic

Disadvantages of Views

  • Performance: Complex views with joins can be slow
  • Update Limitations: Some views cannot be updated directly
  • Maintenance: Changes to base tables affect dependent views

Updatable Views

Simple views can be updated (INSERT, UPDATE, DELETE) if they meet certain conditions. Complex views with joins or aggregations are typically read-only. 

-- Updatable view (simple, single table)
UPDATE ActiveStudents 
SET email = 'newemail@example.com' 
WHERE student_id = 5;

Dropping Views

DROP VIEW StudentCourseStats;
DROP VIEW IF EXISTS StudentCourseStats; -- Safe drop

Indexing Deep Dive

An index is an on-disk data structure that speeds up data retrieval. It maps values to row locations, allowing the DBMS to find data without scanning all rows.

Hash Index

Uses a hash function to distribute values. Excellent for equality searches but doesn't support range queries. 

CREATE INDEX idx_email_hash ON Students USING HASH (email);

B-Tree Index

Balanced tree structure — the most common index type. Supports both equality and range queries. Self-balancing ensures logarithmic search time. 

CREATE INDEX idx_age ON Students (age);
SELECT * FROM Students WHERE age BETWEEN 20 AND 25; -- Uses index

Bitmap Index

Uses bit vectors (0s and 1s) to represent data. Efficient for columns with low cardinality (few distinct values). 

CREATE BITMAP INDEX idx_status ON Students (status);
-- Good for status='active' or status='inactive'

Full-Text Index

Optimized for full-text search on document or text columns. 

CREATE FULLTEXT INDEX idx_article_content 
ON Articles (content);

SELECT * FROM Articles 
WHERE MATCH(content) AGAINST('database' IN BOOLEAN MODE);

Composite Index Strategy

Multi-column indexes optimize queries that filter on multiple columns. Order matters! 

-- Create composite index with good selectivity first
CREATE INDEX idx_dept_salary ON Employees(department_id, salary);

-- This query can use the composite index efficiently
SELECT * FROM Employees 
WHERE department_id = 5 AND salary > 50000;

Index Maintenance

Indexes must be maintained as data changes (INSERT, UPDATE, DELETE). This has performance overhead.

  • Rebuild Index: Reorganizes entire index for optimal performance
  • Defrag Index: Consolidates fragmented index pages
-- MySQL
OPTIMIZE TABLE Students;

-- PostgreSQL
REINDEX INDEX idx_age;

-- SQL Server
ALTER INDEX idx_age ON Students REBUILD;

Index Best Practices

  • Index columns used in WHERE, JOIN, and ORDER BY clauses
  • Don't index low-cardinality columns (e.g., gender with only 2 values)
  • Avoid indexing columns updated frequently
  • Regular index maintenance for large tables
  • Monitor index usage; remove unused indexes
  • Balance between read performance (more indexes) and write performance (fewer indexes)

Summary

Views provide logical abstraction and security, while indexes dramatically improve query performance. Understanding when and how to use both is essential for designing databases that are secure, maintainable, and performant. Views hide complexity; indexes accelerate complexity.
