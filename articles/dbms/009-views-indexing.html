Views & Indexing: Abstraction, Security, and Performance

Database Design & Performance | 22 min read

Part 1: Views

What Is a View?

A view is a named, stored SQL query that appears to users as a virtual table. It does not store data itself (with the exception of materialized views); it stores the query definition and reruns it each time the view is queried.

Views serve three main purposes:

  • Abstraction: Hide complex joins or business logic behind a simple name.
  • Security: Show only the columns or rows a user is authorized to see.
  • Logical independence: Applications query the view; the underlying table structure can change without breaking application code.

Creating and Using Views

-- Simple view: active students only CREATE VIEW ActiveStudents AS SELECT student_id, name, email, dept_id FROM Students WHERE status = 'active'; -- Query the view exactly like a table SELECT * FROM ActiveStudents WHERE dept_id = 3; -- Drop a view DROP VIEW ActiveStudents; -- Replace (update) an existing view definition CREATE OR REPLACE VIEW ActiveStudents AS SELECT student_id, name, email, dept_id, gpa FROM Students WHERE status = 'active';

View Types

Simple View

Based on a single base table, no aggregation, no GROUP BY. Usually updatable.

CREATE VIEW CSStudents AS SELECT student_id, name, email FROM Students WHERE dept_id = 1;

Complex View

Based on multiple tables (JOINs), has aggregation or DISTINCT. Generally not directly updatable.

CREATE VIEW StudentEnrollmentSummary AS SELECT s.student_id, s.name, d.dept_name, COUNT(e.course_id) AS courses_enrolled, AVG(s.gpa) AS gpa FROM Students s JOIN Departments d ON s.dept_id = d.dept_id LEFT JOIN Enrollments e ON s.student_id = e.student_id GROUP BY s.student_id, s.name, d.dept_name;

Updatable View

A view is updatable if modifications (INSERT, UPDATE, DELETE) on the view translate unambiguously back to modifications on a single base table. Conditions for updatability:

  • Based on a single base table.
  • No DISTINCT, GROUP BY, HAVING, UNION, or aggregates.
  • No subqueries in the SELECT list.
  • Selected columns map directly to columns in the base table.
-- This simple view is updatable CREATE VIEW ActiveStudents AS SELECT student_id, name, email FROM Students WHERE status = 'active'; -- This update goes through to the Students table UPDATE ActiveStudents SET email = 'new@example.com' WHERE student_id = 101; -- WITH CHECK OPTION: prevents updates that would make the row invisible through the view CREATE VIEW ActiveStudents AS SELECT student_id, name, email, status FROM Students WHERE status = 'active' WITH CHECK OPTION; -- This would be REJECTED because setting status='inactive' hides the row from the view UPDATE ActiveStudents SET status = 'inactive' WHERE student_id = 101;

Materialized View

A materialized view actually stores the result of the query on disk. It is refreshed periodically or on-demand. This trades storage space and potential staleness for dramatically faster read performance on complex aggregation queries.

-- PostgreSQL CREATE MATERIALIZED VIEW DeptGPASummary AS SELECT dept_id, AVG(gpa) AS avg_gpa, COUNT(*) AS student_count FROM Students GROUP BY dept_id; -- Refresh the materialized view (recompute and store latest data) REFRESH MATERIALIZED VIEW DeptGPASummary; -- Concurrent refresh (allows reads while refreshing) REFRESH MATERIALIZED VIEW CONCURRENTLY DeptGPASummary;

Views as a Security Mechanism

-- A table with sensitive data CREATE TABLE Employees ( emp_id INT PRIMARY KEY, name VARCHAR(100), email VARCHAR(255), salary DECIMAL(10,2), ssn CHAR(11) ); -- A safe view for the HR reporting role (hides salary and SSN) CREATE VIEW EmployeeDirectory AS SELECT emp_id, name, email FROM Employees; -- Grant access to the view only; never grant access to the base table GRANT SELECT ON EmployeeDirectory TO hr_reporting_role; REVOKE ALL ON Employees FROM hr_reporting_role;

Part 2: Indexing

What Is an Index?

An index is a separate data structure built on one or more columns of a table. It allows the DBMS to find rows matching a predicate without scanning every row in the table. Think of it as a sorted lookup table mapping column values to physical row locations (heap pointers).

Every index trades write overhead and storage space for faster reads. Adding an index always slows down INSERT, UPDATE, and DELETE on the indexed columns because the index must be maintained in sync with the table.

B-Tree Index — The Default

The vast majority of database indexes are B-trees (technically B+ trees). A B+ tree is a balanced tree where:

  • All leaf nodes are at the same depth (balanced).
  • Leaf nodes contain (key value, row pointer) pairs, sorted by key value.
  • Leaf nodes are linked in a doubly-linked list for efficient range scans.
  • Internal nodes contain only routing keys to guide tree traversal.
-- Conceptual B+ tree for index on Students(gpa): Internal: [7.5 | 8.5] Leaf: ... [6.8,r12] [7.1,r3] -- [7.5,r8] [8.2,r5] -- [8.5,r1] [9.0,r7] [9.5,r2] linked list → → → -- Finding WHERE gpa = 8.2: traverse tree → reach leaf → find r5 → fetch row -- Finding WHERE gpa BETWEEN 7.0 AND 8.0: traverse to 7.0 → scan linked list to 8.0

B-tree indexes support: equality (=), range (<, >, BETWEEN), ORDER BY (if sorted correctly), and prefix LIKE searches (LIKE 'M%').

Creating Indexes

-- Single-column index CREATE INDEX idx_students_dept ON Students(dept_id); -- Multi-column (composite) index CREATE INDEX idx_enroll_student_sem ON Enrollments(student_id, semester); -- Unique index (enforces uniqueness like UNIQUE constraint) CREATE UNIQUE INDEX idx_students_email ON Students(email); -- Partial index (only indexes rows matching a condition) CREATE INDEX idx_active_students ON Students(dept_id) WHERE status = 'active'; -- Expression (functional) index CREATE INDEX idx_lower_email ON Students(LOWER(email)); -- Now WHERE LOWER(email) = 'mudit@example.com' can use this index -- Dropping an index DROP INDEX idx_students_dept;

Composite Index: Column Order Matters

A composite index on (A, B, C) is sorted first by A, then by B within the same A, then by C within the same A and B. Queries can use the index from the left. This is called the "leading column" rule.

CREATE INDEX idx_enroll ON Enrollments(student_id, semester, course_id); -- Can USE the index: WHERE student_id = 101 -- leading column only WHERE student_id = 101 AND semester = '2026S1' WHERE student_id = 101 AND semester = '2026S1' AND course_id = 5 -- CANNOT efficiently use the index: WHERE semester = '2026S1' -- skips leading column WHERE course_id = 5 -- skips both leading columns

Clustered vs Non-Clustered Index

A clustered index determines the physical order in which rows are stored on disk. The leaf nodes are the actual table rows. There can be only one clustered index per table (the rows can only be in one order). In MySQL InnoDB, the primary key is always the clustered index.

A non-clustered index stores key values and pointers to the heap (or to the clustered index key). There can be many non-clustered indexes per table. A non-clustered index lookup requires an extra step to fetch the actual row from the heap.

PropertyClustered IndexNon-Clustered Index
Row storage orderDetermines physical orderNo effect on row order
Per tableOnly oneMany (up to DBMS limit)
Lookup costNo heap fetch neededExtra heap fetch (key lookup)
Range scansVery efficientEfficient with covering index, otherwise many heap fetches

Covering Index

A covering index includes all columns referenced in a query (SELECT list + WHERE + JOIN + ORDER BY), so the DBMS can answer the query entirely from the index without reading the heap at all.

-- Query: get dept_id and gpa for active students SELECT dept_id, gpa FROM Students WHERE status = 'active'; -- Covering index: includes all three columns referenced CREATE INDEX idx_cover ON Students(status, dept_id, gpa); -- The index stores (status, dept_id, gpa) -- all needed columns are in the index leaf. -- No heap fetch required -- index-only scan.

Hash Index

A hash index stores (key value, row pointer) in a hash table. Lookups are O(1) for exact equality matches but cannot serve range queries or ORDER BY.

-- PostgreSQL supports hash indexes CREATE INDEX idx_hash_email ON Students USING HASH (email); -- Effective for: WHERE email = 'x@example.com' -- NOT effective for: WHERE email < 'n...' or ORDER BY email

Bitmap Index

A bitmap index stores a bit array (vector) per distinct value of a column. Each bit in the vector corresponds to one row in the table: 1 means the row has that value, 0 means it doesn't. Very efficient for low-cardinality columns in analytical (OLAP) databases. Not suitable for OLTP tables with frequent inserts/updates because the entire bitmap for a value must be rewritten on every change.

-- Example: status column with values 'active', 'inactive', 'graduated' -- Bitmap for 'active': 1 0 1 1 0 0 1 ... (rows where status = 'active') -- Bitmap for 'inactive': 0 1 0 0 1 0 0 ... -- Bitmap for 'graduated': 0 0 0 0 0 1 0 ... -- Combining two conditions with AND/OR = bitwise AND/OR on the bitmaps -- very fast WHERE status = 'active' AND dept_id = 3 -- = bitmap_active AND bitmap_dept_3

Full-Text Index

Full-text indexes enable natural-language search over large text columns. Instead of a B-tree on the raw text, the DBMS builds an inverted index: a mapping of each word to the rows that contain it.

-- MySQL full-text index CREATE FULLTEXT INDEX ft_articles ON Articles(title, body); SELECT * FROM Articles WHERE MATCH(title, body) AGAINST('database normalization' IN NATURAL LANGUAGE MODE);

Index Selection Guidelines

  1. Index foreign keys: JOINs on FK columns are very common. Without an index, every join scans the child table.
  2. Index high-cardinality columns used in WHERE: The more distinct values a column has, the more selective the index.
  3. Composite index column order: Put the most selective condition first, or put equality conditions before range conditions.
  4. Use covering indexes for critical hot paths: Eliminate heap fetches for very frequent, latency-sensitive queries.
  5. Do not over-index write-heavy tables: Each index adds overhead to every INSERT, UPDATE, DELETE.
  6. Remove unused indexes: Check pg_stat_user_indexes (PostgreSQL) or sys.dm_db_index_usage_stats (SQL Server) for indexes never used by the optimizer.
  7. Partial indexes for subset queries: If you always filter by status = 'active', index only active rows — smaller index, faster scans.

Indexing and NULL Values

In most DBMS, B-tree indexes include NULL values. Some DBMS (Oracle) exclude NULLs from single-column B-tree indexes by default. Queries using IS NULL may or may not use an index depending on the DBMS. If you frequently query for NULL values, verify the behavior for your specific DBMS version.

What You Should Remember

  • Views are virtual tables. They simplify queries, enforce security, and provide logical independence from schema changes.
  • Updatable views require a single base table, no aggregation, no DISTINCT. Use WITH CHECK OPTION to prevent unsafe updates.
  • Materialized views store precomputed results — faster reads, stale data until refreshed. Good for reporting workloads.
  • B-tree indexes support equality, range, ORDER BY, and prefix LIKE. They are the default for most indexes.
  • Clustered index stores rows in sorted order — one per table. Non-clustered indexes are separate structures that point to rows.
  • Covering indexes eliminate heap fetches by including all columns needed by a query.
  • Hash indexes are O(1) for equality only. Bitmap indexes are efficient for low-cardinality OLAP columns.
  • Composite index column order follows the leading column rule — queries that skip leading columns cannot use the index efficiently.
