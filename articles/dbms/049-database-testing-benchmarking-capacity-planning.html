Database Testing, Benchmarking and Capacity Planning

Performance | 18 min read

Production incidents are often performance surprises that could have been caught with realistic load tests. Database testing has three layers: correctness, performance, and growth forecasting.

Correctness Testing

-- Integrity assertions in CI SELECT COUNT(*) FROM orders WHERE total < 0; -- must be 0 SELECT COUNT(*) FROM users WHERE email IS NULL; -- must be 0 SELECT COUNT(*) FROM order_items oi LEFT JOIN orders o ON oi.order_id = o.id WHERE o.id IS NULL; -- orphan check must be 0

Benchmarking with pgbench

-- Initialize pgbench schema pgbench -i -s 100 mydb -- Run 10-minute benchmark with 50 clients and 10 threads pgbench -c 50 -j 10 -T 600 mydb -- Custom workload script pgbench -c 100 -j 20 -T 900 -f workload.sql mydb -- Key metrics: -- TPS, p95 latency, lock wait time, WAL generation rate, replication lag

Capacity Planning

-- Simple throughput forecast: -- required_qps_next_quarter = current_qps * growth_factor -- Example: 8k qps * 1.7 = 13.6k qps -- Storage forecast: -- monthly_growth_gb = avg_daily_ingest_gb * 30 -- 40 GB/day -> 1200 GB/month -- 12-month projected raw = 14.4 TB (+ indexes, replicas, backups) -- Memory rule of thumb: -- shared_buffers ~ 25% RAM -- effective_cache_size ~ 75% RAM -- monitor cache hit ratio target > 95%

Summary

  • Test correctness with SQL assertions and migration checks.
  • Benchmark using realistic concurrency and SQL mix, not synthetic single-query loops.
  • Capacity plans should include data growth, index growth, backups, and replica overhead.
