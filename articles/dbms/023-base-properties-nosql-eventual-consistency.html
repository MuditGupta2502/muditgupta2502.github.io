BASE Properties and NoSQL Trade-offs: Eventual Consistency Explained

NoSQL & Distributed Systems | 24 min read

ACID guarantees are powerful but costly at scale. As systems grew to billions of users and global deployments, a different consistency model emerged — BASE — which trades strict consistency for availability and partition tolerance. Understanding BASE is prerequisite to making informed architectural choices between relational and NoSQL datastores.

ACID Recap: The Relational Guarantee

PropertyMeaning
AtomicityAll operations in a transaction succeed or all fail — no partial writes
ConsistencyDatabase transitions between valid states only; all constraints hold
IsolationConcurrent transactions see results as if they ran sequentially
DurabilityCommitted transaction survives crashes (persisted to durable storage)

BASE: The NoSQL Alternative

BASE stands for Basically Available, Soft state, Eventual consistency — coined by Eric Brewer (of CAP theorem fame) as an acronym to contrast with ACID.

PropertyMeaning
Basically AvailableThe system responds to every request, but the response may reflect stale data
Soft stateThe state of the system can change over time even without new inputs (replicas converging)
Eventual consistencyGiven no new updates, all replicas will eventually converge to the same value
Real-world example — Amazon shopping cart: Amazon famously used an eventually consistent shopping cart: adding an item always succeeds (basically available) even if the cart's nodes are partially unavailable. In rare cases, two sessions might show different carts (soft state). Eventually, when connectivity is restored, the carts converge — typically using "last write wins" or union merge. The trade-off: a slightly stale cart is acceptable; a cart that fails to add items is not.

Consistency Models: A Spectrum

Strongest Weakest │ │ Linearizability → Sequential → Causal → Read-Your-Writes → Eventual │ │ (most expensive: global coordination) (cheapest: local operations)

Strong Consistency (Linearizability)

After a write completes, every subsequent read (from anywhere in the system) sees that write. Requires coordination across all replicas before acknowledging the write.

-- Example: bank transfer -- Write: transfer $100 from A to B deducted from A and added to B atomically -- Any read of A or B afterwards reflects the transfer -- Used by: ZooKeeper, Google Spanner, etcd (distributed consensus systems)

Causal Consistency

Operations that are causally related are seen in order by all nodes. Concurrent (causally unrelated) operations may be seen in different orders.

-- Social media example: -- User A posts: "Just got promoted!" (post P) -- User B replies: "Congratulations!" (reply R, causally after P) -- Causal consistency guarantees: any reader who sees R also sees P first -- A reader might not see any posts yet — but if they see R, they see P -- Used by: MongoDB (causal sessions), some Cassandra configurations

Read-Your-Writes Consistency

A process always sees the effects of its own writes. Other processes may not see them yet.

-- E-commerce profile update example: -- User updates their shipping address -- Immediately navigates to checkout -- Read-your-writes: they see their NEW address at checkout -- Without it: they might see the old address (because the read hit a stale replica) -- Implementation: route the same session's reads to the replica that has the write, -- or use a read-after-write timestamp fencing approach

Vector Clocks: Tracking Causality

-- Vector clock: a list of counters, one per node -- Every event increments the local counter -- Messages carry the sender's vector clock; receiver updates accordingly -- Three nodes: A, B, C -- Initial state: A=[0,0,0] B=[0,0,0] C=[0,0,0] -- A writes X=5: A=[1,0,0] -- A sends to B: B receives A=[1,0,0], updates to B=[1,1,0] (B's counter: 0→1) -- B writes X=7: B=[1,2,0] -- C independently writes X=9: C=[0,0,1] -- Now B=[1,2,0] and C=[0,0,1] are CONCURRENT — neither happened before the other -- This is a conflict! Both claims to modify X happened independently. -- Conflict resolution strategies: -- 1. Last-Write-Wins (LWW): use wall-clock timestamp — simple but can lose writes -- 2. Multi-value register (Riak CRDT): keep both values, return to application to resolve -- 3. Application-specific merge: e.g., shopping cart union of items

Conflict Resolution in Practice

StrategyHow It WorksData Loss Risk
Last Write Wins (LWW)Highest timestamp wins; conflicting write discardedYes — concurrent writes can be silently overwritten
Multi-Value (Sibling)Both values kept; application picks or mergesNo automatic loss; applications must handle siblings
CRDT (Conflict-free Replicated Data Type)Data structure designed to merge automatically (counters, sets, registers)No — always converges correctly by definition
Read-RepairOn read, coordinator detects inconsistency and repairsWorks after-the-fact; reads may briefly see stale data

CRDT Examples

-- G-Counter (Grow-only Counter): -- Each node has its own counter slot; increment is local -- Merge: take max of each slot; sum all slots for total value -- No conflict possible: you can only add -- G-Set (Grow-only Set): -- Elements can only be added, never removed -- Merge: union of both sets -- Example: "seen messages" set in a messaging app -- OR-Set (Observed-Remove Set): -- Adds a unique tag per-element-per-add operation -- Remove marks that specific tag -- Concurrent add + remove: add wins (the new tag survives) -- Used in: shopping carts (Riak), collaborative editing -- PN-Counter (Positive-Negative Counter): -- Two G-Counters: one for increments, one for decrements -- Value = sum(increments) - sum(decrements) -- Example: like/dislike counters on posts

Tunable Consistency: The Quorum Model

-- Cassandra / DynamoDB style tunable consistency -- N = replication factor (copies of each row) -- W = write quorum (how many replicas must confirm the write) -- R = read quorum (how many replicas must respond to the read) -- Strong consistency requires: R + W > N -- Example: N=3, W=2, R=2 → 2+2=4 > 3 → strong consistency -- ANY read will overlap with at least one write: guaranteed to see latest -- Eventual consistency: R + W ≤ N -- Example: N=3, W=1, R=1 → 1+1=2 ≤ 3 → eventual consistency only -- A write goes to one replica; a read may hit a different replica: stale data possible -- Maximum availability: operations succeed even if multiple nodes are down -- Common production settings: -- QUORUM (W=N/2+1, R=N/2+1): strong consistency, tolerates up to N/2 failures -- LOCAL_QUORUM: quorum within the local datacenter only -- ONE: fastest, least consistent — good for analytics or time-series data -- ALL: needs all N replicas — highest consistency, lowest availability -- In Cassandra: INSERT INTO orders (id, status) VALUES (123, 'pending') USING CONSISTENCY QUORUM; SELECT * FROM orders WHERE id = 123 USING CONSISTENCY LOCAL_QUORUM;

Choosing SQL vs NoSQL

DimensionChoose RDBMS (ACID)Choose NoSQL (BASE)
Data relationshipsComplex joins, many foreign keysDenormalized documents, single-entity access
Consistency needsFinancial, inventory, booking (must be exact)Social feeds, activity logs, IoT metrics
Query patternsAd-hoc queries, reporting, diverse filtersKnown access patterns, low-latency single entity
Schema flexibilityStable, well-defined schemaEvolving schema, heterogeneous fields per document
Scale dimensionVertical scaling, some horizontal read-replicasHorizontal write scaling
Team expertiseSQL is universal; easy toolingRequires understanding of consistency trade-offs
Common mistake: Choosing NoSQL because "it scales" without understanding the consistency guarantees you're giving up. A payment system needs ACID; using eventual consistency can lead to double-charges, lost orders, or phantom inventory. For most applications, a well-tuned PostgreSQL handles millions of requests per second — evaluate your actual bottleneck before trading consistency for scalability.

Summary

  • BASE = Basically Available, Soft state, Eventual consistency — trades strict consistency for availability
  • Eventual consistency: given no new updates, all replicas converge — but reads may be stale in the interim
  • Consistency models form a spectrum: linearizability → causal → session → eventual
  • Vector clocks track causality; concurrent writes become conflicts requiring resolution strategies
  • CRDTs design data structures that merge automatically without conflicts (counters, sets)
  • Quorum (R+W>N) gives strong consistency; lower quorums give higher availability with eventual consistency
  • NoSQL shines for single-entity access patterns, flexible schemas, and write-heavy horizontal scale
