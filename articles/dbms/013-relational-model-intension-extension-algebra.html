Relational Model: Intension, Extension & Relational Algebra

Database Theory | 25 min read

Edgar Codd introduced the relational model in 1970, providing a formal mathematical foundation for data management. Before SQL existed, relational algebra defined exactly what operations a relational database must support. Understanding the formal model — intension vs extension, constraints, and the complete set of algebraic operations — gives you a far deeper understanding of how database engines work and why SQL is designed the way it is.

Formal Definitions

Relation

A relation (table) is a subset of the Cartesian product of its attribute domains. If a relation R has attributes A1, A2, ..., An with domains D1, D2, ..., Dn, then:

R ⊆ D1 × D2 × ... × Dn A tuple (row) t ∈ R is an ordered list (v1, v2, ..., vn) where vi ∈ Di Key properties: - No duplicate tuples (set-based semantics; SQL tables are bags unless DISTINCT used) - No order among tuples - No order among attributes (tuples are accessed by attribute name, not position) - Atomic attribute values (first normal form requirement)

Domain

A domain is the set of possible values for an attribute. Domains can be:

  • Atomic types: INTEGER, VARCHAR(100), DATE, BOOLEAN
  • Formally: a named set with an associated data type, e.g., dom(age) = {x ∈ ℤ | 0 ≤ x ≤ 150}

Intension vs Extension

This is one of the most conceptually important distinctions in the relational model:

Intension (Schema / Intent)

The intension (also called the schema or relation scheme) is the structural definition of a relation — the attribute names, their domains, constraints, and metadata. The intension is relatively stable; it changes only when the structure of the data changes.

-- Intension: the CREATE TABLE definition CREATE TABLE Orders ( order_id INT PRIMARY KEY, customer_id INT NOT NULL REFERENCES Customers(customer_id), order_date DATE NOT NULL, total DECIMAL(10,2) NOT NULL CHECK (total >= 0), status VARCHAR(20) DEFAULT 'pending' ); -- This definition is the INTENSION of the Orders relation -- It doesn't change unless columns are added/dropped/altered

Extension (Instance / State)

The extension (also called the instance or relation state) is the actual set of tuples in the relation at a given point in time. The extension is dynamic; it changes with every INSERT, UPDATE, and DELETE.

Relational Constraints

Every valid extension must satisfy all constraints defined in the intension:

  • Domain constraint: Every attribute value must come from its domain (e.g., age must be an integer in [0,150])
  • Key constraint: No two tuples may have the same value for the primary key
  • Entity integrity: Primary key attributes cannot be NULL
  • Referential integrity: Foreign key values must reference an existing PK in the referenced relation (or be NULL)
  • Semantic constraints: Application-level rules expressed via CHECK, triggers, or application code

Relational Algebra

Relational algebra is a procedural query language that defines a set of operations on relations, each of which produces a new relation. SQL is implemented on top of these formal operations. There are 8 fundamental operations: 5 foundational + 3 derived.

1. Selection (σ — Sigma)

Filters tuples (rows) based on a predicate. Returns tuples satisfying the condition. Equivalent to SQL WHERE.

σ condition(R) Example: Get all orders with total > 100 σ total > 100 (Orders) SQL equivalent: SELECT * FROM Orders WHERE total > 100; Compound conditions use ∧ (AND), ∨ (OR), ¬ (NOT): σ status='shipped' ∧ total > 50 (Orders)

2. Projection (π — Pi)

Extracts specific attributes (columns). Equivalent to the SELECT column list in SQL. Automatically eliminates duplicates (set semantics).

π attribute_list(R) Example: Get only order_id and total from Orders π order_id, total (Orders) SQL equivalent: SELECT DISTINCT order_id, total FROM Orders; -- Note: SQL without DISTINCT is bag semantics; RA is set semantics

3. Cartesian Product (× )

Combines every tuple from relation R with every tuple from relation S. If R has n tuples and S has m tuples, the result has n×m tuples. Equivalent to SQL CROSS JOIN.

R × S Orders (3 tuples) × Customers (100 tuples) = 300 tuples -- Rarely used alone; always followed by selection to form a join SQL equivalent: SELECT * FROM Orders CROSS JOIN Customers;

4. Union (∪)

Combines all tuples from R and S, removing duplicates. R and S must be union-compatible: same number of attributes with compatible domains.

R ∪ S Example: All customers from New York OR Chicago σ city='New York'(Customers) ∪ σ city='Chicago'(Customers) SQL equivalent: SELECT * FROM Customers WHERE city='New York' UNION SELECT * FROM Customers WHERE city='Chicago';

5. Set Difference (−)

Returns tuples in R that are not in S. Also requires union-compatible relations.

R − S Example: Customers who placed an order but have never been refunded π customer_id(Orders) − π customer_id(Refunds) SQL equivalent: SELECT customer_id FROM Orders EXCEPT SELECT customer_id FROM Refunds;

6. Rename (ρ — Rho) — Derived

Renames a relation and/or its attributes. Useful for self-joins and disambiguating operations.

ρ new_name(A1, A2, ...) (R) Example: To find pairs of employees in the same department: ρ E1(emp_id1, name1, dept1) (Employee) × ρ E2(emp_id2, name2, dept2) (Employee) → then filter: σ dept1 = dept2 ∧ emp_id1 ≠ emp_id2

7. Natural Join (⊳⊲) — Derived

Joins two relations on all common attribute names, keeping one copy of each common attribute. This is the "default join" in theoretical discussions.

R ⊳⊲ S Example: Orders joined with Customers (on customer_id) Orders ⊳⊲ Customers → returns all orders with their matching customer info Composed from primitives: σ Orders.customer_id = Customers.customer_id (Orders × Customers) → then π (remove duplicate customer_id column) SQL equivalent: SELECT * FROM Orders JOIN Customers USING (customer_id); -- or equivalently: SELECT * FROM Orders NATURAL JOIN Customers;

8. Division (÷) — Derived

The most complex operation. R ÷ S returns the values in R's attributes not in S that are associated with every tuple in S. Used for "find X associated with all Y" queries.

R ÷ S Example: Find students who have taken ALL courses offered in CS101 section Enrollments(student_id, course_id) ÷ CS101Courses(course_id) → Returns student_ids enrolled in every CS101 course Real-world: Find customers who have ordered every product in category "Essential" OrderItems(customer_id, product_id) ÷ π product_id (σ category='Essential' (Products)) SQL equivalent (using NOT EXISTS): SELECT DISTINCT e.customer_id FROM OrderItems e WHERE NOT EXISTS ( SELECT p.product_id FROM Products p WHERE p.category = 'Essential' EXCEPT SELECT oi.product_id FROM OrderItems oi WHERE oi.customer_id = e.customer_id );

Set vs Bag semantics: Relational algebra operates on sets (no duplicates). SQL operates on bags (multisets) by default — duplicates are preserved unless you use DISTINCT or set operators (UNION, EXCEPT, INTERSECT). This is why SQL is technically not pure relational algebra.

Composing Operations: A Complex Query

The real power of relational algebra is composability. Let's express: "Find the names of customers who ordered products from both the 'Electronics' and 'Books' categories and spent more than $200 total."

-- Step 1: Electronics orders Elec ← π customer_id ( σ category='Electronics' (Products ⊳⊲ OrderItems ⊳⊲ Orders) ) -- Step 2: Books orders Books ← π customer_id ( σ category='Books' (Products ⊳⊲ OrderItems ⊳⊲ Orders) ) -- Step 3: Customers who ordered BOTH Both ← Elec ∩ Books -- Step 4: Customers who spent > $200 BigSpenders ← π customer_id (σ total > 200 (Orders)) -- Step 5: Intersection Target ← Both ∩ BigSpenders -- Step 6: Get names π name (Target ⊳⊲ Customers)

Relational Completeness

A query language is relationally complete if it can express every query that relational algebra can express. SQL is relationally complete. Any operation expressible using the 5 fundamental RA operations (σ, π, ×, ∪, −) can be expressed in SQL.

For interviews and exams: The division operator (÷) is the most commonly tested because it's the hardest to translate to SQL. Memorize the NOT EXISTS double negation pattern — it always works for "find X related to ALL Y" problems.

Summary

  • Intension: The schema/structure definition — stable, defines what the data can look like
  • Extension: The actual data instance at a point in time — changes with every DML operation
  • The 5 foundational RA ops: σ (selection), π (projection), × (Cartesian product), ∪ (union), − (difference)
  • The 3 derived ops: ρ (rename), ⊳⊲ (natural join), ÷ (division)
  • Every SQL query can be expressed in relational algebra; SQL is built on these formal foundations
