NoSQL Databases: Models, CAP Theorem & When to Use Them

Database Systems | 24 min read

Why NoSQL Exists

Relational databases were designed for a world of structured, predictable data and single-server deployments. By the mid-2000s, internet-scale companies — Google, Amazon, Facebook — were confronting workloads that relational engines struggled with: billions of users, petabytes of semi-structured data, globally distributed deployments, and write rates that no single SQL server could absorb.

NoSQL (Not Only SQL) databases emerged to address specific combinations of scale, schema flexibility, and access pattern that relational models handle poorly. They are not replacements for relational databases — they are tools optimized for different problems.

ACID vs BASE

Relational databases are defined by ACID guarantees. Many NoSQL databases trade some of those guarantees for availability and horizontal scalability, following what is called the BASE model.

PropertyACID (Relational)BASE (Many NoSQL)
AtomicityAll-or-nothing transactionsBest effort; partial updates may be visible
ConsistencyStrict: follows all constraintsEventual: converges over time
IsolationConcurrent txns appear serialVaries; often weaker isolation
DurabilityCommitted data survives failuresUsually durable; sometimes async replication
  • BAsically Available — the system guarantees availability even during partial failures.
  • Soft state — the state of the system may change over time even without input.
  • Eventually consistent — given enough time without new writes, all replicas converge to the same value.

CAP Theorem

CAP theorem (Brewer, 2000) states that a distributed data store can only guarantee two of the following three properties simultaneously:

  • Consistency (C): Every read receives the most recent write or an error. All nodes see the same data at the same time.
  • Availability (A): Every request receives a (non-error) response, though the data may not be the most recent.
  • Partition Tolerance (P): The system continues operating even when network partitions split nodes and messages are lost.
In practice, network partitions happen in any distributed system. Partition tolerance is therefore non-negotiable. The real choice is between Consistency and Availability during a partition.

CAP Classification of Databases

ClassificationGuaranteesExamples
CP (Consistent + Partition-tolerant)Returns error or timeout when partition occurs rather than serving stale dataHBase, Zookeeper, MongoDB (configurable), etcd
AP (Available + Partition-tolerant)Always responds; may serve stale data during partitionCassandra, CouchDB, DynamoDB, Riak
CA (Consistent + Available)Only possible in a single-server setup — no partition toleranceTraditional single-node RDBMS

Real systems are not binary on CAP. Cassandra lets you tune consistency per query: reading from a quorum of nodes is more consistent than reading from a single node. The CAP theorem defines the ceiling of guarantees, not a fixed setting.

NoSQL Data Models

1. Key-Value Stores

The simplest NoSQL model: a hash map at scale. Every record is a key-value pair. Keys are unique identifiers; values can be any blob (string, JSON, binary). The database does not interpret the value — that is entirely the application's job.

-- Redis (in-memory key-value store with data structures) -- Strings SET session:123 "{'user':'mudit','role':'admin'}" GET session:123 -- returns the JSON string EXPIRE session:123 3600 -- auto-delete after 1 hour (TTL) -- Counters INCR page_views:homepage -- atomic increment (thread-safe) INCRBY product_stock:42 -5 -- decrease stock by 5 -- Hashes (field-value pairs under one key) HSET user:101 name "Mudit" email "m@example.com" gpa 9.2 HGET user:101 name -- "Mudit" -- Lists (push/pop from either end) LPUSH notifications:user:101 "New message" RPOP notifications:user:101 -- Sets (unique members) SADD enrolled:course:DBMS student:101 student:102 SMEMBERS enrolled:course:DBMS

Best for: session storage, caching, rate limiting, leaderboards, pub/sub messaging, feature flags.

Examples: Redis, Memcached, Amazon DynamoDB (also supports document mode).

2. Document Stores

A document store organizes data as semi-structured documents, typically JSON or BSON. Each document can have its own schema — different documents in the same collection can have different fields. Documents are identified by a unique _id field.

// MongoDB — insert a document db.students.insertOne({ _id: ObjectId("64a1b2c3d4e5f6a7b8c9d0e1"), student_id: 101, name: "Mudit", email: "mudit@example.com", gpa: 9.2, dept: { id: 3, name: "Computer Science" // embedded sub-document }, courses: [ // array of embedded documents { course_id: "C001", name: "DBMS", grade: "A" }, { course_id: "C002", name: "OS", grade: "B+" } ], tags: ["full-time", "scholarship"] }); // Query: students in CS with GPA >= 8.5 db.students.find({ "dept.name": "Computer Science", gpa: { $gte: 8.5 } }); // Update: add a new course to the array db.students.updateOne( { student_id: 101 }, { $push: { courses: { course_id: "C003", name: "Algorithms", grade: null } } } ); // Aggregation (similar to SQL GROUP BY) db.students.aggregate([ { $group: { _id: "$dept.name", avg_gpa: { $avg: "$gpa" }, count: { $sum: 1 } } }, { $sort: { avg_gpa: -1 } } ]);

When to use: Catalogs (products, articles) with variable schema; hierarchical/nested data that would need many JOINs in SQL; rapid schema evolution in early development.

Weakness: Cross-document transactions (multi-document ACID) are available in MongoDB 4+ but less efficient than relational JOINs for heavily relational data.

Examples: MongoDB, Couchbase, Amazon DocumentDB, Firestore.

3. Column-Family (Wide-Column) Stores

Column-family databases (often called wide-column stores) group related columns into "column families." Unlike relational databases — where every row has all columns — wide-column stores allow each row to have a different set of column instances. Rows are identified by a row key; within a row, columns can vary.

Cassandra's data model is especially important because it is designed to be written fast and distributed. Queries must be designed around Cassandra's partition key and clustering key: the partition key determines which node stores the data; the clustering key determines the sort order within a partition.

-- Cassandra CQL (Cassandra Query Language) CREATE KEYSPACE university WITH replication = { 'class': 'SimpleStrategy', 'replication_factor': 3 }; -- Table designed for query: "get all enrollments for a student in a semester" -- Partition key: student_id -- all data for one student on one node -- Clustering key: semester, course_id -- sorted within partition CREATE TABLE enrollments ( student_id INT, semester TEXT, course_id TEXT, grade TEXT, PRIMARY KEY (student_id, semester, course_id) ) WITH CLUSTERING ORDER BY (semester DESC, course_id ASC); INSERT INTO enrollments (student_id, semester, course_id, grade) VALUES (101, '2026S1', 'DBMS', 'A'); -- Efficient: follows partition key SELECT * FROM enrollments WHERE student_id = 101 AND semester = '2026S1'; -- Inefficient / not allowed without ALLOW FILTERING: -- SELECT * FROM enrollments WHERE grade = 'A'; -- grade is not in PK

Design principle: In Cassandra, model tables around queries, not around normalized entities. Duplication is acceptable; JOINs do not exist.

Best for: Time-series data, event logs, IoT sensor data, write-heavy workloads at massive scale.

Examples: Apache Cassandra, Apache HBase, Google Bigtable, Amazon Keyspaces.

4. Graph Databases

A graph database stores data as nodes (entities) and edges (relationships). Both nodes and edges can have properties. Graph databases are optimized for traversal queries — "find all friends of friends within 3 hops" — which in a relational database would require recursive CTEs or repeated self-joins.

-- Neo4j: Cypher query language // Create nodes and relationships CREATE (mudit:Person { name: 'Mudit', age: 24 }) CREATE (aisha:Person { name: 'Aisha', age: 23 }) CREATE (dbms:Course { name: 'DBMS', code: 'CS301' }) CREATE (mudit)-[:FRIENDS_WITH]->(aisha) CREATE (mudit)-[:ENROLLED_IN { grade: 'A', semester: '2026S1' }]->(dbms) CREATE (aisha)-[:ENROLLED_IN { grade: 'B+', semester: '2026S1' }]->(dbms) // Find Mudit's friends and their courses MATCH (mudit:Person { name: 'Mudit' })-[:FRIENDS_WITH]->(friend)-[:ENROLLED_IN]->(course) RETURN friend.name, course.name // Find all people within 2 hops of Mudit MATCH (mudit:Person { name: 'Mudit' })-[*1..2]-(connected) RETURN DISTINCT connected.name // Shortest path MATCH p = shortestPath( (mudit:Person { name: 'Mudit' })-[*]-(target:Person { name: 'Ravi' }) ) RETURN p

Best for: Social networks, recommendation engines, fraud detection (unusual connection patterns), knowledge graphs, access control hierarchies, route planning.

Examples: Neo4j, Amazon Neptune, JanusGraph, TigerGraph.

5. Time-Series Databases

Time-series databases (TSDB) are optimized for data that is indexed by time. They use specialized compression (successive timestamps/values differ little) and downsampling (reduce old high-resolution data to aggregates).

-- InfluxDB line protocol cpu_usage,host=server01,region=us-east value=72.5 1711483200000000000 cpu_usage,host=server01,region=us-east value=74.1 1711483260000000000 -- Query using Flux from(bucket: "metrics") |> range(start: -1h) |> filter(fn: (r) => r._measurement == "cpu_usage" and r.host == "server01") |> aggregateWindow(every: 5m, fn: mean)

Best for: Infrastructure monitoring, IoT sensors, financial tick data, application performance metrics.

Examples: InfluxDB, TimescaleDB (PostgreSQL extension), Prometheus, OpenTSDB.

Choosing the Right Database

Workload / NeedBest FitWhy
Transactional records (banking, orders)Relational (PostgreSQL, MySQL)ACID, complex queries, strong consistency
Session / cachingKey-Value (Redis)Sub-millisecond, TTL support, memory speed
Variable-schema content (product catalog)Document (MongoDB)Flexible schema, nested data, easy updates
Write-heavy IoT / logs at massive scaleColumn-family (Cassandra)Linear horizontal scale, tunable consistency
Relationships as first-class dataGraph (Neo4j)Native relationship traversal, no JOINs needed

Multi-Model Databases

Many modern databases support more than one model. PostgreSQL can store JSON documents, act as a key-value store, and handle time-series with extensions. MongoDB has added multi-document ACID transactions. The category boundaries are blurring, but the core strength of each model remains the same.

What You Should Remember

  • NoSQL databases sacrifice some ACID guarantees in exchange for horizontal scalability, schema flexibility, or specialized access patterns.
  • CAP theorem: distributed systems can only guarantee two of Consistency, Availability, Partition tolerance. Since partitions are inevitable, the real choice is C vs A during failure.
  • BASE (Basically Available, Soft state, Eventually consistent) describes the consistency model of many NoSQL systems.
  • Key-value stores are fastest for point lookups; document stores handle nested/variable data; column-family stores handle massive write throughput; graph databases handle relationship-heavy queries.
  • In Cassandra, model tables around query patterns, not entities. Denormalization and duplication are intentional design choices.
  • Relational and NoSQL are not enemies — many production systems use both: SQL for transactional data, Redis for caching, Elasticsearch for search, Cassandra for logs.
