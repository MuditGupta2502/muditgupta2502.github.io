Common Database Anti-Patterns and Practical Fixes

Best Practices | 16 min read

N+1 Query Problem

-- Anti-pattern: query orders, then query customer for each row -- Fix: single join query SELECT o.id, o.total, c.name FROM orders o JOIN customers c ON o.customer_id = c.id WHERE o.created_at >= CURRENT_DATE - INTERVAL '7 days';

Missing Composite Indexes

-- Query pattern: -- WHERE tenant_id = ? AND status = ? ORDER BY created_at DESC -- Correct index: CREATE INDEX idx_orders_tenant_status_created ON orders(tenant_id, status, created_at DESC);

Oversized Transactions

-- Anti-pattern: update millions of rows in one transaction -- Fix: batch updates UPDATE events SET processed = true WHERE id IN ( SELECT id FROM events WHERE processed = false LIMIT 5000 );
Never disable constraints to "speed up" production writes without controlled backfill and verification. Integrity bugs are more expensive than slow jobs.

Summary

  • Profile real query patterns before adding indexes.
  • Prefer small batched writes over long locking transactions.
  • Treat schema constraints as correctness guarantees, not optional checks.
