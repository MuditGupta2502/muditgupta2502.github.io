What are Isolation Levels?

Isolation levels in DBMS control the visibility of data changes made by one transaction to other concurrent transactions. Different isolation levels provide varying degrees of isolation, balancing data consistency with performance. Isolation levels help prevent issues like dirty reads, non-repeatable reads, and phantom reads.

Real-Life Analogy:

Imagine a library where several people are allowed to read or update the catalog simultaneously. The level of isolation determines when others can see the updates made by a person working on the catalog. Some may see changes immediately (low isolation), while others may only see them after everything is finalized (high isolation).

Dirty Reads

A Dirty Read occurs when a transaction reads data that another transaction has modified but not yet committed. If the first transaction is later rolled back, the data read by the second transaction becomes invalid or "dirty." Dirty reads can lead to inconsistencies in the database.

Example of Dirty Read:

Consider a Banking System with two transactions:

  • T1: Updates Account A’s balance but has not yet committed.
  • T2: Reads the balance of Account A.

If T1 rolls back after T2 reads the balance, T2 would have accessed incorrect information.

Isolation Levels in DBMS

The four main isolation levels in DBMS are Read Uncommitted, Read Committed, Repeatable Read, and Serializable. Each level restricts data visibility differently.

1. Read Uncommitted

In the Read Uncommitted isolation level, transactions are allowed to read uncommitted changes made by other transactions. This is the lowest isolation level, allowing dirty reads.

Advantages:

  • Fastest performance, minimal locking.

Disadvantages:

  • High risk of inconsistencies due to dirty reads.

Example of Read Uncommitted:

In an Inventory Management System:

  • T1: Updates the quantity of Product A but it’s uncommitted.
  • T2: Reads Product A’s quantity, seeing the uncommitted value.

If T1 rolls back, T2 will have read incorrect information.

2. Read Committed

In the Read Committed isolation level, transactions can only read data that has been committed by other transactions. This isolation level prevents dirty reads but allows non-repeatable reads (data may change if read again in the same transaction).

Advantages:

  • Prevents dirty reads, reasonable performance.

Disadvantages:

  • Allows non-repeatable reads.

Example of Read Committed:

In a Ticket Reservation System:

  • T1: Updates seat availability and commits.
  • T2: Reads the seat availability after T1 commits, so it always sees committed data.

If T1 makes another update after T2 reads, T2 would see different results on re-reading.

3. Repeatable Read

In the Repeatable Read isolation level, a transaction is guaranteed to see the same data when it reads the same item multiple times within the same transaction. This level prevents dirty reads and non-repeatable reads but does not prevent phantom reads (new rows appearing during a transaction).

Advantages:

  • Prevents dirty and non-repeatable reads.

Disadvantages:

  • Still allows phantom reads.

Example of Repeatable Read:

In an E-commerce System:

  • T1: Reads the stock level of Product X.
  • T2: Tries to update Product X’s stock but has to wait until T1 completes.

When T1 reads Product X’s stock again within the same transaction, it sees the same value.

Illustrative Example:

T1 T2
Balance + 100 (Uncommitted)
Read Balance 0 or 100 (depends on isolation level)
Commit
Read Balance 100
