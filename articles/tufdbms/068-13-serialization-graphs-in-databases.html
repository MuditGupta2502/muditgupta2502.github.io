Serialization Graphs in Database

Serialization graphs help to analyze transaction schedules for serializability. They provide a graphical representation of how transactions interact with each other based on their operations. These graphs allow us to test if a non-serial schedule can be rearranged to yield a consistent outcome.

Real-Life Analogy:

Imagine a group project where each member (transaction) is working on different sections (operations) of the same document. The final report must make sense, meaning their sections should be properly ordered to avoid conflicts. Serialization graphs help ensure that the "project" (transaction schedule) is consistent by showing dependencies and order between members.

Precedence Graphs

A Precedence Graph (also known as a Serialization Graph) is a directed graph that represents the order of operations between transactions. Each node in the graph represents a transaction, and each edge represents a dependency between two transactions due to conflicting operations.

How Precedence Graphs Work

  • Nodes: Represent transactions (e.g., T1, T2, T3).
  • Edges: A directed edge from T1 to T2 means that T1 has a conflicting operation with T2, which must occur before T2 to maintain consistency.
  • Cycle Detection: If there is a cycle in the graph, the schedule is non-serializable (cannot be rearranged to match a serial schedule).

Example: Precedence Graph for Conflict Detection

Consider transactions T1 and T2 in an Inventory System:

  • T1 reads and writes Item A.
  • T2 reads Item A after T1 writes to it.

A precedence graph would have:

  • Node for T1 and T2.
  • Edge from T1 to T2 to represent the dependency due to the read-write conflict.

Cascading Graphs

A Cascading Graph is used to analyze cascading rollbacks, where one transaction's failure leads to the rollback of multiple dependent transactions. A cascading graph illustrates dependencies between transactions that can cause a cascade effect if one of the transactions aborts.

Example of Cascading Rollbacks

In a Banking System:

  • T1 updates Account A, and T2 reads the updated balance of Account A.
  • If T1 is aborted, T2 must be rolled back as it relied on uncommitted data from T1.

This creates a cascading effect, where T2 depends on the success of T1.

Recoverable Schedules

A recoverable schedule ensures that if a transaction reads data modified by another transaction, it only does so after that transaction has committed. Recoverable schedules prevent cascading rollbacks by ensuring dependent transactions can safely continue.

Characteristics of Recoverable Schedules:

  • Safe Reads: Transactions only read from committed data, reducing the risk of cascading failures.
  • Dependency Control: The schedule ensures dependencies are consistent by only allowing committed data to be accessed by other transactions.

Example of a Recoverable Schedule

Consider two transactions T1 and T2:

  • T1 writes to X, and T2 reads X.
  • If T1 commits before T2 reads, the schedule is recoverable.

Non-Serializable Schedules

A non-serializable schedule is one where transactions are interleaved in such a way that it is impossible to rearrange them to achieve a consistent result equivalent to any serial schedule. Non-serializable schedules can lead to data inconsistencies and are undesirable in a DBMS.

Example of Non-Serializable Schedule

Consider transactions T1 and T2:

  • T1 writes to A, then T2 reads and writes to A.

Due to conflicts, no serial order can be achieved, making the schedule non-serializable.

Testing for Serializability

Testing for serializability involves checking if a schedule can be transformed into a serial schedule without conflicts. This is typically done using Precedence Graphs. Here’s how the testing process works:

  • Build a Precedence Graph: Identify transactions and conflicts, and create directed edges between them.
  • Check for Cycles: If the graph has no cycles, the schedule is serializable. If there is a cycle, it’s non-serializable.

Example of Serializability Testing

In an Online Reservation System:

  • T1 books a ticket, and T2 views ticket availability.

By building a precedence graph, we can check if these transactions conflict, ensuring consistency in reservations.
